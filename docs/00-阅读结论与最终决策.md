# 阅读结论与最终决策

## 输入材料

本次整理综合了三类材料：

| 来源 | 作用 |
|---|---|
| `总设计.md` | 提供完整的产品定位、模块地图、架构、数据模型、安全、评测和路线图 |
| `聊天主链路设计.md` | 提供聊天主链路、Context Gateway、Memory、Task、Safety、Response Composer 等主干设计 |
| ChatGPT 分享会话 | 补充多轮决策过程，包括多生命体、资产中心、壳系统、公司壳 MVP、聊天页简洁规则、智能体默认独立、Skill/MCP/Resource 分工 |

分享会话标题为“分支 · 分支 · 分支 · 智能体平台开发构想”，解析出的线性会话节点共 117 个。核心要求从“能干活、有记忆、越用越聪明的智能体平台”逐步收束为“可单机部署、组织化、多成员、可换壳、聊天极简、管理完整的个人智能体操作系统”。

## 最终产品定义

产品最终定义为：

> 可单机部署的个人智能体操作系统。它以极简聊天为前台，以组织、成员、资产、技能、任务、记忆、工具、MCP、插件、安全和审计为后台，让用户像和真人协作一样完成聊天、陪伴、创作、执行和长期任务。

它不是：

| 不是 | 原因 |
|---|---|
| 简单聊天机器人 | 聊天只是入口，核心价值在长期记忆、任务执行和能力成长 |
| 纯 Agent 自动化工具 | 需要有性格、情绪、关系连续性和自然聊天体验 |
| 企业 SaaS | 首发是个人单机使用，不做多租户和企业级权限复杂度 |
| 单一壳世界观产品 | 首发公司壳，但底层必须可扩展到宗门、朝廷、学院、舰队等壳 |
| 把所有功能塞进聊天页的系统 | 管理功能要有独立页面，聊天页保持干净 |

## 最高优先级取舍

| 设计面 | 最终选择 | 说明 |
|---|---|---|
| 部署形态 | 单机优先、本地优先、可混合云 | 个人家用电脑可运行，隐私和低门槛优先 |
| 主入口 | 聊天 | 所有日常使用从聊天开始 |
| 管理入口 | 独立管理页面 | 资产、成员、组织、任务、系统设置不塞进聊天页 |
| 首发壳 | 公司壳 | 公司壳最容易理解：用户是老板，AI 是员工 |
| 底层模型 | 组织化抽象 | 使用 Organization、Member、Role、Shell、Asset、Skill 等通用对象 |
| 壳逻辑 | 只改展示和模板，不自动改底层数据 | 换壳时标签可变，字段值不自动变 |
| 智能体感知 | 默认独立，按需注入 | 不把全组织和全资产塞进上下文 |
| 资源访问 | Asset Broker + Capability Graph + Tool/MCP | 资源发现、授权和执行分层 |
| Skill 定位 | 使用方法和流程沉淀 | Skill 不是资源库本身，而是“如何使用资源” |
| 任务执行 | Workflow + Agent 混合 | 固定流程走 workflow，探索型任务走 agent |
| 安全策略 | 舒适默认 + 高风险确认 | 不用复杂权限向导打断首体验，但危险动作必须拦截 |

## 用户体验最终原则

### 聊天页只保留关系感

聊天界面不展示组织和壳。无论公司壳、宗门壳还是朝廷壳，聊天顶部都只显示：

```text
头像
人名
在线/忙碌/执行中状态
```

聊天页不显示：

```text
组织名称
当前壳名称
部门树
岗位/身份标签
资产归属
权限等级
复杂任务面板
右侧管理卡片
```

必要时可以在消息中出现轻量按钮，例如确认执行、查看过程、继续任务、暂停任务，但不做大量卡片式工作台。

### 管理页面必须完整

用户平时在聊天页完成绝大多数操作，但需要查看、编辑、配置时必须有清晰的管理页面：

```text
聊天
成员管理
组织管理
资产管理或壳映射后的名称
任务管理
系统管理
```

### 上手必须舒服

首次使用不让用户理解 Skill、MCP、Capability Graph、Asset Broker、权限策略等后台概念。推荐流程：

1. 选择或默认进入公司壳。
2. 输入组织名称，默认“一人公司”。
3. 选择大脑，支持本地、云端或混合。
4. 系统创建默认主成员，例如“小曜”。
5. 直接进入聊天。

创建智能体的首屏只需要：

```text
名字
头像
大脑
人格模板
```

高级能力放到“更多设置”里，不在首屏打扰用户。

## 壳系统最终规则

壳的职责是：

```text
菜单名称映射
字段标签映射
创建模板
默认组织结构
默认成员模板
引导文案
视觉主题
聊天语气建议
```

壳不能自动修改：

```text
成员真实名称
部门真实值
岗位真实值
资产真实值
权限真实策略
任务历史
记忆内容
审计日志
```

示例：

```text
公司壳：
姓名：阿珩
部门：研发部
岗位：技术经理

切到宗门壳后：
姓名：阿珩
堂口：研发部
身份：技术经理
```

字段名从“岗位”显示为“身份”，但字段值仍然是“技术经理”。只有用户主动把它改成“炼器长老”，底层值才变为“炼器长老”。

## 菜单变化最终规则

| 模块 | 一级菜单是否随壳变化 | 二级菜单是否随壳变化 | 规则 |
|---|---:|---:|---|
| 聊天 | 是 | 按壳决定 | 公司壳可叫企业聊天，朝廷壳可叫朝议 |
| 成员管理 | 是 | 是 | 公司壳员工管理，宗门壳弟子管理 |
| 组织管理 | 是 | 是 | 公司壳部门管理，朝廷壳衙署管理 |
| 资产管理 | 是 | 否 | 一级可改名，二级固定 |
| 任务管理 | 是 | 是 | 公司壳工作任务，朝廷壳政务中心 |
| 系统管理 | 否 | 否 | 名称、说明、排版、二级菜单全部固定 |

资产管理二级分类永远固定：

```text
大脑
账号
钱包
硬件
知识库
```

系统管理永远固定，建议二级菜单：

```text
模型设置
技能包
MCP 服务
插件
记忆设置
安全策略
日志与审计
备份恢复
壳与外观
开发者设置
关于系统
```

## 智能体独立性原则

每个智能体默认只拥有：

```text
自己的名字
自己的头像
自己的大脑
自己的人格
自己的私有记忆
当前会话上下文
默认工具集合
默认技能集合
```

它不天然知道：

```text
完整组织结构
所有部门
所有成员
所有资产
其他智能体的私有记忆
系统密钥
全部知识库内容
```

它要感知系统资源，必须通过：

```text
Resource API
Asset Broker
Capability Graph
Tool 调用
MCP Server
知识库检索
Skill 执行
用户授权
Context Gateway 注入
```

这样做的好处：

| 好处 | 说明 |
|---|---|
| 降低上下文污染 | 不把全量组织信息塞进 prompt |
| 降低越权风险 | 所有资源访问都有句柄和策略 |
| 提升可解释性 | 任务回放能看到资源如何被发现和使用 |
| 支持多壳 | 底层资源不和世界观绑定 |
| 支持多成员协作 | 每个成员像真实员工一样按权限获取信息 |

## Skill、Tool、MCP、Asset 的分工

最终分工如下：

| 概念 | 职责 | 示例 |
|---|---|---|
| Asset | 系统里的资源实体 | 小红书账号、ETH 钱包、客厅灯、项目知识库 |
| Resource API | 查询资源是否存在、能否使用、摘要信息 | 查询“运营部有哪些账号可用” |
| Capability Graph | 判断谁能对什么资源做什么 | 墨白可以用小红书账号生成草稿，但发布要确认 |
| Tool | 一次具体动作 | 读取文件、打开网页、点击按钮、发送请求 |
| MCP | 外部能力协议和服务接入 | Playwright MCP、Home Assistant MCP |
| Skill | 可复用的做事方法 | “用小红书账号发草稿”“整理下载文件夹” |
| Plugin/Bundle | 打包能力 | 一个包里包含 Skill、MCP 配置、脚本、权限声明和测试 |

核心结论：

> 资源感知不应全部交给 Skill。资源发现走 Resource API、Asset Broker、Capability Graph、MCP 和 Tool；Skill 负责把资源、工具和步骤组织成可复用方法。
