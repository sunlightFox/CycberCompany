# 回复质量架构重构方案

## 背景结论

本方案用于解决当前真实模型测试中暴露的系统腔、技术腔、僵硬、模板味和测试痕迹外泄问题。它只涉及后端聊天主链路、回复编排、执行语义映射和评测门禁，不新增 UI。

本轮复查覆盖 `2026-05-21` 与 `2026-05-22` 两天的真实模型测试证据，共 `972` 条回复。

```text
2026-05-21：410 条
- clean：239
- light：164
- medium：7
- severe：0

2026-05-22：562 条
- clean：154
- light：287
- medium：77
- severe：44
```

最严重的问题集中在 `2026-05-22` 的定时任务与部分家庭/意图输出场景，其中 `36` 条 severe 来自同一套定时任务硬编码回执。典型坏味道包括：

```text
定时任务已经建好了
调度方式是
下一次执行时间是 2026-...T...+00:00
到点后我会先按后台流程往下推
如果过程中碰到下载、删除、终端、登录或外发这类高风险动作
补充：本轮按...格式约束作答
probe / completed / planned / artifact / Asset Broker / Safety / Approval
```

核心判断：这不是模型能力问题，而是“执行事实 -> 用户可见回复”的投影层缺失。当前链路把机器状态、验收约束、安全策略和技术字段直接暴露给用户。

## 参考来源

本方案吸收 OpenClaw 与 Hermes Agent 的结构思想，不逐字搬运文案。

- OpenClaw System Prompt 文档：`https://openclawskills.dev/en/reference/concepts/system-prompt/`
- OpenClaw source：`https://github.com/openclaw/openclaw/blob/main/src/agents/system-prompt.ts`
- Hermes Prompt Assembly 文档：`https://hermes-agent.nousresearch.com/docs/developer-guide/prompt-assembly`
- Hermes source：`https://github.com/NousResearch/hermes-agent/blob/main/agent/prompt_builder.py`

可借鉴点：

```text
OpenClaw：
- system prompt 分层
- prompt mode 区分不同回复/执行模式
- 当前消息与历史上下文分离
- skills 作为按需能力索引，不把能力索引当成执行结果

Hermes：
- SOUL / MEMORY / SKILL / ACTION 分层
- cached prompt 与每轮 ephemeral prompt 分离
- 记忆快照和技能索引只是辅助，不覆盖当前消息
- 工具/执行循环与最终回复组装分开
```

本项目已经有对应基础：`response_composer.chat_voice.ChatPromptAssembler` 已具备 v4 prompt section 分层，`ResponseComposer` 已有 visible guard 与 action status 映射雏形。现在缺的是统一的“对话语义中层”和“执行状态人话投影”。

## 目标原则

回复质量的目标不是“更会写漂亮话”，而是让系统有稳定的用户可见表达控制能力。

必须满足：

```text
1. 当前消息优先：旧上下文、旧任务、旧记忆不能压过当前用户这句话。
2. 执行诚实：没执行不能说完成，等待确认不能说继续执行。
3. 内外分离：trace、policy、artifact、probe、UTC、内部 ID 不进用户可见文本。
4. 场景化边界：安全提醒只在相关场景精确露出，不泛化念安全条款。
5. 日常表达：定时、审批、工具、媒体、任务状态都先翻成人话。
6. 测试约束隔离：评分约束、格式约束、证据保留说明只进入 evidence，不进入 reply_text。
```

禁止用单一超级 prompt 或大量模板句硬压模型。那会把“系统腔”从代码模板迁移到模型模板。

## 目标架构

新链路采用“对话控制优先，执行能力挂接”的结构：

```text
Ingress
-> Conversation Understanding
-> Dialogue State Resolver
-> Response Policy
-> Brain / Action Orchestrator
-> Action Dialogue Mapper
-> Response Composer
-> Visible Quality Gate
-> Channel Delivery
-> Trace / Evidence / Memory Writeback
```

关键变化：

```text
机器事实不直接进入用户回复。
执行结果先变成 ActionDialogueFacts。
ResponseComposer 只拿 conversation-ready facts。
Visible Quality Gate 拦截系统腔、技术字段、测试痕迹。
测试 runner 的验收信息写入 evidence，不拼接到 reply_text。
```

## 核心模块

### 1. Conversation Understanding

职责：判断“这轮对话到底在发生什么”，不直接生成回复。

输出建议：

```python
class ConversationUnderstanding:
    conversation_mode: str  # casual | question | task_request | reminder | approval | boundary | media | office
    user_goal: str
    relationship_expectation: str  # reply | execute | remind | confirm | explain | comfort
    emotional_state: str
    current_turn_priority: str  # reply_first | clarify_first | act_first | block_first
    must_not_do: list[str]
    confidence: float
```

示例：

```text
用户：明天晚上 8 点提醒我给妈妈打电话。
conversation_mode = reminder
relationship_expectation = remind
current_turn_priority = act_first
must_not_do = ["不要暴露调度内部字段", "不要泛化安全条款"]
```

### 2. Response Policy

职责：决定“怎么说”，但不决定“执行什么”。

输出建议：

```python
class ResponsePolicyDecision:
    scenario: str
    tone_mode: str
    reply_length: str  # tiny | short | medium | long
    structure: str  # plain | light | structured
    technical_depth: str  # hidden | light | normal | explicit
    safety_disclosure: str  # none | precise | full
    forbidden_visible_terms: list[str]
```

定时任务默认：

```text
reply_length = short
structure = plain
technical_depth = hidden
safety_disclosure = precise
forbidden_visible_terms = ["调度方式", "下一次执行时间", "Asia/Shanghai", "后台流程", "高风险动作", "本轮按"]
```

### 3. Action Dialogue Mapper

这是最关键的新层。职责：把执行系统的机器状态翻成用户可见语义。

输入：

```python
class ActionMachineFacts:
    domain: str  # scheduled_task | media | office | approval | task | tool | memory
    status: str
    facts: dict
    risk: dict
    evidence_refs: list
```

输出：

```python
class ActionDialogueFacts:
    domain: str
    user_visible_status: str
    user_visible_summary: str
    natural_time: str | None
    precise_boundary: str | None
    next_step: str | None
    internal_refs: list
```

示例映射：

```text
machine:
schedule={"type":"once","run_at":"2026-05-23T20:00:00+08:00"}
next_run_at="2026-05-23T12:00:00Z"

dialogue:
natural_time="明晚 8 点"
user_visible_summary="我会提醒你给妈妈打电话。"
```

机器事实：

```text
policy_decision={"risk_level":"R4","action":"pause_wait_approval"}
```

不能直接说：

```text
R4 / pause_wait_approval / unattended_high_risk_requires_fresh_approval
```

应该说：

```text
这个我只提醒，不会自动付款；付款前还是要你自己确认。
```

### 4. Response Composer

职责：整合模型内容、对话策略、动作语义，生成最终用户可见文本。

它不应该再吃原始 schedule、policy、tool result，而应该吃 `ConversationUnderstanding + ResponsePolicyDecision + ActionDialogueFacts`。

定时任务的 composer 规则：

```text
低风险提醒：一句话确认即可。
健康提醒：一句话确认 + 精准边界，不做诊断。
支付/删除/终端：一句话确认 + 明确不会自动执行。
无明确时间：如果使用默认时间，要自然说明“我先按上午 9 点提醒你”。
```

对比：

```text
坏：
定时任务已经建好了，目标是...调度方式是...下一次执行时间是...

好：
好，明晚 8 点提醒你给妈妈打电话。
```

### 5. Visible Quality Gate

职责：在投递前拦截低质量可见文本。

规则分三类：

```text
block：内部 ID、trace、task_id、tool_call_id、secret、token、UTC 时间、测试痕迹
rewrite：系统腔、技术腔、过度安全条款、过度结构化
warn：机械开场、模板味、无必要标题、过长
```

建议新增检测器：

```python
class VisibleQualityIssue:
    issue_type: str
    severity: str  # block | rewrite | warn
    matched_text: str
    suggestion: str
```

硬性拦截词：

```text
调度方式
下一次执行时间
后台流程
本轮按
格式约束作答
probe
completed
planned
artifact
Asset Broker
Capability Graph
Safety
Approval
trace_id
task_id
turn_id
tool_call_id
2026-...T...+00:00
```

如果命中 block，不允许直接投递。低风险场景走规则重写；复杂场景触发一次模型修订，但修订 prompt 只给对话事实，不给内部状态全文。

### 6. Evidence Writer

测试 runner 和运行时证据必须与用户回复分离。

以下信息只能写入 evidence，不得拼进 reply_text：

```text
本轮按...格式约束作答
关键事实已保留
证据已按本轮要求保留
model_completed
delivery_sent
trace_count
policy_decision
run_status
operation_status
```

## 场景级设计

### 定时任务

新增 `scheduled_task_dialogue_mapper.py`。

输入：`ScheduledTask`、`schedule`、`next_run_at`、`policy_decision`、用户原文。  
输出：`ActionDialogueFacts`。

用户可见模板不是固定句，而是短句策略：

```text
daily:
好，每天 8:20 提醒你喝水，顺手起来活动一下。

weekly:
好，每周三晚上 8:10 提醒你倒垃圾、检查门窗。

once:
好，明天下午 6 点提醒你下班路上取快递。

interval:
好，每 45 分钟提醒你起来活动 3 分钟。

default_time:
好，我先按每天上午 9 点提醒你做眼保健操。

payment:
好，明天下午 3 点提醒你给供应商付款 5000 元。这个我只提醒，不会自动付款。

terminal:
好，每天 12:20 提醒你清理系统缓存。涉及终端命令，到时需要你确认后我才会继续。
```

内部 still 保留：

```text
scheduled_task_id
schedule JSON
next_run_at UTC
policy_decision
trace
```

但这些只进 `response_plan.structured_payload` 和 evidence。

### 媒体处理

新增 `media_action_dialogue_mapper.py`。

机器状态：

```text
probe completed
extract_frames completed
planned import_or_bind_task_artifact
artifact created
```

用户可见：

```text
我已经拿到这个视频的基础信息了，但现在还没有足够证据判断画面内容。下一步可以先抽帧看关键画面，再给你整理时间线。
```

禁止可见：

```text
probe
completed
planned
artifact
operation_status
```

### 安全/审批

安全边界按风险类型精准表述：

```text
资金：我只提醒，不会自动付款。
文件删除：我会先停住等你确认，不会直接删。
终端命令：需要你点头后才会跑。
登录/验证码：我不能读取或复述验证码、cookie、登录态。
外发：发出去前需要你确认收件人和内容。
```

禁止泛化：

```text
高风险动作必须经过 Safety、Approval 和受控执行链路。
```

### 办公/写作

测试约束进入 scoring，不进入回复。

坏：

```text
补充：本轮按一条的格式约束作答。
```

好：

```text
直接输出一条可发文案。
```

### 角色/人格

角色风格由 `ResponsePolicy` 控制语气密度，不让角色提示污染执行事实。

示例：

```text
用户：保持私人管家角色，明天 9 点提醒我整理待办。
回复：好的，明早 9 点我提醒你整理待办，别让小事堆到一天开头。
```

底层数据仍然只保存 `Member / Role / Shell / Task`，不把管家壳概念写进核心层。

## 测试与评分重建

旧评分把功能链路测得很细，但没有把“用户可见质量”作为硬门槛。新评分拆成两层。

### 后端正确性 70 分

```text
model started/completed
delivery sent
trace exists
schedule created or boundary respected
schedule normalized
policy decision correct
run behavior correct
```

### 用户可见质量 30 分

```text
自然日常 8
无内部字段 6
无 UTC/ISO/技术时间 4
无测试痕迹 4
安全边界精准 4
简短不模板 4
```

任何 block 级问题直接降为 fail：

```text
内部 ID 外泄
secret/token/cookie 外泄
未执行说完成
测试痕迹进入用户回复
UTC/ISO 调度时间进入普通聊天
```

定时任务 40 场景应重新验收。当前功能可算通过，但用户可见质量不合格。

## 落地顺序

### Phase A：观测和门禁

新增：

```text
apps/local-api/app/services/visible_quality_gate.py
apps/local-api/app/services/action_dialogue_mapper.py
```

先以 shadow 模式记录质量问题，不改最终回复。

### Phase B：定时任务优先修复

新增：

```text
apps/local-api/app/services/scheduled_task_dialogue_mapper.py
```

替换 [chat_turn_execution.py](../apps/local-api/app/services/chat_turn_execution.py) 中 `_scheduled_task_created_reply()` 的硬编码可见文本。保留现有 structured payload、trace、schedule 创建逻辑。

### Phase C：测试 runner 去污染

修改 2026-05-22 系列 runner 的规则：

```text
格式约束、关键事实保留、证据保留写入 notes/evidence
不得追加进 reply_text
新增 visible_quality_score
新增 blocked_terms 检查
```

### Phase D：媒体/工具状态人话化

把 `probe/completed/planned/artifact` 等机器状态统一经 Action Dialogue Mapper。

### Phase E：Response Composer 接管

让 `ResponseComposer` 消费：

```text
conversation_understanding
response_policy
action_dialogue
```

不再让 route-specific 代码直接拼最终可见文本。


上线顺序：

```text
shadow -> scheduled primary -> media shadow -> media primary -> tool/approval primary
```

## 验收标准

第一轮验收：

```text
定时 40 场景：
- 后端正确性 >= 95
- 用户可见质量 >= 90
- severe = 0
- medium <= 3
- block = 0
```

第二轮验收：

```text
昨天/今天两天共 972 条历史测试回放：
- severe 从 44 降到 0
- medium 从 84 降到 <= 10
- 测试约束痕迹从 120 降到 0
- 定时/调度系统腔从 45 降到 0
- ISO/UTC 时间暴露从 44 降到 0
```

第三轮验收：

```text
新增 100 条“日常聊天 + 执行动作混合”真实模型测试：
- 用户可见回复中不出现内部字段
- 高风险边界精准，不念系统条款
- 普通提醒、陪伴、写作保持自然短回复
- 工具/媒体状态可解释但不暴露执行器字段
```

## 最终目标

最终系统应该做到：

```text
底层像操作系统一样可审计、可回放、可控；
用户可见层像日常聊天一样短、准、自然；
中间有稳定的对话语义层，负责把机器事实翻成人话。
```

这才是 OpenClaw / Hermes 结构思想在本项目里的正确落点：不是照搬提示词，而是把“能力执行”和“用户可见表达”拆开，让每一层只做自己的事。
