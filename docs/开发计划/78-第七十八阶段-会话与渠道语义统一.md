# 第七十八阶段 - 会话与渠道语义统一

## 阶段定位

本阶段目标是把本地、微信、飞书等入口统一到同一套 session / conversation / ingress 语义上，避免会话连续性、安全隔离和渠道回放继续散落在各渠道私有逻辑里。

## 目标问题

需要统一的不是“渠道 API 细节”，而是以下运行时语义：

```text
一个用户从哪里发来消息
这条消息进入哪个 session
这个 session 与哪个 conversation 绑定
这个 session 是否允许跨渠道续接
一条消息是否是重放、补发、重复或延迟送达
```

## 核心对象定义

### session

`session` 是聊天运行时的最小连续性边界，代表：

```text
当前一段可连续推进的对话运行态
当前 pending action、clarification、recent continuity 的归属单位
当前渠道入站与后台任务回写的绑定单位
```

特点：

```text
session 可以跨多个 turn
session 不一定等于数据库 conversation
session 强调“当前进行中的连续性”
```

### conversation

`conversation` 是长期可存档、可查询、可回放的对话容器，代表：

```text
消息序列的长期归档空间
message / turn / summary / memory source 的关联容器
```

特点：

```text
conversation 是持久化对象
一个 session 必须归属一个 conversation
同一 conversation 在受控条件下可以承接多个 session
```

### channel peer session

`channel peer session` 是渠道层的对端绑定对象，代表：

```text
某个外部渠道对端与本地 session/conversation 的路由映射
```

它不等价于聊天 runtime 的 session，而是渠道适配层的持久桥接记录。

## direct / peer / thread / group / channel 隔离规则

### local

```text
本地 CLI / API 默认按 conversation 显式指定
未指定 conversation 时创建新的本地 session，并绑定默认 conversation
```

### direct / peer

```text
私聊对端默认隔离到单一 peer session
同一渠道、同一 account、同一 peer_ref 进入同一 peer session
peer session 默认续接其绑定的 session/conversation
```

### thread

```text
thread 必须独立于父 channel session
同一 thread_id 的消息进入同一 session
没有 thread_id 的同频道消息不得混入 thread session
```

### group / room / channel

```text
群组或频道消息默认按 group/channel 容器隔离
是否再按 sender 细分，只允许通过显式渠道策略启用
默认不允许把两个不同 group/channel 混入同一 session
```

### system / webhook / worker

```text
后台 worker、通知回写、hook、webhook 事件不能创建新的用户连续性 session
它们只能附着到已有 session/conversation 或落到受控的系统 session
```

## session 与 conversation 的关系

固定规则：

```text
每个 session 必须绑定一个 conversation_id
一个 conversation 可以先后承载多个 session
session rollover 允许继续写同一 conversation，也允许按策略切换新 conversation
pending action 绑定的是 session，不是整个 conversation
memory source 必须记录 conversation_id 与 source_turn_id
```

### 不允许的情况

```text
一个 session 同时归属多个 conversation
一个 turn 在执行中途切换 conversation_id
渠道入站直接覆写已存在 session 的 conversation 归属
```

## 渠道入站标准化字段

`ChatIngressMetadata` / `ChannelSessionRoute` 必须统一支持以下字段：

```text
channel
channel_message_id
channel_account_id
channel_peer_id_redacted
channel_thread_id
delivery_mode
queue_policy
dedupe_key
source_timestamp
raw_payload
```

### 字段语义

#### channel

```text
固定为 local / wechat / feishu / future channels
```

#### channel_message_id

```text
渠道原始消息唯一键
用于重放与去重
```

#### channel_account_id

```text
本地接入的渠道账号标识
必须与 peer 路由分离
```

#### channel_peer_id_redacted

```text
对端标识的脱敏版本
用于日志、trace、审计、去重关联
不得把原始 peer_ref 直接注入模型上下文
```

#### channel_thread_id

```text
thread / topic / reply chain 的隔离键
没有 thread 能力时为空
```

#### delivery_mode

```text
dm
group
channel
thread
system
```

#### queue_policy

```text
immediate
collect
followup
steer
```

#### dedupe_key

```text
同一渠道重复投递识别键
由 channel + account + peer + thread + message_id 组合生成
```

## 会话生命周期

### 创建

仅以下情况允许创建新 session：

```text
本地显式新建 conversation
渠道首次配对的 direct / peer
group / thread 首次入站
旧 session 已失效且策略要求 rollover
retry turn 明确创建新运行 session
```

### 绑定

```text
渠道 peer session -> runtime session
runtime session -> conversation
后台 delivery binding -> session / turn
```

### 续接

默认续接条件：

```text
同一 channel_account_id
同一 peer/group/thread 隔离键
消息未命中 dedupe
session 未被显式 revoke / invalidate
```

### 失效

session 失效原因固定为：

```text
manual_reset
security_revoke
channel_rebind
idle_rollover
daily_rollover
corruption_recovery
```

### 恢复

恢复只允许：

```text
修复 running/queued 残留 turn
恢复 session 到 idle/closed/recovered 的一致状态
不允许在恢复过程中悄悄改绑 conversation
```

## 跨渠道复用 conversation 规则

默认策略：

```text
不允许自动跨渠道复用 conversation
```

允许跨渠道续接的唯一条件：

```text
存在显式 identity link 或显式 session binding
用户或系统已授权该跨渠道桥接
渠道身份映射可审计、可撤销、可回放
```

即使允许跨渠道复用 conversation，也必须满足：

```text
新渠道创建自己的 channel peer session
runtime session 重新建立
旧渠道 pending action 不得被新渠道模糊确认释放
```

## 去重、重放、防串线规则

### 去重

每条入站消息必须先按 `dedupe_key` 检查。

规则：

```text
已处理消息不得重复创建 turn
重复消息可记录为 duplicate inbound audit
不允许因为重复投递触发二次工具执行
```

### 重放

可重放消息分为：

```text
渠道重试投递
worker 重投
系统恢复后补拉
```

处理规则：

```text
可重放只允许补齐状态，不允许二次执行高风险动作
```

### 防串线

以下情况必须 fail closed：

```text
peer 匹配到多个 session
thread 与非 thread 命中同一 session
一个 pending action 被多个渠道模糊确认
session 与 conversation 绑定关系不一致
```

## 后续新增渠道必须遵循的 ingress contract

新增渠道接入必须满足：

```text
先完成原始消息 -> standardized ingress metadata
再完成 session route
最后调用 ChannelIngressRuntime -> SessionRuntime
```

明确禁止：

```text
新增渠道自己复制一套 turn 创建逻辑
新增渠道自己判断 direct / tool / task
新增渠道直接释放 pending approval
```

## 验收标准

```text
local / wechat / feishu 全部进入统一 session/conversation 语义
peer/group/thread 隔离规则明确且可测试
跨渠道默认不串 conversation
pending action 只绑定唯一 session
重复投递不会重复执行高风险动作
```

## 最小测试集

```text
微信同一 peer 两次入站正确续接 session
飞书同一 peer 两次入站正确续接 session
不同 peer 不得串 conversation
同一 group 下不同 thread 不得串 session
重复 inbound message 不得重复 create_turn
跨渠道未配对时不得复用 conversation
存在显式绑定时允许跨渠道进入同一 conversation，但 pending action 不跨渠道误释放
```
