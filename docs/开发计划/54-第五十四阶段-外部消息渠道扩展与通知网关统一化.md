# 第五十四阶段 - 外部消息渠道扩展与通知网关统一化

## 阶段背景

第五十三阶段已经把微信真实绑定、账号资产和通知网关接通，但网友最常见的期待并不止于微信，而是“把我常用的消息入口都接进来，让 Agent 在我习惯的地方收发消息、确认动作和回传结果”。当前仓库对外部消息的后端契约已经有基础，但平台覆盖、入站标准化、审批回复和失败语义还需要继续统一。

本阶段继续只做后端、schema、migration、repository、service、API、tests、evals 和文档；不新增前端页面、组件、样式、Tauri 窗口或桌面端交互代码。

## 参考结论

### OpenClaw 采用点

OpenClaw 的强项是多入口 Gateway 思路：

```text
多聊天入口统一接入
不同平台共享同一执行和审批语义
渠道状态、绑定状态和消息状态可诊断
```

本项目采用：

```text
所有外部消息统一进入 Notification Gateway / Channel Gateway
渠道 provider 只负责适配，不负责业务决策
消息、审批回复、失败通知走同一套 trace 与审计
```

### Hermes Agent 采用点

Hermes 的强项是会话级 pending action 和外部消息回路：

```text
用户在消息平台里直接确认、拒绝或继续
外部消息可回流到 pending action resolver
消息平台不是单向通知，而是交互入口
```

本项目采用：

```text
外部消息平台可作为入站交互通道
审批回复保持单动作绑定
确认语义不依赖复制技术 ID
```

## 核心目标

本阶段完成后，后端应支持：

```text
微信之外的外部消息渠道 provider 接入统一 ChannelConnector interface
飞书、Telegram、Slack、邮箱等平台的入站事件标准化
通知、审批、回执、失败重试和人工回复统一进入 Notification Gateway
外部消息会话与内部 chat turn、task、approval、asset 之间建立稳定映射
所有外部消息都保留 trace、audit 和 redaction
```

## 阶段原则

1. 外部平台是渠道，不是产品核心模型。
2. 渠道适配器只做协议转换，不做权限越界。
3. 入站消息默认不可信，不能直接触发高风险动作。
4. 审批回复必须绑定 pending action，不允许泛化口令误触发。
5. 通知失败必须诚实返回 provider_unavailable、retryable_failure 或 rejected，不伪装 sent。
6. 群聊、频道、私聊等不同会话类型必须有明确语义隔离。

## 阶段范围

### 本阶段必须完成

```text
Notification Gateway 的多平台 provider 适配统一化
ChannelInboundEvent / ChannelOutboundEvent 统一 schema
外部平台会话映射与消息幂等去重
审批回复、拒绝、继续、取消等文字语义解析
消息发送失败重试、死信、回执和审计
外部消息 trace、audit、redaction 和 replay 证据
```

### 本阶段不做

```text
不新增外部平台 UI
不把平台 token、cookie、session 直接暴露给模型
不允许渠道插件越过 Asset Broker 或 Capability Graph
不把群聊自动发言作为默认能力
```

## 主要待补

```text
多平台 provider 注册表
会话级入站标准化
审批回复解析器
回执和失败语义
渠道健康检查与版本诊断
```
