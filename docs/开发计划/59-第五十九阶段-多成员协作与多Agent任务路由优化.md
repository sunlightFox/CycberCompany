# 第五十九阶段 - 多成员协作与多 Agent 任务路由优化

## 阶段背景

我们的核心模型不是单一聊天机器人，而是 Organization / Member / Role / Shell / Asset / Skill / Task 的组织化系统。OpenClaw 和 Hermes 都强调多会话、多 profile、多执行上下文；而我们这里更进一步，要让多个成员、多个任务、多个执行后端在同一组织中稳定协作，不互串数据，不互串权限，不互串记忆。

本阶段继续只做后端、schema、migration、repository、service、API、tests、evals 和文档；不新增前端页面、组件、样式、Tauri 窗口或桌面端交互代码。

## 参考结论

### OpenClaw 采用点

OpenClaw 的强项是 workspace / sender / agent 边界清晰：

```text
不同执行者有独立上下文
不同会话能共享同一工作空间语义
动作与身份分层清楚
```

本项目采用：

```text
组织、成员、任务和会话保持稳定映射
壳只影响展示与模板，不自动改底层值
```

### Hermes Agent 采用点

Hermes 对 profile / session / long-term growth 的理解更偏“多角色持续协作”：

```text
不同 profile 有不同记忆和习惯
不同会话能接续同一任务链
```

本项目采用：

```text
多成员之间的任务分工、接力和回放可追踪
不同成员的能力、资产和记忆严格隔离
```

## 核心目标

本阶段完成后，后端应支持：

```text
多成员并行任务与接力任务
任务路由按成员能力、资产和风险自动选择
协作任务的状态、审批和证据统一管理
不同成员之间的记忆、资产和工具权限不串用
跨会话的协作上下文可回放
```

## 阶段原则

1. 成员是独立主体，不是一个共享提示词槽位。
2. 路由依据能力图、风险和上下文，而不是固定写死身份。
3. 协作任务必须有明确 owner、host 和 trace。
4. 接力动作必须记录来源和去向。
5. 壳系统只能改称谓，不改底层事实。

## 阶段范围

### 本阶段必须完成

```text
多成员任务分配、接力和回收
多 agent / 多 profile 的任务路由规则
组织级共享资产与个人私有资产边界
协作任务的 trace、approval 和 replay
成员间上下文摘要和边界治理
```

### 本阶段不做

```text
不让一个成员直接读取另一个成员的私有记忆
不让壳系统自动修改真实角色字段
不把协作任务伪装成单人任务
不新增 UI
```

## 主要待补

```text
成员路由
协作 replay
资产隔离

```
