# 第五十三阶段 - 资产中心通讯渠道与微信真实对接后端

## 阶段背景

第四十阶段已经建立 Notification Gateway，支持通知渠道、出站消息、入站消息、审批回复和失败重试。第四十八阶段继续把 notification pending action 与 approval/task resume 串起来。现在需要补齐用户在资产中心新增“通讯”资产时的真实接入体验：

```text
用户选择连接微信渠道
后端创建绑定会话
返回二维码给客户端展示
用户用手机微信扫码确认
后端收到绑定成功事件
系统把该微信身份登记为账号资产和通讯渠道
后续通知、外部消息、审批确认都通过同一个受控渠道运行
```

本阶段的开发内容不是只冻结接口或提供 mock，而是把微信渠道作为真实可用的通讯 provider 接入后端：通过受控 Wechat Connector/sidecar 完成扫码绑定、会话状态保存、私聊出站通知和入站消息标准化。`wechat_mock` 只作为自动化测试和无外部依赖 CI 的替身，不能替代 `wechat` provider 的真实对接验收。

本阶段继续只做后端、schema、migration、repository、service、API、tests、evals 和文档；不新增前端页面、组件、样式、Tauri 窗口或桌面端交互代码。

## 参考结论

### OpenClaw 采用点

OpenClaw 的渠道设计值得借鉴：

```text
多通讯平台统一接入 Gateway
WeChat 通过外部插件接入，核心保持 channel-agnostic
微信登录由插件启动二维码流程
登录成功后保存本地账号状态
入站消息被标准化为统一 channel contract 后再路由给 agent
DM pairing 与 allowlist 用于安全控制
```

本项目采用：

```text
微信能力放在 ChannelProvider / Connector 层
核心只依赖统一通讯渠道接口
二维码绑定是资产绑定会话，不是普通工具调用
绑定成功后沉淀为 Asset(account) + NotificationChannel
真实 wechat provider 必须通过受控 connector/sidecar 对接，不降级为 mock 成功
```

不照搬：

```text
不把 OpenClaw CLI 命令作为产品接口
不把微信 token 暴露给模型、Skill 或普通 API
不让渠道插件拥有全系统资源
不把未配置或未通过健康检查的 connector 当成可用微信渠道
```

### Hermes Agent 采用点

Hermes Agent 的 Gateway 思路值得借鉴：

```text
平台 adapter 把原始事件标准化为 MessageEvent
session key 包含 platform / chat_type / chat_id
消息进入运行中会话时有 guard 和 queue
/approve /deny /stop 等命令可绕过普通 agent 队列，直接处理 pending action
授权默认 deny，通过 allowlist 或 DM pairing 放行
```

本项目采用：

```text
ChannelInboundEvent 统一封装微信、飞书、Telegram 等渠道事件
会话定位使用 organization_id + channel_id + account_binding_id + peer_ref
审批确认、拒绝、停止等命令先进入 pending action resolver
外部消息默认不可信，不能直接触发任意工具
```

不照搬：

```text
不使用斜杠命令作为唯一交互方式
不把通讯会话直接等同于内部聊天会话
不允许 allow-all 作为默认配置
```

## 核心目标

本阶段完成后，后端应支持：

```text
资产中心新增 asset_type=account、asset_subtype=communication_channel 的通讯资产语义
新增微信渠道 provider：wechat，并对接真实 Wechat Connector/sidecar
创建微信绑定会话并从真实 connector 返回二维码内容、二维码图片数据或二维码 artifact 引用
支持绑定状态轮询和事件回调：pending、qr_ready、scanned、confirmed、bound、expired、failed、revoked
微信绑定成功后自动创建或关联 NotificationChannel
微信渠道可作为 Asset Broker 查询到的 message_channel handle
后续飞书、Slack、Telegram、邮箱等渠道复用同一 ChannelConnector interface
入站微信消息进入 NotificationGateway inbound pipeline 或 ChannelInbound pipeline
出站微信通知通过真实 connector 发送；connector 不可用时必须明确 provider_unavailable/failed，不能伪装 sent
出站微信通知继续经过 Asset Broker、Capability Graph、Safety、DLP、Approval、Trace
所有二维码、token、session、联系人 ID、消息内容按敏感等级脱敏、trace 和审计
wechat_mock 仅用于测试，不作为生产 provider 的成功路径
```

## 阶段原则

1. 通讯渠道是账号资产，不新增第六类资产二级分类。
2. 资产中心可以在“账号”下出现通讯渠道 subtype；壳系统不能改变底层类型。
3. 连接微信是绑定会话，不是直接创建已绑定资产。
4. 二维码只作为短期绑定挑战，不长期保存明文可登录凭证。
5. 微信 connector 只拥有微信渠道所需最小权限，不拥有资产中心其他 secret。
6. 微信 token、cookie、iLink credential、session state 只进入 secret store 或 provider state store。
7. 入站消息全部标记 `untrusted_external_content=true`。
8. 外部一句“确认/好的”不能绕过 pending action binding。
9. 群聊能力默认关闭；首发只支持私聊或个人绑定通道。
10. 渠道不可用、二维码过期、登录失败必须有明确状态，不伪装已绑定。
11. `provider=wechat` 必须走真实 connector；只有显式 `provider=wechat_mock` 才允许使用 fake 行为。
12. 真实 connector 的启用、健康检查、版本和状态目录必须可诊断、可审计、可关闭。

## Wechat Connector 配置契约

配置建议放在 `config/channels.yaml` 或既有配置系统的 channels 段：

```yaml
providers:
  wechat:
    enabled: false
    endpoint: "http://127.0.0.1:18787"
    callback_secret_ref: "secret://channels/wechat/callback"
    state_dir: "data/channel-providers/wechat"
    timeout_seconds: 10
    min_version: "0.1.0"
    allow_mock_fallback: false
    private_chat_only: true
    group_messages: "disabled"
  wechat_mock:
    enabled: true
    test_only: true
```

约束：

```text
allow_mock_fallback 必须固定为 false；真实 provider 不允许自动落到 mock。
wechat.enabled=false 时，provider=wechat 只能返回 provider_unavailable。
callback_secret_ref 只进入 secret store，不进入普通配置回显。
state_dir 只存 provider state 引用和 connector 自有状态，不被模型、Skill、MCP 直接读取。
health API 只能返回脱敏诊断，例如 enabled、reachable、version、login_state、last_error_code。
```

首发 provider：

| provider | 实现策略 | 说明 |
|---|---|---|
| `wechat` | 真实 connector/sidecar | 本阶段主交付；负责真实扫码、登录状态、私聊出站和入站事件 |
| `wechat_mock` | 测试用 fake connector | 只用于单测、CI 和离线演练；生成测试二维码、模拟 scanned/confirmed/bound |
| `feishu` | 仅保留 provider slot | 后续阶段接入 |

`provider=wechat` 的任何成功结果都必须来自真实 connector。若 connector 未配置、未启动、版本不兼容、登录态失效或健康检查失败，绑定和发送必须返回明确失败状态，不能自动切到 `wechat_mock`。

## 微信绑定状态机

```text
created
-> qr_ready
-> scanned
-> confirmed
-> bound
```

失败或终止状态：

```text
expired
failed
cancelled
revoked
```

状态规则：

```text
qr_ready 后必须有 expires_at
expired/cancelled/failed 不能 finalize
confirmed 后 finalize 才创建 Asset 和 NotificationChannel
bound 后重复 finalize 幂等返回已绑定资产
revoke 后 Asset.status=revoked，NotificationChannel.status=revoked，secret_ref 失效
```

## 绑定成功后的内部链路

```text
ChannelBindingService.finalize_bind
-> ChannelConnector.finalize_bind
-> SecretStore.save(provider_state)
-> AssetService.create_asset(asset_type=account, asset_subtype=communication_channel)
-> CapabilityGraph.create_grant(message_send/message_receive/approval.reply)
-> NotificationGatewayService.create_channel(provider=wechat, asset_id=...)
-> ChannelAccountRepository.insert
-> Trace/Audit write
```

其中 `create_channel` 需要支持“使用既有 asset_id 创建渠道”，避免重复创建 message_channel 资产。

真实微信绑定的 provider state 至少要能表达：

```text
connector_instance_id
provider_session_ref
provider_account_ref_redacted
login_state
capabilities
last_health_check_at
expires_or_refresh_hint
```

这些字段只保存脱敏摘要或 secret/provider_state 引用；不得把 cookie、session key、二维码原始登录票据写入普通表、trace、audit 或模型上下文。

## 入站消息处理链路

```text
Wechat connector raw event
-> ChannelConnector.normalize_inbound
-> ChannelEventRepository.insert(redacted)
-> Peer policy check
-> pending action resolver
  -> approval_once / approval_deny / task_cancel 等有限动作
-> 如果不是 pending action，再按策略进入 Chat Runtime 或本地待处理 inbox
```

关键约束：

```text
未 paired peer 默认不能触发任务执行
多个 pending action 时必须 clarification
高风险 approval 必须匹配动作对象
微信群/群聊事件首发默认 rejected_or_ignored
```

## 出站消息处理链路

```text
NotificationGatewayService.create_message
-> DLP/redaction
-> Asset Broker query(message_send)
-> Capability Graph decide
-> Safety evaluate outbound
-> WechatChannelProvider.send
-> provider delivery attempt
-> notification_messages status
-> trace/audit
```

首发只允许：

```text
任务完成摘要
任务失败摘要
审批提醒
系统降级提醒
定时任务摘要
```

首发禁止：

```text
向任意联系人主动发起营销、群发或推广消息
发送包含 secret/token/cookie/private_key/mnemonic 的内容
发送钱包、支付、验证码、敏感身份信息
```

## 小阶段 53.1：契约冻结与资产语义

### 目标

把“通讯”明确为账号资产下的 subtype，而不是新增资产大类，保证资产中心五类固定规则不被破坏。

### 必须完成

```text
定义 asset_subtype=communication_channel
定义 provider 枚举：wechat、wechat_mock、feishu_reserved
定义 capability：message_channel、notification.inbound、notification.outbound、approval.reply
定义风险矩阵和默认禁止项
定义 provider capability matrix
```

### 验收

```text
资产类型仍只有 brain/account/wallet/hardware/knowledge_base
通讯渠道查询时可用 subtype 区分
```

## 小阶段 53.4：真实微信二维码绑定闭环

### 目标

### 必须完成

```text

wechat_mock 只用于测试同一状态机
```


## 验收标准

本阶段完成时必须满足：

```text
资产中心能通过后端接口表达“新增通讯资产：微信”
创建微信绑定会话时后端能从真实 connector 返回二维码结构给客户端展示
二维码绑定有完整状态机和过期/取消/失败语义
绑定成功后自动创建账号资产和通知渠道
通讯资产仍属于 account 类型，不破坏五类资产固定规则
微信渠道可通过 Asset Broker 查询到脱敏 handle
微信出站通知经过 Asset Broker、Capability Graph、Safety、DLP、Trace，并由真实 connector 发送
微信入站消息由真实 connector 回调进入，默认不可信，只能安全释放已有 pending action 或进入受控聊天入口
真实 connector 未配置、不可达、版本不兼容、登录失效时有明确 provider_unavailable/failed 诊断
provider=wechat 不能自动 fallback 到 wechat_mock
未授权 peer、多个 pending、模糊确认、无 pending 均 fail closed
解绑后 secret/provider state 失效，渠道不可继续发送
所有二维码、token、cookie、provider account id、peer id、消息敏感内容均脱敏
保留飞书等后续渠道扩展位，不把 wechat 写死到核心层
不新增任何前端 UI、Tauri 窗口或桌面端交互代码
```

## 与其他阶段关系

```text
第四阶段提供资产中心和能力权限图，本阶段扩展账号资产 subtype 和 message_channel handle
第四十阶段提供 Notification Gateway，本阶段复用其出站、入站、重试、DLP 和审批回复能力
第四十八阶段提供 pending action 治理闭环，本阶段复用外部确认 fail-closed 语义
第四十二、第四十七、第五十阶段提供外部平台 provider 插件化和浏览器适配思路，本阶段限定为通讯渠道 provider
第五十二阶段的部署/安装任务可通过微信渠道发送完成、失败、审批提醒
后续飞书、Slack、Telegram、邮箱等渠道应新增 provider，不修改 ChannelBindingService 核心契约
```

## 最终验收定义

第五十三阶段完成时，应能回答：

```text
用户想连接微信时，后端能否通过真实 connector 给出二维码让用户扫码
扫码成功后，微信是否成为资产中心里受控的通讯账号资产
微信渠道是否能通过真实 connector 用于通知、审批回复和后续外部消息入口
微信 token/session 是否从未进入模型上下文、普通 API、trace 或审计明文
入站微信消息是否默认不可信，不能任意执行工具
wechat_mock 是否只作为测试 provider，而不替代真实微信对接验收
未来接飞书时是否只需要新增 provider，而不是重写资产、通知和权限系统
```
