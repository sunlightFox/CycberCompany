# 第七十六阶段 - 聊天主链路完善总计划

## 阶段定位

这一阶段不是直接新增用户可见功能，而是为聊天主链路后续收口建立统一施工总图。现有聊天链路已经叠加了聊天质量、自然语言确认、渠道桥接、工具捷径、任务恢复、记忆与审批语义，但运行时边界仍然分散在多个服务和大量兼容层里。

本阶段目标是把后续第七十七到第八十五阶段统一收束到一条明确主链路：

```text
SessionRuntime
  -> ChatRuntime
  -> ContextGateway
  -> Brain
  -> Safety
  -> Tool/Task
  -> ResponseComposer
  -> Memory/Trace
```

## 当前现状与主要缺口

### 1. 运行时主链路尚未完全收口

当前入口已经名义上通过：

```text
routes_chat -> SessionRuntime -> ChatService
```

但实际 turn 执行、事件收尾、direct route、自然语言确认、快捷工具路径、任务恢复等关键能力仍主要堆叠在：

```text
apps/local-api/app/services/chat.py
apps/local-api/app/services/chat_turn_execution.py
apps/local-api/app/services/chat_model_execution.py
```

这意味着：

```text
ChatRuntime 仍未成为唯一真实运行路径
SessionRuntime 还是薄壳，但 ChatService 仍过重
后续任一聊天行为变更仍容易互相串扰
```

### 2. 会话语义尚未完全统一

本地、微信、飞书都能入站，但 session、conversation、peer session、ingress metadata 的边界仍是分散实现。

风险：

```text
同一用户跨渠道是否共享连续性语义不清晰
direct/group/thread/peer 的隔离策略不统一
重复消息、重放消息和延迟消息容易靠渠道特判兜底
```

### 3. Context Gateway 还偏向“会聊”，不够“会执行”

当前上下文已具备：

```text
recent history
summary
memory
persona / heart
workbench
```

但仍存在明显缺口：

```text
能力摘要不完整
资源句柄主要限于知识库
untrusted_context 仍是空壳
safety_notes 仍是固定占位
```

### 4. 聊天内工具闭环不完整

当前更多依赖：

```text
前置 route 分流
只读捷径
task handoff
```

尚未形成稳定的：

```text
model -> tool_call -> safety -> approval -> result reinjection -> final reply
```

### 5. 可见输出与内部诊断仍需进一步分层

当前已经有 visible filter、pending honesty、shadow policy，但用户可见层与内部 trace/recovery/tool diagnostics 的隔离还需要更强的统一契约，避免后续新链路继续泄漏内部运行信息。

## 对标 OpenClaw / Hermes 的结论

本阶段吸收的不是 UI 形态，而是后端运行时组织方式。

### OpenClaw 提供的关键启发

```text
一条权威 agent loop
按 session 串行化的运行时语义
明确的 lifecycle / assistant / tool 事件流
before/after prompt、tool、session 的稳定 hook 面
```

### Hermes Agent 提供的关键启发

```text
统一的 gateway session source 语义
显式的 session context prompt 注入
统一 state store 与 run ledger
跨平台工具网关与运行时一致性
```

### 本项目不照搬的部分

```text
不照搬它们的前端或 CLI 交互
不引入会破坏本项目 Organization/Member/Asset 模型的外来概念
不允许任何工具或插件绕过 Asset Broker / Capability Graph / Safety
```

## 总体目标

完成第七十六到第八十五阶段后，聊天主链路必须满足：

```text
ChatRuntime 成为唯一真实 turn runtime
SessionRuntime 只保留外部入口与生命周期入口
会话、上下文、工具、任务、回复、记忆、trace 边界清晰
聊天可以在单轮内安全完成只读工具动作
高风险动作必须显式走 approval 或 deny
所有模型、工具、审批、记忆写入都可回放、可追踪、可审计
```

## 总体验收原则

### 1. 真实性优先

任何阶段都不允许引入以下回退：

```text
未执行说成已完成
待审批说成已继续
失败说成处理好了
没有看过说成已经核实
```

### 2. 安全边界优先

```text
任何资产访问必须经过 Asset Broker
任何能力判断必须经过 Capability Graph
任何高风险动作必须经过 Safety 和 Approval
任何用户输入不得直接进入 shell command
```

### 3. Trace 完整优先

```text
模型调用、工具调用、审批、记忆写入必须可回放
任何新增 hook 和 runtime 支路都必须有 trace 归属
```

### 4. 聊天页契约不变

后端重构不允许影响以下 UI 原则：

```text
聊天页仍只显示当前聊天对象的人名、头像、状态和消息
不在聊天页暴露组织、壳、部门树、资产面板
```

### 5. 不引入临时架构债

```text
不新增未来必删的旁路服务
不把旧兼容分支继续扩散进新运行时
不允许用新文档为旧大文件继续膨胀背书
```

## 本阶段完成后的意义

完成这一阶段后，后续开发不再是“继续往 `chat.py` 塞逻辑”，而是：

```text
先按统一 runtime 改造
再按统一 session/context/tool/response/ledger 契约实现
最后按统一测试矩阵验收
```

这会把聊天主链路从“功能已经很多但运行时边界分散”，推进到“结构清晰、可持续收敛、可做最终封版”的工程态。
