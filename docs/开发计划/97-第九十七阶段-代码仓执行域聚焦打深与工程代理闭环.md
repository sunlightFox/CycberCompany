# 第九十七阶段 - 代码仓执行域聚焦打深与工程代理闭环

## 阶段定位

前面阶段已经把聊天、任务、工具、浏览器、记忆、渠道、审批、账本铺得很广。

但真正的做事能力，不能只看“支持多少域”，还要看：

**有没有一个高频真实做事域，被系统狠狠干透，做到稳定交付。**

第九十七阶段选择的聚焦域是：

```text
代码仓执行
```

原因不是它最浪漫，而是它最适合检验系统是否具备成熟代理的执行肌肉：

```text
读仓库
理解上下文
做局部改动
运行测试
根据失败修复
输出最终结果
```

## 阶段目标

```text
把代码仓执行定义为首个被深打透的高频做事域
打通 read repo -> plan -> edit -> test -> repair -> summarize 的完整闭环
让代码仓任务优先走稳定 workflow + agent hybrid，而不是松散 route 分支
把 patch、命令、测试、失败恢复、审批、工件、总结做成统一工程代理链路
形成一套固定的工程代理评测题库
```

## 为什么先选代码仓执行

```text
反馈快
成功标准清楚
失败信号直接
便于 replay 和 regression
最容易检验代理是否真的会连续干活
```

与其同时做：

```text
浏览器调研
内容整理
项目部署
外部平台动作
```

不如先把代码仓执行打透，再把模式复制到其他域。

## 本阶段范围

### 必须完成

```text
建立代码仓任务专用 route / mode / capability profile
统一 repo exploration、patch planning、command execution、test verification、failure repair 闭环
补齐代码仓任务的最小工件契约
把“改代码但不验证”判为未完成
把“测试失败后自动停机”升级为有条件自修复
形成工程代理基准任务集
```

### 明确不做

```text
不扩浏览器研究域新需求
不扩外部平台发帖、渠道回复等非代码高频域
不引入新的大型 IDE / 前端交互面
不把代码仓任务等价成任意终端任务
```

## 实施拆解

### 97.1 代码仓任务分类收口

目标：

```text
把代码任务从普通终端/项目工具请求中独立出来
```

交付：

```text
repo_readonly_request
repo_patch_request
repo_test_request
repo_fix_after_failure
repo_refactor_request
```

要求：

```text
route 决策要能识别只读、可编辑、需验证、需审批几类差异
高风险命令仍走 Safety / Approval
```

### 97.2 代码仓执行最小闭环

目标：

```text
让系统具备最小但完整的工程代理执行路径
```

交付：

```text
读仓库上下文
生成改动计划
执行 patch
运行 targeted verification
失败后局部修复
最终输出 change summary
```

要求：

```text
改代码默认必须伴随验证动作
只读任务不强制 patch
验证范围随 blast radius 变化
```

### 97.3 工程任务专用观察与恢复

目标：

```text
把代码仓做事的关键观察源和恢复手段做成专项能力
```

交付：

```text
test failure observation
lint / typecheck observation
diff summary observation
workspace dirty-state awareness
partial success recovery
```

要求：

```text
不能误删用户已有改动
不能把 unrelated dirty files 当成失败证据
失败修复优先局部改，不做无关重构
```

### 97.4 工程代理工件与结果语义

目标：

```text
让代码仓任务输出更像工程代理结果，而不是聊天摘要
```

交付：

```text
patch summary
verification summary
residual risk
files changed
not-run checks
```

要求：

```text
“已修改但未验证”必须显式标明
“测试失败但已有局部修复”必须显式标明剩余风险
不能把命令原始噪声直接暴露成用户结果
```

### 97.5 工程代理基准题库

目标：

```text
建立能持续衡量代码仓执行成熟度的固定任务集
```

交付：

```text
single-file bugfix
multi-file contract update
failing test repair
safe refactor with targeted tests
dirty workspace compatibility
approval-gated host mutation denial
```

```text

系统能稳定完成读仓库、改代码、跑验证、基于失败修复、输出结果
工程代理执行不再只是工具拼接，而是统一主链路能力
后续浏览器研究或其他做事域可以复用同一闭环模板继续扩展
```
