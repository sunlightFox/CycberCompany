# 第八十七阶段 - 动作状态机与完成态证据统一

## 阶段定位

第八十六阶段先解决“谁拥有主链路”。第八十七阶段解决“系统到底什么时候算开始做事、做到哪一步、什么时候才算真的完成”。

本阶段要把聊天里的动作型请求统一进一套状态语义，不再允许不同分支各说各话。

## 阶段目标

```text
统一动作型请求状态机
统一 approval_pending / running / partial / completed / failed 的内部语义
建立完成态证据门槛
禁止没有证据的“已完成”回复
统一 tool loop 与 task handoff 的 visible facts
```

## 重点问题

当前系统虽然已经有 tool / task / approval / response plan，但用户可见层仍存在这些问题：

```text
同类动作在不同分支里用不同词表达
有的分支会过早说“已经处理好了”
有的分支只生成计划，却没有明确说明还没执行
approval pending、tool completed、task created 的 visible 语义还不够统一
```

## 本阶段范围

### 必须完成

```text
定义统一动作状态机
建立 completed_with_evidence 判定规则
建立 partial / blocked / approval_pending 的标准事实结构
把 single-turn tool loop 和 task handoff 统一到同一状态语义层
ResponseComposer 按状态机而不是按散落 route 文案生成回复
```

### 明确不做

```text
不处理渠道去重和 no_turn
不新增新的外部平台能力
不扩展新的审批 scope 模式
不引入第二套 replay 机制
```

## 建议状态机

动作型请求内部状态统一为：

```text
requested
planned
waiting_for_approval
executing
partially_completed
completed_with_evidence
failed_with_reason
blocked_by_boundary
cancelled
```

### 关键语义

#### requested

```text
用户刚提出动作目标
系统还没真正产生活动执行
```

#### planned

```text
系统已生成计划或准备好执行参数
还没有产生真实副作用
```

#### waiting_for_approval

```text
动作尚未执行
已有明确审批对象
任何用户可见话术都不得暗示已完成
```

#### executing

```text
已经进入真实工具或任务执行
允许汇报进度
不允许提前宣称完成
```

#### partially_completed

```text
部分步骤成功，部分步骤失败或待继续
必须明确哪些部分已拿到证据，哪些没有
```

#### completed_with_evidence

```text
只有拿到证据才能进入 completed
```

## 完成态证据规则

系统只有在至少满足以下一类或多类证据后，才允许把动作写成 completed：

```text
tool result 成功返回
artifact refs 已生成
browser/page evidence 已保存
approval 已 resolve 且后续动作成功
task step / workflow result 明确完成
```

没有证据时，只允许说：

```text
已生成计划
已进入等待确认
已完成只读检查
已尝试但失败
还未实际执行
```

## 实施拆解

### 87.1 动作状态事实层统一

目标：

```text
所有 direct / tool / task / approval 分支先产出统一 facts
```

交付：

```text
action_status_facts contract
tool_status_semantics contract
task_status_semantics contract
approval_pending visible facts contract
```

### 87.2 完成态证据门槛落库

目标：

```text
completed 必须经过 evidence gate
```

交付：

```text
evidence summary builder
completed_with_evidence 判定器
partial / failed / blocked 的标准 reason_code
```

### 87.3 ResponseComposer 按状态机编排

目标：

```text
可见回复基于状态机，不基于 route-specific 手写文案
```

交付：

```text
pending / running / partial / completed / failed 的统一可见模板入口
action button 与 visible status hint 跟随状态机
```

### 87.4 honesty gate 与 regression gate

目标：

```text
阻止“没有证据却写成已完成”
```

交付：

```text
honesty regression suite
虚假完成率监控项
release gate 中加入 evidence completion 指标
```
