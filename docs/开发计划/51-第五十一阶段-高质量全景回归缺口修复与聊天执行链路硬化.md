# 第五十一阶段 - 高质量全景回归缺口修复与聊天执行链路硬化

## 阶段背景

`2026-05-01-quality` 高质量全景回归批次完成了 120 条聊天主链路用例，预检通过，结果为：

```text
PASS 108 / FAIL 12 / BLOCKED 0
待修复问题 19 个
```

证据来源：

```text
docs/测试/聊天主链路/2026-05-01-quality/07-高质量全景回归测试执行报告.md
docs/测试/聊天主链路/2026-05-01-quality/08-高质量全景回归待修复问题.md
docs/测试/聊天主链路/2026-05-01-quality/09-聊天输入回复质量总表.md
```

本轮失败不只是“接口不通”，更多是聊天主链路在真实用户体验上的偏差：意图误判、未走真实模型、越权动作被创建为任务、等待确认时声称完成、浏览器交互缺少 session/URL 绑定、终端执行缺少可读证据、自然语言确认不够稳定、拒绝和取消回复质量不足。

本阶段目标是把这些问题收敛为可开发、可验收、可回归的后端任务。仍然遵守当前仓库约束：只做后端、schema、migration、repository、service、API、tests、evals 和文档；不新增前端页面、组件、样式、Tauri 窗口或桌面端交互代码。

## 问题归类

| 安全拒绝与越权拦截缺失 | `FIX-004`、`FIX-017`、`FIX-018`、`FIX-019` | `MEM-QLT-020`、`SAFE-QLT-013`、`SAFE-QLT-014` | 跳过审批、登录别人账号、禁止创建任务等请求仍创建/完成任务 |
| 自然语言动作确认不稳定 | `FIX-005`、`FIX-013`、`FIX-014`、`FIX-015` | `TASK-QLT-005`、`BRW-QLT-022` | 下载/截图/修改参数语义串线，等待确认时说“已完成”，质量分低 |
| 浏览器交互执行绑定缺失 | `FIX-009`、`FIX-010`、`FIX-011` | `BRW-QLT-009` | `browser.click/fill/type` 因 URL 为空被安全策略拒绝 |

| 桌面原生能力边界 | `FIX-012` | `BRW-QLT-020` | 当前无 `desktop.*` 工具；回复诚实，但产品能力仍是缺口 |

## 阶段目标

本阶段完成后，聊天主链路应满足：

```text
普通聊天、知识总结、策略建议默认走真实模型或明确的非模型模板事件语义
越权、跳过审批、登录别人账号、危险操作绕过等请求必须 fail closed
用户通过文字聊天即可确认、拒绝、修改参数，但确认必须绑定唯一 pending action
等待确认、未执行、执行失败时不得声称完成
浏览器 click/fill/type/submit 必须带 session、URL、页面状态或 tool_call 绑定证据
终端 run/read_log/stop 形成可回放日志工件，读不到日志时给稳定错误和任务状态
回复编排能输出自然、准确、结构清楚、有情绪支持、有边界的聊天文本
桌面原生操作能力给出稳定后端能力边界；未实现时不伪造执行，不作为已完成能力
所有模型调用、工具调用、审批、记忆写入和任务状态变更有 trace
所有 secret、token、cookie、password、private key、本地敏感路径继续脱敏
```

## 阶段原则

1. 用户交互入口统一是文字聊天，后端可以返回结构化 pending action，但可见文本必须自然。
2. 低风险动作尽量直接推进，高风险动作只在必要处确认，避免“什么都弹确认”。
3. 确认不是泛化的“好的”，而是绑定到一个明确、可解释、可撤销的 pending action。
4. 安全拒绝要有人味：先承接用户状态，再说明边界，再给可替代做法。
5. 执行类回复必须证据优先：没有 task replay、工具结果、artifact 或页面状态，就不能说完成。
6. 工具、Skill、MCP、浏览器、终端都不能绕过 Safety、Approval、Capability Graph 和 Trace。
7. 网页内容、MCP 输出、工具日志都视为不可信输入，不能影响系统提示或审批策略。
8. 桌面原生能力未完成前只能作为明确边界或后续阶段输入，不伪装为可操作窗口。

## 核心设计

### 1. 意图识别和模式选择修复

需要把“用户要建议/总结/比较/知识说明”和“用户要执行工具/修改文件/删除资源”分开。

目标规则：

```text
输入包含“取舍、建议、策略、总结、解释、对比、方案”时，默认进入 qa/deep_reasoning
只有出现明确外部状态变更对象和动作时，才进入 task_request
“成本”“覆盖率”“速度”等工程语义不能被误识别为 filesystem/delete
低置信度工具意图必须进入模型辅助语义复核或自然澄清，不走硬编码危险分支
```

关联修复：

```text
CHAT-E2E-QUALITY-REG-FIX-002
CHAT-E2E-QUALITY-REG-FIX-001
```

### 2. 安全拒绝和越权请求 fail closed

以下请求必须不创建任务、不触发工具、不生成 approval：

```text
跳过审批直接执行危险操作
登录别人账号、冒用身份、绕过授权
要求系统假装已执行
要求在未确认时继续执行高风险动作
```

期望回复形态：

```text
我理解你着急，但这个我不能跳过确认或替你越权执行。
可以继续做两件安全的事：先帮你整理方案，或在你确认范围后走受控执行链路。
```

关联修复：

```text

CHAT-E2E-QUALITY-REG-FIX-017
CHAT-E2E-QUALITY-REG-FIX-018
CHAT-E2E-QUALITY-REG-FIX-019
CHAT-E2E-QUALITY-REG-FIX-003
```

### 3. 自然语言动作确认契约

后端保留结构化 `pending_action_id`、`approval_id`、`task_id`、`tool_call_id`，但聊天可见文本不直接暴露技术 ID。

可见文本示例：

```text
我准备下载这个 CSV：report.csv。
这会在本地生成一个文件，所以需要你确认。

你可以直接回复：
- 只允许这一次
- 不下载
- 换成这个地址：...
```

结构化契约示例：

```json
{
  "scenario": "natural_action_confirmation",
  "visible_status": "waiting_confirmation",
  "pending_actions": [
    {
      "pending_action_id": "pact_xxx",
      "action_type": "browser.download",
      "risk_level": "R3",
      "confirm_scopes": ["once", "session", "deny", "edit"],
      "visible_target_summary": "下载 report.csv",
      "evidence_required_before_done": true
    }
  ]
}
```

关键约束：

```text
一个确认回复只能释放一个当前 pending action
“确认下载这个 CSV”如果没有待确认动作，应说明缺少可确认对象，而不是声称已完成
修改参数只修改原 pending action 的 payload，不得串线成截图或其他动作
等待确认时 visible_status 必须是 waiting_confirmation，不得 completed
```

关联修复：

```text
CHAT-E2E-QUALITY-REG-FIX-005
CHAT-E2E-QUALITY-REG-FIX-013
CHAT-E2E-QUALITY-REG-FIX-014
CHAT-E2E-QUALITY-REG-FIX-015
CHAT-E2E-QUALITY-REG-FIX-006
```

### 4. 回复编排质量门

`ResponseComposer` 增加质量前置检查，不让低质量执行回复直接出现在聊天里。

最低要求：

```text
准确性：不能把未执行说成已执行
完整性：拒绝/取消/等待确认必须说明当前状态和下一步
结构：复杂信息用短段落或列表，避免一坨状态文本
自然语言：少用 task_id、approval_id、risk_level 等内部术语
情感/人格：用户着急、害怕、困惑时先承接情绪
执行诚实性：只有证据齐全才可使用“已完成”
```

建议增加 `response_quality_guard`：

```text
输入：response_plan、task_state、pending_actions、tool_evidence、safety_decision
输出：allow / rewrite / block_with_boundary
```

关联修复：

```text

CHAT-E2E-QUALITY-REG-FIX-003
CHAT-E2E-QUALITY-REG-FIX-005
CHAT-E2E-QUALITY-REG-FIX-006

```

### 5. 任务状态和证据闭环

任务只有在实际步骤执行成功、证据落盘并可回放后才能进入 `completed`。

状态约束：

```text
waiting_approval：等待用户文字确认，不得执行工具
planned：已有计划但尚未执行，不得说完成
running：正在执行，可报告进度
completed：必须有完成证据
failed：必须有失败原因和可恢复建议
cancelled：用户拒绝或取消后进入
```

执行证据最小集：

```text
tool_call_id
trace_id
task_id
input_redacted
output_redacted 或 artifact_ref
status
started_at / completed_at
failure_reason
```

关联修复：

```text
CHAT-E2E-QUALITY-REG-FIX-014
CHAT-E2E-QUALITY-REG-FIX-018
CHAT-E2E-QUALITY-REG-FIX-019
```

### 6. 浏览器交互 session 绑定

`browser.open` 应产生可复用的 browser session/page state。后续 `browser.click/fill/type/submit` 必须携带：

```text
browser_session_id 或 page_id
current_url 或 expected_url
selector / target
originating_task_id 或 tool_call_id
trace_id
```

Safety 校验不应看到空 URL；如果 URL 缺失，应返回可恢复错误：

```text
BROWSER_SESSION_REQUIRED
请先打开页面，或提供要操作的页面地址。
```

验收场景：

```text
打开本地登录页
填写账号密码
点击登录
提交表单
读取 snapshot
截图保存 artifact
验证页面文本显示登录成功或失败
```

关联修复：

```text
CHAT-E2E-QUALITY-REG-FIX-009
CHAT-E2E-QUALITY-REG-FIX-010
CHAT-E2E-QUALITY-REG-FIX-011
```

### 7. 终端日志工件

`terminal.run` 和任务引擎之间要形成稳定日志工件。

要求：

```text
高风险 terminal.run 仍需审批
审批后执行时创建 terminal_session_id / log_artifact_id
stdout、stderr、exit_code、started_at、completed_at 可通过 terminal.read_log 读取
日志不存在时要说明任务未执行、审批未通过、日志过期或 artifact 缺失
read_log 不应只返回 404 空错误，应给可恢复状态
```

关联修复：

```text
CHAT-E2E-QUALITY-REG-FIX-007
CHAT-E2E-QUALITY-REG-FIX-008
```

### 8. 桌面原生能力边界

当前阶段不新增桌面 UI/Tauri 交互代码。后端需要明确表达能力边界，让测试和产品文案一致：

```text
若未注册 desktop.* 工具，能力注册表返回 desktop_native = unavailable
聊天回复明确说明“当前不能直接操作本机窗口”
提供可替代路径：browser.* / file.* / terminal.run
测试将该行为判定为明确能力边界，而不是伪装执行
```

如未来明确要做桌面原生能力，应另起独立阶段，设计 `desktop.screenshot`、`desktop.list_windows`、`desktop.focus_window`、`desktop.minimize_window` 等 OS 级后端工具，并接入 Safety、Approval、Trace 和平台隔离。

关联修复：


## 风险和注意事项

1. 不要为了减少确认而降低高风险动作门槛；优化体验靠更准的风险识别和更自然的确认文案。
2. 不要让模型直接决定是否跳过审批；审批策略必须由 Safety/Approval 后端规则控制。
3. 不要把 `approval_id`、`task_id`、`tool_call_id` 等内部 ID 暴露成主要聊天内容。
4. 不要把 browser session 的 cookie、localStorage、密码或 token 写入模型上下文。
5. 不要把测试夹具中的失败改成通过；先修后端行为，再复跑报告。
6. 桌面原生能力如果要做，需要单独评审权限、沙箱、OS 差异和用户可见风险，不在本阶段偷偷实现。
