# 第一百一十一阶段 - 任务交付证据与可见完成态硬化

## 阶段定位

phase103 已经把任务级闭环指标接入了门禁，但当前最突出的掉分项不是“任务没跑”，而是：

```text
final_deliverable_rate 低
once_success_rate 低
visible_publish_proof_missing
completed_unverified / failed_verification 仍存在
```

这说明系统现在更像“能执行一段流程”，还不像“能对外宣称任务已完成并拿出证据”。

这一阶段不新增大块执行域，而是补齐交付证明、完成态判定和可见结果诚实性。

## 目标

```text
把任务完成从状态声明升级成证据驱动
统一不同执行域的 deliverable proof contract
降低 completed_unverified 和 failed_verification
提升 final_deliverable_rate 与 once_success_rate
```

## 重点

### 111.1 交付证据契约统一

```text
定义不同执行域最小可交付证据
内容平台需要发布可见证明
工程执行域需要产物、diff、日志或运行结果证明
外部平台动作需要可见状态回执而不是只记录“已执行”
```

### 111.2 完成态判定硬化

```text
completed 不再等于 task state machine 走到终态
必须结合 verification_status、delivery_status、artifact evidence、visible summary 一起判断
等待审批、等待接管、等待外部确认不能伪装成完成
```

### 111.3 Response / Replay / Report 对齐

```text
聊天中对用户可见的话术要和真实 delivery_status 对齐
replay、release report、task detail 页面接口使用统一完成态语义
失败时说明缺什么证据，不只说“任务失败”
```
