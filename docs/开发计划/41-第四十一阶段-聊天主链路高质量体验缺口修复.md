# 第四十一阶段 - 聊天主链路高质量体验缺口修复

## 阶段背景

第三十一、第三十三、第三十四阶段已经把真实聊天主链路、重型压力测试、Hermes Agent 式自然语言文字交互分别建立为后端能力方向。第四十阶段继续扩展外部消息渠道与通知网关。现在需要把 `CHAT-E2E-20260430-QUALITY` 高质量体验测试中暴露的问题整理成一个独立开发阶段，专门修复“能回答”和“回答得好”之间的差距。

本阶段不是继续做测试，也不是修测试脚本来让分数变绿，而是把真实模型、真实 `data`、真实 `mem_xiaoyao` 默认大脑跑出的质量缺口转成后端开发任务。核心关注用户最终看到的聊天文本：是否自然、准确、有结构、有情感、有记忆、有个性；涉及浏览器、系统、文件、任务、Skill、MCP 时，是否诚实说明真实执行结果并提供证据。

本阶段继续遵守仓库约束：

```text
只做后端能力、schema、service、API、tests、evals 和文档
不新增前端页面、组件、样式、Tauri 窗口或桌面端交互代码
不修测试数据来掩盖真实问题
不把未实现能力包装成已完成
不绕过 Safety、Approval、Trace、Asset Broker、Capability Graph
不删除当前 data 或历史会话
```

## 阶段原则

1. 用户最终看到的文本是第一验收面，不能只看后端事件成功。
2. 自然语言优先。普通用户不应被迫理解 `approval_id`、`tool_call_id`、`trace_id`、`browser.download` 等内部名词。
3. Hermes Agent 式执行模型保持不变：低风险直接推进，危险点阻塞确认，极高风险直接拒绝。
4. 明确执行意图不是跳过安全的理由；它只是进入执行链路的信号。
5. 每个动作回复必须诚实区分状态：未执行、等待确认、执行中、已完成、失败、被拒绝。
6. 记忆写入必须有 `source`，回复要说明记住了什么、用途和边界。
7. 隐私高输入不能空白；应说明已保护敏感信息，并给出用户可继续的安全方式。
8. 桌面原生操作没有后端工具时，应明确能力边界，不伪造操作成功。
9. 所有修复必须有 trace、事件、turn detail 或工具证据支撑。
10. 修复后用真实模型重跑质量批次，不用假回复替代。

## 小阶段 41.3：记忆、Persona、拒绝类回复质量升级

### 目标

修复 `MEM-QLT-001`、`MEM-QLT-005`、`MEM-QLT-013`、`MEM-QLT-015`。这些用例多数不是功能错误，而是用户体验不足：太短、太硬、过度系统化，缺少“我理解了什么、我能做什么、不能做什么、下一步怎么继续”。

### 实现要求

```text
记忆写入确认应说明记住的内容摘要、使用场景和边界
记忆遗忘能力不足时，应说明不能直接删除的原因，并提供可执行替代路径
Persona 边界回复要自然，不用过多系统术语，不声称真人或隐藏账号
拒绝系统提示/内部 trace 时，要给替代帮助，例如解释可见功能、安全原则或审计摘要
Heart 降温只调整语气，不覆盖安全边界和事实诚实
```

### 推荐回复模板

记忆写入：

```text
记好了：这批测试里，你偏好我先讲风险，再给结论。
后面同一批聊天质量分析里，我会优先按这个顺序组织回复；如果你临时改口，我会以新的要求为准。
```

遗忘边界：

```text
我不能在聊天里假装已经删除长期记忆，因为删除需要走明确的记忆管理权限和审计。
我现在能做的是：后续不再主动使用这条临时偏好；如果要真正删除，需要通过记忆管理接口处理。
```

Persona 边界：

```text
我不是现实中的真人，也没有隐藏账号可以私下登录。
我能做的是在受控任务里帮你操作浏览器、文件或工具；涉及账号、登录和外部动作时，会按安全规则确认并留下证据。
```

系统提示拒绝：

```text
这部分我不能完整输出，因为系统提示、开发者指令和内部 trace 属于运行边界。
我可以改为说明当前可见能力、审批规则、隐私保护方式，或帮你生成一份面向用户的安全边界说明。
```

## 小阶段 41.4：Hermes 式 pending action 与任务结果诚实性

### 目标

修复 `TASK-QLT-001`、`TASK-QLT-005`、`TASK-QLT-008`。任务和工具类回复不能用“任务已处理完成”这类模板糊过去，必须说明真实状态和证据。

### 实现要求

```text
任务创建后，如果只是创建/排队/等待执行，不得说已处理完成
执行完成时必须引用真实证据：artifact、snapshot、download、terminal log、task replay、trace
pending action 队列必须绑定会话、任务、动作、参数快照和风险摘要
用户说“确认下载这个 CSV”时，只有存在唯一匹配 pending download 才释放执行
没有 pending action 时，应自然说明当前没有等待确认的下载，并给出下一步
拒绝或取消时，回复要说明没有继续执行、任务状态如何、用户可以如何重新开始
```

### Hermes 式用户可见话术

```text
我准备下载 report.csv，并保存到当前任务结果里。
这会在本机生成一个文件，所以需要你确认。

你可以回复：
- 只允许这一次
- 本会话内同类下载都允许
- 拒绝下载
- 把下载地址改成：...
```

执行完成后：

```text
下载已完成，文件已保存为当前任务工件。
证据：下载记录、文件名、文件大小、任务回放步骤都已写入 trace。
```

等待确认时：

```text
还没有下载。我现在停在确认步骤，等你决定是否允许这一次操作。
```

## 小阶段 41.6：桌面、系统、浏览器能力边界契约

### 目标

修复 `BRW-QLT-020` 所暴露的产品边界：当前产品有浏览器、文件、终端等工具，但未发现独立 `desktop.*` 原生桌面操作工具。系统必须明确区分“能通过已有后端工具执行的系统操作”和“不能直接操作桌面窗口的原生桌面操作”。

### 实现要求

```text
建立桌面原生能力边界说明，避免把 OS 窗口控制伪装成已支持
若未来要支持 desktop.*，先定义 schema、capability、safety、approval、trace，不在本阶段加 UI
系统操作通过 terminal/file/browser 工具验证时，回复必须给出真实证据
浏览器动作继续沿用 snapshot、screenshot、download、URL、title、关键文本证据
桌面请求无法执行时，回复要自然、诚实，并给用户可行替代路径
```

```json
  "safe_fallbacks": ["browser.*", "file.*", "terminal.*"],
```

### 验收门槛

```text
QUALITY 预检 PASS
96 条质量用例全部落报告
10 个已知失败全部通过，或每个未通过项有明确 accepted risk 和下一阶段归属
PASS 96 / FAIL 0 / BLOCKED 0 作为本阶段封版目标
09 输入回复质量总表可人工快速检查回复质量
08 待修复问题清零或只保留非本阶段 accepted risk
报告敏感信息扫描无明文命中
不新增任何前端 UI 或桌面端交互代码
```

## 不做事项

```text

不实现未经设计的 desktop 原生控制工具
不把聊天文本写死成只通过当前测试的固定答案
不降低安全策略来减少确认
不把敏感输入透传给模型来换取回复
不清空 data 或删除历史测试记录
不让 API handler 直接执行工具
```

## 与其他阶段关系

```text
第三十三阶段负责重型压力缺口和链路硬化，本阶段负责质量体验缺口闭环
第三十四阶段定义 Hermes Agent 式文字聊天交互，本阶段把实测失败转成可落地修复
第四十阶段负责外部消息渠道，后续外部确认也要复用本阶段 pending action 和自然语言确认模型
后续若要实现 desktop.*，应另起阶段设计 OS 原生控制、沙箱、安全审批和 trace
```

## 最终验收定义

第四十一阶段完成时，应能回答：

```text
用户改口后，系统是否立即以最新指令为准
用户要求记住、纠正或忘记时，回复是否自然且诚实
用户问人格边界、系统提示、隐藏账号时，是否安全又不生硬
用户要求创建任务、确认下载、拒绝操作时，回复是否真实反映任务状态
隐私高输入是否有可恢复回复，而不是空白或泄漏
浏览器、系统、桌面能力边界是否清楚，不伪造执行
96 条质量用例是否能稳定重跑并产出输入、回复、评分、证据和结果
```
