# 第一百零四阶段 - 质量门禁修复与检查脚本恢复

## 阶段定位

前面的阶段把聊天主链路、长期记忆、扩展兼容、视频工作流和任务闭环一路铺到了 `103`，现在真正卡住持续开发效率的，不再是“有没有某个能力”，而是：

```text
门禁入口自己是否可信
本地 smoke / fast / release 检查是否还能稳定运行
脚本报告和真实 pytest 结果是否一致
失败时能不能给出明确、可复现、可定位的证据
```

第一百零四阶段不是继续扩能力，而是把“工程验证入口”本身修回可用状态。  
如果门禁脚本失真、损坏或和实际测试结果脱节，那么后续阶段的 ready、partial、passed 都会被稀释成不可靠信号。

## 阶段目标

```text
恢复 scripts/check.ps1 的可执行性、可维护性和输出可信度
建立 check.ps1、直接 pytest、release gate、readiness 之间的一致口径
把编码、参数矩阵、报告写入、失败退出语义彻底收口
让 smoke / fast / api / security / release 六类检查都具备清晰边界
让“门禁失败”优先暴露真正阻塞，而不是先死在脚本自身语法或编码问题上
```

## 核心问题

当前工程已经具备大量测试和阶段验收，但质量入口本身存在几类结构性风险：

```text
PowerShell 脚本编码损坏或字符串断裂，导致脚本无法解析
command matrix 与实际 pytest 命令逐渐漂移
报告目录、日志路径、失败摘要和退出码语义不统一
release/report 中宣称 ready，但本地检查入口并不可靠
新增阶段回归进入仓库后，没有一个稳定的最小门禁集合及时兜底
```

如果这一层不先修好，后续任何新阶段都会遇到同一个问题：

```text
开发者不知道该信 pytest、check.ps1、release report 还是 readiness
阶段“已完成”无法快速复核
失败信号被脚本自身异常掩盖
```

## 本阶段范围

### 必须完成

```text
修复 scripts/check.ps1 的语法、编码和参数解析问题
统一 smoke / fast / api / security / release profile 的职责边界
让 check.ps1 的 command matrix 与仓库当前测试现实同步
补齐脚本级最小自测，覆盖成功、失败、报告写入和退出码
让 release/report/readiness 能引用同一批高信号测试入口
建立 check report 最小 schema，保证后续工具可消费
```

### 明确不做

```text
不在本阶段引入新的业务能力或新执行域
不把所有慢测试都塞进 smoke，只追求“看起来全绿”
不为了让 release 更好看而隐藏真实失败、warning 或 open issue
不重写整套 CI 平台；本阶段先把本地与仓库内门禁入口修实
```

## 实施拆解

### 104.1 检查脚本解析与编码修复

目标：

```text
先让 check.ps1 能稳定解析、执行并写出报告
```

交付：

```text
PowerShell 字符串与 here-string 清理
UTF-8 / BOM / 控制台输出编码策略固定
脚本语法错误、异常 token、断裂字符串全部移除
```

要求：

```text
脚本在默认 PowerShell 环境可直接运行
错误优先以脚本自身失败暴露，而不是静默吞掉
```

### 104.2 Profile 职责边界收口

目标：

```text
让 full / smoke / fast / api / security / release 六个 profile 真正各司其职
```

交付：

```text
profile -> command mapping 清单
最小 smoke 套件清单
release profile 的慢测试和 issue gate 边界说明
command matrix 与 README/文档同步
```

要求：

```text
smoke 只保留最高信号、最快发现结构性回归的集合
release 负责真实重型门禁，不反向污染开发日常反馈速度
```

### 104.3 报告与退出码契约统一

目标：

```text
让脚本输出既能给人看，也能给后续工具和 release 读取
```

交付：

```text
check report schema
command result summary
slow test summary
failed command log path
overall status / exit code 契约
```

要求：

```text
只要任何必跑命令失败，脚本退出码必须非零
失败报告必须包含哪条命令失败、日志在哪、耗时多久
```

### 104.4 release / readiness 口径对齐

目标：

```text
减少“readiness 说 ready，但本地检查没法跑”这类错位
```

交付：

```text
release gate 高信号测试引用表
readiness 依赖的最小测试映射
phase90 ~ phase104 关键门禁集合
```

要求：

```text
readiness 的 ready/partial 必须能被本地命令复核
release summary 里引用的门禁名称要能映射到具体测试或脚本
```

### 104.5 检查脚本回归测试

目标：

```text
避免 check.ps1 修一次、下个阶段又悄悄坏掉
```

交付：

```text
脚本级回归测试
最小 fixture
失败场景验证
报告文件结构断言
```

要求：

```text
至少覆盖：成功运行、命令失败、报告生成、日志路径输出、退出码传播
```
