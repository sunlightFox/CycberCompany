# 第八十阶段 - 聊天内工具调用闭环

## 阶段定位

本阶段目标是补齐单轮聊天内的：

```text
model
  -> tool call
  -> safety / approval
  -> tool / mcp execution
  -> result reinjection
  -> final reply
```

这一步完成后，聊天不再只能依赖前置 route 捷径和 task handoff，而是具备稳定的单 turn 执行能力。

## 请求分类

### 1. direct answer

满足以下任一条件时，必须直接回复，不进入 tool loop：

```text
纯闲聊
纯解释、比较、总结、方案讨论
显式要求“只分析，不执行”
当前能力边界说明
pending action 文字确认/拒绝/修改，不需要新工具
```

### 2. single-turn tool loop

满足以下条件时，允许进入聊天内工具闭环：

```text
动作范围小
可在当前 turn 内完成
不需要后台长时间运行
结果可以直接回注并给出最终回复
```

典型例子：

```text
浏览器只读检索
知识库查询
受控终端只读命令
受控文件列表或只读读取
账号资产上的只读外部查询
```

### 3. task handoff

满足以下任一条件时，不走 single-turn tool loop，而是转 Task Engine：

```text
需要多步执行
需要跨多个工具
需要长时间等待
需要后台重试或恢复
需要产出 artifact
需要多人协作或子任务
```

典型例子：

```text
下载并整理文件
项目部署
安装软件
复杂表格文档生成
长链路浏览器操作
```

## Tool loop 与 Task Engine 的分界

### tool loop 负责

```text
单轮内完成的小动作
只读或低风险短动作
可同步回注结果的调用
```

### Task Engine 负责

```text
后台长任务
多步工作流
需要 checkpoint、恢复、通知、产物管理的执行
```

### 不允许的模糊地带

```text
聊天 runtime 不能偷偷执行一个实质长任务再假装是单轮完成
Task Engine 不能承接“明明只需只读查询”的微动作
```

## 工具分类

### A. 只读工具

定义：

```text
不改变外部世界状态
不写本地重要文件
不发消息、不下单、不登录、不安装、不删除
```

允许：

```text
优先进入 single-turn tool loop
如果 capability / safety 通过，可不经 approval
```

### B. 外部交互工具

定义：

```text
会访问外部服务、账号、页面或平台
但动作本身可能是只读，也可能是低风险写操作
```

要求：

```text
必须经过 capability 判断
必须有资产句柄来源
是否需要 approval 由 safety policy 决定
```

### C. 高风险工具

定义：

```text
可能改变真实世界状态
涉及下载、登录、删除、安装、支付、发布、发送、写盘、执行命令
```

要求：

```text
必须经过 safety
必须显式 approval 或 deny
默认不在 single-turn tool loop 中自动完成
必要时转 Task Engine
```

## approval 插点

### 执行前审批

以下动作在工具真正执行前必须停下：

```text
写文件
删除文件
下载安装
外部账号写操作
浏览器下载
发送消息
系统命令执行
任何 wallet / credential 相关动作
```

### 审批结果

审批结果固定为：

```text
approved_once
approved_session
denied
modified
expired
```

### 修改路径

用户如果在审批时改条件：

```text
不能直接复用旧 tool request
必须重新生成新的 tool request 或 task request
原 request 进入 superseded / cancelled 状态
```

## tool result 回注规则

### 回注到模型的内容

只允许回注：

```text
tool name
sanitized result summary
structured evidence summary
trusted_level
artifact refs
failure summary
```

禁止回注：

```text
raw secret
raw token
cookie
明文 credential
超长原始网页正文
未经清洗的终端原始大块输出
```

### 回注层次

工具结果进入 `ContextPacket` 或临时 continuation context，标记为：

```text
tool_result_context
```

并且必须附带：

```text
trace_ref
tool_call_id
source_type
trusted_level
```

## 失败、超时、取消、部分成功的用户可见语义

### 失败

用户可见回复必须明确：

```text
哪里失败了
哪些动作没有执行成功
是否还能继续
是否建议转任务或补确认
```

明确禁止：

```text
把失败包装成“已处理好”
只暴露内部错误码，不解释实际影响
```

### 超时

必须明确：

```text
当前没有拿到结果
未确认完成
可选择重试、转后台任务或缩小动作范围
```

### 取消

必须明确：

```text
本次操作已中止
不声称已完成后续动作
```

### 部分成功

必须明确：

```text
哪些步骤成功
哪些步骤未成功
最终结论是否可用
```

## trace / audit / redaction 要求

每次工具调用必须生成：

```text
tool request trace span
safety decision trace span
approval trace span（如有）
tool execution trace span
sanitized tool result summary
audit event
```

明确要求：

```text
聊天内工具调用不能绕过 ToolRuntime
ToolRuntime 不能绕过 Asset Broker / Safety
所有可见结果必须先过 redaction
```

## 推荐运行时序列

标准 single-turn tool loop 序列：

```text
context.ready
intent.detected
mode.selected
model.started
tool.requested
safety.checked
approval.required?      可选
approval.resolved?      可选
tool.started
tool.completed | tool.failed
model.continue.started  可选
response.delta
response.completed
turn.completed
```

## 验收标准

```text
聊天内可安全完成只读工具动作
高风险动作不会绕过 approval
tool loop 与 Task Engine 边界清晰
工具结果能够回注并生成最终自然回复
失败、超时、取消、部分成功都诚实可见
所有工具调用都有 trace / audit / redaction
```

## 最小测试集

```text
只读知识查询在单 turn 内完成
只读浏览器检索在单 turn 内完成
高风险下载请求被拦到 approval.required
用户 deny 后不执行真实动作
tool 失败后回复不伪装完成
tool timeout 后 turn 诚实收尾
部分成功时明确列出完成与未完成部分
task handoff 场景不会误走 single-turn tool loop
```
