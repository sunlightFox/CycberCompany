# 第一百一十四阶段 - 真实场景稳定性与主链路可观测性收口

## 阶段定位

phase109 与 phase110 已经把真实场景长稳运行、NoTurn 根因、渠道路由稳定性做成了 readiness 和 release summary。
但当前仓库给出的结论仍然是：真实稳定性处于 `partial`，长稳证据存在，主链路 blocker 仍未完全清零。

这一阶段的目标不是再新增聊天能力，而是把主链路变成「能持续观察、能量化定位、能闭环修复」的系统。

## 目标

```text
清理 phase109 暴露的真实场景 P0 gap 和 NoTurn 问题
把 turn 创建、排队、执行、审批、交付几个关键节点做成统一可观测指标
让聊天主链路从阶段性证据通过，升级为持续稳定性可量化
为后续长期运行和发布门禁提供主链路健康基线
```

## 重点

### 114.1 NoTurn 与路由根因收敛

```text
围绕 turn_not_created、turn_created_but_not_queued、worker_not_running_or_disabled 等 canonical reason 持续收敛
统一渠道入站、session 绑定、conversation bootstrap、queue handoff 的失败归因
把 evidence gap 和 runtime gap 分开治理，避免混在一起
```

### 114.2 主链路指标化

```text
建立 turn_created_rate、turn_queued_rate、turn_completed_rate、approval_resolution_rate、final_deliverable_rate
指标要能按 channel、task domain、member、runtime path 分层查看
让 release gate 和 readiness 读取同一份主链路统计口径
```

### 114.3 失败回放与定位收口

```text
主链路失败后，trace、task replay、channel diagnostics、readiness summary 要能互相对上
失败记录要能直接回答：卡在入口、路由、worker、审批还是交付
避免出现 summary 提示有问题，但回放里找不到对应证据
```
