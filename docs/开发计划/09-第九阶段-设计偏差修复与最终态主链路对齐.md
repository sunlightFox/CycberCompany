# 第九阶段：设计偏差修复与最终态主链路对齐

## 摘要

第九阶段不是新产品版本，也不是新增一批“未来功能”。它是基于当前代码审查结果，对已经实现但仍处于降级、占位或偏离最终设计的关键模块做集中修复。

本阶段完成后，再回到第八阶段的安全审计、评测与封版门禁中复跑全部检查。换句话说，第九阶段负责“把偏差修掉”，第八阶段负责“证明已经能封版”。

## 阶段定位

第九阶段专门修复当前代码与最终设计之间的偏差：

```text
Safety Gate 不能再是占位实现，必须成为聊天、任务、工具、MCP、Skill 的统一安全入口。
Tool Runtime、MCP、Skill 必须全部经过 Capability Graph、Asset Broker、Safety、Approval、Trace。
Asset Broker 必须补齐最终态 resolve_for_tool，工具只能拿到最小化、脱敏、可审计的资源解析结果。
Task Engine 必须从关键词规则推进到 workflow / agent / supervisor 混合调度。
Response Composer 必须从文本透传升级为结构化回复编排。
Heart 与 Persona Engine 必须从占位升级为真实后端服务。
Memory 与 Knowledge 必须接入真实向量检索，不再长期停留在 fts_fallback。
运行契约、README、runtime contracts 必须如实标注 implemented / degraded / placeholder。
公司壳默认文案必须收敛到 shell 模板，不让核心层写死 Boss / Company 语义。
```

第九阶段不是替代第八阶段的封版阶段。第八阶段仍然是最终封版门禁；第九阶段是为了让当前实现有资格进入第八阶段最终验收。

## 阶段原则

1. 不新增前端实现代码，仍然遵守当前“后端优先”约束。
2. 不推翻已有模块，只补齐最终态缺口。
3. 不把降级能力标成完成能力。
4. 修复顺序遵循：schema -> migration -> repository -> service -> API -> tests -> evals。
5. 每个修复都必须有 trace、audit 或 eval 证据。
6. 所有安全相关修复必须同时覆盖 allow、deny、approval 三类路径。
7. 本阶段不降低第八阶段封版门禁，只为通过封版门禁补齐真实能力。

| 方法 | 路径 | 说明 |
|---|---|---|

### 验收

```text

```

## 小阶段 9.2：Safety Gate 最终态实现

### 目标

把 `SafetyService.evaluate_action()` 从占位实现升级为统一安全决策服务。

### 修复内容

定义 `ActionRequest` 最终字段：

```text
actor/member
organization
task
tool/skill/mcp/action
input/output payload 摘要
asset handles
destination / outbound target
risk hints
untrusted content refs
```

实现统一风险分级：

| 风险级别 | 场景 | 默认策略 |
|---|---|---|
| R0 | 普通聊天、普通总结 | allow |
| R1 | 只读查询、读取授权知识 | allow |
| R2 | 写入新工件、生成报告 | allow 或轻确认 |
| R3 | 覆盖、批量移动、修改已有工件 | approval_required |
| R4 | 外部提交、登录后动作、表单发布 | approval_required |
| R5 | 删除、脚本执行、系统修改 | approval_required 或 deny |
| R6 | 支付、转账、敏感外发 | strong approval 或 deny |
| R7 | 钱包签名、系统级持久动作 | deny 或二次强确认 |

接入安全检查：

```text
prompt injection 标记
DLP 扫描
secret pattern scanner
sensitive path denylist
outbound destination policy
approval policy
wallet/payment deny policy
hardware control policy
```

输出统一 `SafetyDecision`：

```text
decision: allow / deny / approval_required
risk_level
reason
redactions
required_controls
policy_sources
trace_refs
```

### Trace / Audit

每次安全判断必须写：

```text
safety.evaluate span
risk_level
decision
reason
policy_sources
redaction_count
```

高风险 `deny` 或 `approval_required` 必须写 audit event。

### 验收

```text
高风险动作不能只靠 ToolRuntime 内部判断
聊天任务、工具执行、Skill、MCP 调用都能调用 Safety Gate
R5+ 默认 approval 或 deny
prompt injection 不能触发 secret 外发
Safety 决策必须写 trace/audit
```

## 小阶段 9.3：Tool Runtime 与 MCP 沙箱硬化

### 目标

让工具和 MCP 不再依赖简单黑名单，而是进入最终态执行边界。

### 修复内容

ToolRuntime 执行前统一调用：

```text
Capability Graph
Asset Broker
Safety Gate
Approval Flow
Trace Service
Audit Service
```

终端工具从黑名单模型升级为策略模型：

```text
默认工作目录限制在 artifact/task sandbox
禁止直接访问敏感路径
命令分级
高风险命令 approval
R6/R7 默认 deny
stdout/stderr 进入 DLP 和 redaction
```

MCP stdio server 启动前增加：

```text
command allowlist
env_refs 校验
权限预览
safety evaluation
trace/audit
启动失败清晰错误
```

MCP tool call 执行前增加：

```text
tool schema validation
capability decision
safety decision
approval gate
output DLP scan
tool output redaction
```

浏览器类工具区分风险：

| 动作 | 风险基线 |
|---|---|
| snapshot/read | R1 |
| navigation | R2 |
| download | R3 |
| upload | R4 |
| form fill | R4 |
| submit/post | R4/R5 |
| payment/transfer/sign | R6/R7 |

### 验收

```text

工具输出中的 secret 被脱敏
```

## 小阶段 9.4：Asset Broker `resolve_for_tool` 与 Secret 隔离

### 目标

补齐设计中“模型只能拿 handle，工具执行器才能解析资源”的最终闭环。

### 修复内容

实现应用层 `AssetBrokerService.resolve_for_tool()`。

`resolve_for_tool` 必须：

```text
验证 handle 未过期
验证 handle 所属 member/task/action
二次查询 Capability Graph
调用 Safety Gate
只返回该工具所需的最小资源
secret 只在必要时临时解析
解析结果不进入模型上下文
解析结果不明文进入 trace/audit
```

统一工具入参：

```text
工具接收 handle_id，不接收真实 secret/path/token
工具内部通过 broker 解析
模型上下文只出现 handle 摘要
trace/audit 只记录 handle_id、asset_id、action、decision，不记录 secret value
```

修复模型 API key 直读 SecretStore 的边界：

```text
模型密钥应作为 model asset 或受控 secret resolver 管理
Brain/ChatService 不能把密钥明文写入 trace、audit、memory、tool args
模型调用 span 只记录 provider、model、token、latency、route reason
```

### 验收

```text
Skill/MCP/Tool 都不能绕过 Asset Broker 拿真实资源
handle 过期、跨成员、跨任务、跨 action 使用必须失败
trace/audit 中不能出现 secret 明文
secret 解析必须有 span
```

## 小阶段 9.5：Task Engine 混合执行升级

### 目标

把任务引擎从关键词规则推进到 workflow / agent / supervisor 混合最终态。

### 修复内容

拆分 Task Planning：

```text
rule planner
model planner
workflow template planner
agent exploratory planner
supervisor planner
```

增加任务模式选择依据：

| 条件 | 模式 |
|---|---|
| 固定步骤、成功条件明确 | workflow |
| 需要搜索、探索、反复判断 | agent |
| 多视角复杂问题 | supervisor |
| 高风险任务 | plan first + approval before execute |
| 缺少必要上下文 | ask clarification 或 safe fail |

Agent Loop 最小闭环：

```text
observe
think/plan
act
evaluate
revise
stop
```

每次 loop 必须写：

```text
plan span
tool span
safety span
observation summary
stop reason
resource budget usage
```

增加 Reflection Layer：

```text
成功路径候选
失败原因
skill candidate
memory candidate
policy improvement candidate
```

### 限制

```text
agent loop 必须有最大步数
agent loop 必须有最大成本
agent loop 必须有最大时长
高风险动作不得由 loop 自行越权执行
reflection 候选不得自动变成启用技能
```

### 验收

```text
固定任务走 workflow，不乱进 agent
探索任务能进入 agent loop
高风险任务先计划，不直接执行
agent loop 有最大步数、最大成本、最大时长限制
任务结束后可生成 skill/memory/reflection 候选，但不自动启用高风险技能
```

## 第九阶段总验收标准

第九阶段完成时必须满足：
