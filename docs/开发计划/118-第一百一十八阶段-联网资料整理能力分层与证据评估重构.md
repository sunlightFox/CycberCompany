# 第一百一十七阶段 - 联网资料整理能力分层与证据评估重构

## 阶段定位

当前联网资料整理链路已经具备可用能力：可以识别浏览器搜索请求、执行 `browser.search`、提取结果摘要、补充证据来源，并在部分场景下输出时效提醒、来源冲突提醒和可信度说明。

问题不在于功能缺失，而在于这条链路仍然是“能跑但耦合偏紧”的实现：

- `chat_intent_router.py` 直接把搜索 plan 以零散 `metadata` 形式下发
- `chat_direct_routes_runtime.py` 直接绑定 `browser.search`
- `chat_readonly_execution.py` 同时承担工具调用、结果提取、证据判断、可信度分级和文案渲染
- 测试更多在断言最终回复文本，结构化中间结果还不够稳定

这一阶段要做的不是新增更多搜索话术，而是把“联网资料整理”从单文件补丁式实现，升级为可持续演进的能力分层。

## 目标

```text
把联网资料整理链路拆成 plan / capability / evidence assessment / renderer 四层
把 browser.search 从业务逻辑中的直接绑定改成可替换 capability 接口
把时效判断、来源排序、冲突检测、可信度分级沉淀为结构化 assessment
让测试同时覆盖最终回复和中间 assessment contract
为后续接入 MCP 搜索源、多结果比较和官方源优先真实排序预留稳定接口
```

## 重点

### 117.1 搜索计划对象化

```text
把当前 route metadata 中的 query、require_citation、requested_sections、presentation_style 收口成统一 BrowserResearchPlan
router 只负责产出结构化 plan，不再扩散更多 ad-hoc metadata key
direct route runtime 只消费 plan contract，不继续直接拼 metadata
```

### 117.2 SearchCapability 抽象落地

```text
新增 SearchCapability 接口，统一描述搜索请求与搜索结果
当前 browser.search 通过 adapter 接到 capability 层，不再让 runtime 直接依赖 tool_name
为未来接入 MCP 搜索 provider、双搜索源或官方搜索源聚合保留替换点
tool 仍然可以继续使用，但要从“主链路写死调用”升级成“capability 实现细节”
```

### 117.3 EvidenceAssessment 结构化

```text
把现有时效判断、来源强弱、来源冲突、可信度分级从文案拼接逻辑中拆出
新增 RawEvidence 和 EvidenceAssessment 两个核心对象
assessment 至少要包含 freshness、source_rank、conflict_level、confidence、notes
渲染器只根据 assessment 输出人话，不再自己做隐式判断
```

### 117.4 Renderer 与可见回复解耦

```text
把 popular_explainer、步骤清单、普通结构化总结三种表现形式收口到独立 renderer
reply text 可以继续保持当前体验，但生成过程要从 assessment 驱动
避免后续每加一种提示语都回到大 if/else 中继续堆逻辑
```

### 117.5 回归测试升级

```text
现有 phase57 / phase59 浏览器搜索测试继续保留
新增针对 BrowserResearchPlan、SearchCapabilityAdapter、EvidenceAssessment、Renderer 的单测
E2E 继续断言最终 reply，同时增加对 structured payload 中 assessment 字段的断言
飞书联网资料整理场景集继续作为真实回归门禁，覆盖时效、冲突、来源优先级、可信度四类场景
```

## 实施顺序

```text
第一步：引入 BrowserResearchPlan，替换当前零散 metadata 读取
第二步：引入 SearchCapability 和 browser.search adapter，但保持外部行为不变
第三步：抽出 RawEvidence / EvidenceAssessment，先让现有文案逻辑消费结构化 assessment
第四步：拆分 renderer，把 popular_explainer 和普通结构化输出迁移到独立模块
第五步：补齐 assessment contract tests，并刷新飞书联网资料整理场景报告
第六步：视结果收口旧 helper，删除 chat_readonly_execution.py 中已迁出的混杂逻辑
```

## 风险与约束

```text
这一阶段不追求一次性接入多 provider 搜索源，先把分层和 contract 稳定下来
不在本阶段引入真正多结果重排或官方源抓取增强，避免 scope 扩散
不能破坏现有 browser_search_with_citation 路由语义、证据引用输出和 not_created task status
不能为了重构牺牲当前已经通过的飞书联网资料整理回归集
```

## 验收

```text
联网资料整理主链路从 router 到 renderer 形成 plan / capability / assessment / renderer 四层清晰边界
chat_direct_routes_runtime 不再直接写死 browser.search 作为唯一业务依赖，而是依赖 capability 接口
structured payload 中可以稳定看到 assessment 结果，至少包含 freshness、source_rank、conflict_level、confidence
phase57 / phase59 回归通过
飞书 10 轮联网资料整理场景、时效与冲突场景、来源优先级与可信度场景全部通过
后续若接入 MCP 搜索 provider，只需新增 adapter，不需要重写主链路判断和回复渲染
```
