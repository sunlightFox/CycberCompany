# 第三十五阶段 - 聊天主链路安全一致性与状态语义硬化

## 阶段背景

第三十一、第三十三、第三十四阶段已经围绕真实聊天主链路、压力测试和自然语言交互闭环做了大量补齐。当前 `apps/local-api/app/services/chat.py` 已经能串起 turn 创建、事件流、Context Gateway、Brain、Memory、Task、Model、Response Composer、Trace 和取消/重试等主流程，但代码审查暴露出一批更底层的问题：安全过滤时机、上下文脱敏、会话归属、任务状态语义、隐私路由和测试补丁混入生产路径。

这些问题不适合继续分散在后续体验阶段里修补。第三十五阶段专门收敛聊天主链路的安全一致性和状态真实性，把 `chat.py` 从“能跑通”推进到“不会泄漏、不会串会话、不会把未完成说成完成、不会用硬编码测试答案支撑生产体验”。

本阶段仍然只做后端能力、schema、service、repository、API、tests、evals 和文档：

```text
不新增前端页面、组件、样式、Tauri 窗口或桌面端交互代码
不改变聊天页展示规则
不绕过 Asset Broker、Capability Graph、Safety、Approval 和 Trace
不通过写死固定回复来让测试变绿
不清空 data、不删除历史会话、不删除用户文件
```

已确认的主要问题：

| 问题编号 | 严重级别 | 现象 | 风险 |
|---|---|---|---|
| `CHAT35-FIX-001` | P0 | 模型 delta 先流式发出，最终才做 `_phase31_output_guard` 和 `visible_text_guard` | 不安全内容可能已通过 SSE 泄出，流式文本和最终落库文本不一致 |
| `CHAT35-FIX-002` | P0 | 当前输入脱敏，但历史 recent messages 原文进入模型 messages | 上一轮 secret、token、本机路径可能在下一轮被送入模型 |

| `CHAT35-FIX-004` | P1 | 非 `waiting_approval` 任务状态统一发 `task.completed` 并说“处理完成” | `paused`、`failed`、`running`、`cancelled` 可能被伪装成完成 |

| `CHAT35-FIX-006` | P1 | `_phase31_output_guard` 包含大量测试样例式硬编码回复 | 真实用户输入可能被误伤，生产行为依赖关键词补丁 |
| `CHAT35-FIX-007` | P2 | `ChatService._execute_turn` 单函数承担过多职责 | 后续修复安全、任务、模型或记忆时容易互相回归 |

## 核心目标

第三十五阶段完成后，聊天主链路应满足：

```text
任何发给用户的 stream delta 已经过可见性、脱敏和安全边界过滤
最终 assistant message 与 stream delta 语义一致，可回放、可审计
模型上下文中的当前输入、历史消息、摘要、记忆和资源句柄均不含明文 secret
conversation_id 必须属于当前 member 或同 organization 的合法 participant
任务回复严格反映真实状态：等待确认、执行中、暂停、失败、取消、完成不能混用
高隐私输入优先走本地模型；无本地模型时给可恢复回复，不空白、不外发
任务创建、planner、model route 都继承 privacy_level 和敏感命中摘要
生产聊天路径不再依赖 `_phase31_*` 测试补丁式固定答案
`ChatService` 的主流程可以分段测试，关键分支有最小回归覆盖
```

## 阶段原则

1. 安全过滤必须发生在输出前，而不是最终落库前。
2. 脱敏是模型上下文边界，不只是 trace/report 边界。
3. 会话归属校验优先于上下文构建和消息落库。
4. 任务状态只能从 Task Engine 的真实状态和证据派生，不用模板猜测。
5. 高隐私不等于不能回答；它表示不能外发，优先本地模型或受控失败回复。
6. 任何测试补丁都不能留在生产主链路；需要确定性策略时放入 Safety、Response Composer 或 eval fixture。
7. 重构以风险收敛为目标，不做无关大改。
8. 每个安全、trace、状态语义修复都必须有最小测试。

## 小阶段 35.1：流式输出安全与最终文本一致性

### 目标

修复模型流式输出先泄出、最终才过滤的问题。用户通过 SSE 看到的每个 `response.delta` 都必须已经经过安全可见性处理，最终落库文本必须能由可见 delta 拼接得到或由明确的安全替换事件解释。

### 实现要求

```text
在 _call_model 中对 model_event.text 进入 RESPONSE_DELTA 前做统一 visible/redaction/safety filter
delta filter 负责 reasoning tag、内部 trace/id、secret、本地路径和 forbidden main reply terms
最终 assistant_text 不再二次改写成与已发 delta 不一致的内容
如果安全策略必须替换整段回复，应在首个用户可见 delta 前完成判断
记录 response_filter span：输入字符数、输出字符数、redaction_summary、blocked_reason
禁止把未过滤 raw delta 进入 TurnEventStore、chat_events 或 stream
```

### 影响范围

```text
apps/local-api/app/services/chat.py
apps/local-api/app/services/natural_chat.py
services/response-composer/response_composer/contracts.py
services/trace/trace_service/redaction.py
```

### 验收

```text
模型输出包含 trace_id、approval_id、tool_call_id 时，SSE delta 中不可见内部 ID
模型输出包含 token/password/private key/local path 时，SSE delta 与最终 message 均脱敏
最终 content_text 等于可见 delta 拼接结果，或 response_plan 明确记录安全替换
response.completed、turn.completed、trace span 均保留 redaction_summary
```

## 小阶段 35.2：模型上下文脱敏与历史消息隔离

### 目标

修复当前输入已脱敏但历史消息原文进入模型的问题。模型上下文是安全边界：当前输入、最近消息、摘要、记忆块和资源摘要都必须以脱敏后的内容进入模型。

### 实现要求

```text
Context Gateway 读取 recent_messages 后生成 model_safe_content_text
_model_messages 只使用已脱敏内容，不直接读取 content_text 原文
conversation summary 写入和读取均经过 redact，但不能丢失可理解语义
memory block 注入前再次做 redact，source_ref 只保留非敏感引用
附件 metadata、client_context、resource handle 进入模型前做 allowlist
新增上下文脱敏诊断：selected_count、redacted_count、sensitivity_hits_summary
```

### 影响范围

```text
apps/local-api/app/services/context_gateway.py
apps/local-api/app/services/chat.py
apps/local-api/app/services/memory.py
services/trace/trace_service/redaction.py
```

### 验收

```text
上一轮用户贴 api_key/token，下一轮普通聊天不会把明文 secret 送入 model messages
recent_summary 中本地路径、secret、token 均脱敏
trace input/output 和 chat_events payload 不含明文 secret
高隐私历史消息不影响本轮普通内容的可恢复回复
```

## 小阶段 35.3：会话归属与成员边界校验

### 目标

修复已有 `conversation_id` 只校验存在、不校验当前 `member_id` 是否有权使用的问题，避免跨成员串会话、串记忆和串权限。

### 实现要求

```text
create_turn 接收 conversation_id 时读取 conversation participant_json、primary_member_id、organization_id
校验 request.member_id 与 primary_member_id 或 participant member 匹配
校验 member.organization_id == conversation.organization_id
不合法时返回 NOT_FOUND 或 ASSET_PERMISSION_DENIED/CONFLICT，避免暴露其他会话存在性
新增 repository/helper 封装 conversation access check，避免 handler/service 到处散落判断
retry_turn 复用同一归属校验
stream/cancel/retry 后续如涉及用户身份，也预留 owner 校验入口
```

### 影响范围

```text
apps/local-api/app/services/chat.py
apps/local-api/app/db/repositories/chat_repo.py
apps/local-api/app/db/repositories/member_repo.py
apps/local-api/app/api/routes_chat.py
```

### 验收

```text
成员 A 不能向成员 B 的 single conversation 写入 turn
跨 organization conversation_id 被拒绝
合法 retry 不受影响
失败响应不泄漏敏感会话内容
```

## 小阶段 35.4：任务状态语义与事件映射

### 目标

修复任务路径中把非等待确认状态统一当成 completed 的问题。Chat 只负责把 Task Engine 的真实状态翻译成用户可理解的回复，不得伪造完成。

### 实现要求

```text
建立 task.status -> chat event / response_plan 映射函数
只有 TaskStatus.COMPLETED 才发 ChatEventType.TASK_COMPLETED
WAITING_APPROVAL 发 approval.required，并写 pending action
PAUSED 发 task.paused 或 response_plan task_status=paused
FAILED 发 task.failed 或 turn completed with task failure summary，不能说处理完成
CANCELLED 发 task.cancelled 或取消提示
RUNNING/PLANNED/CREATED 返回“已创建/已开始/正在执行/可查看任务回放”，不能说完成
回复文本必须包含真实状态、可用证据和下一步
```

### 影响范围

```text
apps/local-api/app/services/chat.py
apps/local-api/app/services/tasks.py
apps/local-api/app/services/natural_chat.py
services/response-composer/response_composer/contracts.py
packages/core-types/core_types/enums.py
```

### 验收

```text
Task Engine 返回 paused 时，聊天不出现 task.completed
Task Engine 返回 failed 时，回复说明失败原因和可恢复下一步
Task Engine 返回 running/planned 时，回复说明任务已创建但未完成
等待审批时 pending action 与 approval_id 绑定，但用户可见文本不暴露内部 ID
```

## 小阶段 35.5：隐私路由与任务链路继承

### 目标

让高隐私输入遵守“本地优先、禁止云端、无本地可恢复失败”的设计，而不是在聊天层简单硬失败；同时确保任务 planner 和模型辅助规划继承 privacy_level。

### 实现要求

```text
高隐私输入先查询可用本地 routable brain
存在本地模型时允许进入本地 route，不触发云模型
不存在本地模型时生成可见的 privacy block 回复，不返回空白
TaskCreateRequest.planner_context 写入 privacy_level、sensitivity_hits、allow_cloud=false
Task Engine / ModelPlanner 读取 planner_context，禁止高隐私任务使用云 planner
Audit 记录 model_route.blocked_by_privacy 但 payload 只含摘要
```

### 影响范围

```text
apps/local-api/app/services/chat.py
apps/local-api/app/services/model_routing.py
apps/local-api/app/services/tasks.py
apps/local-api/app/services/model_planner.py
services/brain/brain/contracts.py
```

### 验收

```text
高隐私输入 + 可用本地模型：走本地模型，trace 标记 local_only
高隐私输入 + 只有云模型：不调用云模型，回复可恢复且不空白
高隐私任务请求：planner_context 带 privacy_level=high，模型辅助规划不走云
报告、trace、audit 不含明文敏感值
```

## 小阶段 35.6：移除 `_phase31_*` 生产补丁

### 目标

把 `_phase31_output_guard` 及相关测试样例式固定回复从生产主链路剥离。安全、医疗、隐私、工具边界等确定性回复应进入明确的策略组件；eval 用例应在测试里断言行为，不应靠生产关键词表返回固定答案。

### 实现要求

```text
梳理 _phase31_output_guard 中每个分支归属
安全拒绝类迁入 Safety/Response Composer policy
隐私阻断类迁入 privacy response composer
工具/浏览器证据边界迁入 tool boundary composer
知识 padding、固定表格、固定数学题、测试指标等从生产路径移除
必要的 eval 期望放入 tests/evals fixtures，不参与真实用户聊天
保留最小兼容窗口时必须有 feature flag，默认关闭
```

### 影响范围

```text
apps/local-api/app/services/chat.py
apps/local-api/app/services/natural_chat.py
apps/local-api/app/services/design_alignment.py
services/response-composer/response_composer/contracts.py
apps/local-api/tests/test_phase31_real_chat_e2e_full_closure.py
```

### 验收

```text
真实聊天路径不再调用 _phase31_output_guard
安全类固定回复仍通过策略组件产生
知识解释不被硬编码 padding 污染
测试输入“123 和 105”等不会触发生产特殊答案，除非是正常模型/规则能力
旧 phase31/33/34 回归通过或更新为不依赖生产补丁的断言
```

## 小阶段 35.7：ChatService 责任切分

### 目标

降低 `ChatService._execute_turn` 的维护风险，让安全、上下文、记忆、任务、模型和收尾各自有明确边界与测试入口。

### 建议拆分

```text
ChatTurnAccessPolicy：member/conversation/organization 校验
ChatPrivacyCoordinator：隐私分类、路由前置决策、audit
ChatContextCoordinator：context build、模型安全 messages 构建
ChatMemoryCommandCoordinator：显式记忆命令和记忆事件
ChatTaskCoordinator：任务创建、pending action、状态语义映射
ChatModelCoordinator：模型 route、fallback、stream filter、usage
ChatTurnFinalizer：assistant message、events、summary、memory enqueue、trace close
```

### 实现要求

```text
不做一次性大爆炸重构
先围绕 35.1-35.5 的修复抽出最小协作对象
每个对象使用明确输入输出 schema 或 dataclass，不长期传裸 dict
保留 ChatService 对外 API 不变
所有新公共函数有最小测试
```

### 验收

```text
_execute_turn 行数明显下降，关键分支可单测
stream、privacy、task status、conversation access 均有独立测试
对外 API、事件类型和既有成功路径不破坏
```

```text
ChatService 局部拆分
真实 E2E 回归和报告脱敏扫描
```

## 验收标准

```text
SSE 事件、chat_events、assistant message、trace 中均不出现明文 secret/token/password/private key/local path
用户看到的流式文本和最终落库文本一致
跨成员或跨组织 conversation_id 被拒绝
任务未完成时不出现 task.completed 或“处理完成”文本
高隐私输入有本地模型时可继续，无本地模型时有可恢复回复
任务 planner 不在高隐私场景调用云模型
生产聊天路径不再依赖 _phase31_output_guard
新增测试和 phase30/31/33/34 相关回归通过
不新增任何前端 UI 或桌面端交互代码
```

## 与后续阶段关系

```text
第三十六阶段及以后可以继续做后台执行、浏览器会话、外部渠道和平台动作
第三十五阶段先保证聊天主链路安全、归属和状态语义可信
第四十一阶段的高质量体验修复应建立在本阶段的真实状态和安全一致性之上
第四十二阶段的外部平台动作必须复用本阶段的 pending action、privacy 和 stream 安全边界
第四十三阶段的视频媒体能力也必须复用本阶段的 artifact、任务状态和隐私输出规则
```

## 最终验收定义

第三十五阶段完成时，应能回答：

```text
用户流式看到的内容是否已经过滤，而不是事后补救
历史敏感消息是否不会被下一轮送入模型
一个 member 是否只能写入自己有权访问的 conversation
任务处于 paused/failed/running/waiting_approval 时，聊天是否真实表达状态
高隐私输入是否遵守本地优先和云端禁止
生产路径是否摆脱固定测试答案
ChatService 是否具备继续演进而不互相踩踏的分段边界
```
