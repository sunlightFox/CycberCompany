# 第二十三阶段：全量验证提速与封版证据闭环

## 摘要

第二十三阶段聚焦“封版前的验证可信度”。当前主链路相关测试拆分运行可以通过，`ruff check .` 已通过，`mypy .` 仍存在测试侧 `.state` 类型问题，全量 `pytest` 一次性运行会超时。对最终封版来说，能力完成度之外，还必须保证验证体系本身稳定、快速、可定位、可复跑。

本阶段目标是把测试、评测、类型检查、Release Gate、诊断报告和证据归档形成可靠闭环，避免“拆开能过、全量超时、封版证据不完整”的情况。

本阶段只做后端测试、评测、工程验证、诊断和少量测试基础设施，不新增 UI。

## 阶段定位

第二十三阶段回答：

```text
全量 pytest 是否能在可接受时间内完成
mypy 是否完全通过
长耗时测试是否能被标记、隔离和诊断
主链路 eval 是否能稳定进入 Release Gate
封版报告是否能说明每个 accepted risk
失败时是否能定位到模块、阶段、trace、case
```

## 当前基线判断

| 能力 | 当前状态 | 问题 |
|---|---|---|
| ruff | 已通过 | 暂无 blocker |
| mypy | 未完全通过 | 测试侧 `.state` attr-defined 问题 |
| 主链路阶段测试 | 拆分通过 | 32 个 12-17 阶段测试通过 |
| 根目录单元测试 | 拆分通过 | 12 个测试通过 |
| eval 测试 | 拆分通过 | 7 个 eval 通过 |
| 全量 pytest | 超时 | 需要定位耗时和运行策略 |
| Release evidence | 已有基础 | 需纳入后续 18-22 阶段证据 |

## 阶段原则

1. 不通过降低断言制造通过。
2. 不隐藏失败，不把 timeout 当作通过。
3. 长耗时测试必须可标记、可单跑、可诊断。
4. 类型检查问题必须修复，不靠全局 ignore 掩盖。
5. Release Gate 必须引用真实 eval evidence。
6. accepted risk 必须有原因、范围、owner_phase 和复查条件。

## 小阶段 23.2：pytest 全量超时定位

### 目标

### 诊断动作

```text
pytest --durations

按目录拆分运行

检查后台任务未关闭
检查 async task 泄漏
检查 TestClient 未关闭
检查长轮询/SSE/stream 测试 timeout
```

### 输出

```text
slow_test_report
hang_suspect_report
fixture_lifecycle_report
background_task_leak_report
```

### 验收

```text
全量 pytest 在目标时间内完成
若存在 slow 测试，必须标记并有单独 release 命令
没有未关闭后台任务阻塞进程退出
```
