# 第五十八阶段 - 语音与多媒体输入输出能力底座

## 阶段背景

网友对 agent 的直觉期待通常是“能看、能听、能说、能处理图片视频文件”。Hermes 在语音和多模态体验上更接近日用助理，OpenClaw 也更强调媒体和多能力组合。当前仓库已经有 media、browser、knowledge、artifact 等基础，但语音、转写、播报和多媒体处理还没有形成稳定底座。

本阶段继续只做后端、schema、migration、repository、service、API、tests、evals 和文档；不新增前端页面、组件、样式、Tauri 窗口或桌面端交互代码。

## 参考结论

### OpenClaw 采用点

OpenClaw 更强调媒体能力和可调用的外部 provider：

```text
图片、音频、视频、文档可作为任务输入和输出
媒体处理结果要回到统一 artifact 体系
```

本项目采用：

```text
媒体输入输出进入统一 artifact / trace / task 结构
不同 provider 只做适配，不改业务契约
```

### Hermes Agent 采用点

Hermes 更偏语音交互和多模态辅助执行：

```text
语音输入能直接进入 agent 会话
TTS 能把结果转成更自然的交互
视觉能力可辅助浏览器和文档理解
```

本项目采用：

```text
STT / TTS / image understanding / media summary 都有统一后端契约
语音和多媒体结果可以进入任务、记忆和通知链路
```

## 核心目标

本阶段完成后，后端应支持：

```text
语音转写、语音播报和基础音频 artifact
图片/视频/文档的统一接入与摘要
多媒体结果可回放、可脱敏、可审计
媒体能力与任务、记忆、浏览器、Skill 联动
```

## 阶段原则

1. 多媒体是能力底座，不是一次性脚本。
2. 输入文件先入库或入 artifact，再进入处理链路。
3. 结果必须可追踪到原始来源。
4. 涉及隐私音频、照片或视频内容时，默认更严格脱敏。
5. 不把云端生成当作唯一路径，本地优先。

## 阶段范围

### 本阶段必须完成

```text
STT/TTS provider contract
媒体 artifact schema 和 replay
图片、文档、视频摘要后端管线
多媒体输入输出 trace 和 audit
媒体能力与 task / memory / notification 的统一接线
```

### 本阶段不做

```text
不新增 UI 播放器或编辑器
不默认把敏感音视频上传到外部云
不把媒体处理结果伪装成真实完成的业务动作
不绕过用户授权访问私密文件
```

## 主要待补

```text
语音收发
图片视频理解
媒体 artifact 统一
隐私脱敏
多模态任务联动
```
