# 第四十五阶段 - 聊天主链路生产补丁清理与服务拆分

## 阶段背景

第三十五阶段已经修复聊天主链路的关键安全边界，第四十一阶段用确定性质量修补关闭了高质量体验测试中的 10 个已知问题。但当前 `ChatService` 仍承担过多职责，并且历史 `_phase31_*`、质量用例式回复、定时任务文本解析、隐私恢复、任务状态表达等逻辑仍集中在 `apps/local-api/app/services/chat.py`。

这些逻辑让主链路可跑，但不利于继续演进。第四十五阶段要把“测试补丁式行为”沉淀为正式的策略组件，把 `ChatService` 拆成清晰协作对象，确保聊天主链路从“会修补”升级为“可维护、可扩展、可审计”。

## 核心目标

```text
生产路径不再依赖 _phase31_* 固定回复函数
ChatService 只保留 turn 编排，不继续堆积业务策略
隐私、上下文、模型、任务、记忆、质量回复分别有明确 coordinator
质量修补进入 Response Composer / Safety / Memory policy / eval fixture
主链路事件、trace、response_plan 与当前行为保持兼容
```

## 阶段原则

1. 不做大爆炸重构，按风险边界拆。
2. 先建立测试，再迁移逻辑。
3. 每迁出一个分支，都保留等价回归。
4. 生产服务不按测试 case id 写回复。
5. eval fixture 可以包含特定用例，生产策略必须可泛化。
6. 保持 Chat API 和 SSE 事件契约不破坏。
7. 不新增 UI。

## 本阶段必须完成

```text
建立 ChatTurnOrchestrator / ChatModelCoordinator / ChatTaskCoordinator / ChatMemoryCoordinator
迁出 ChatVisibleOutputFilter、privacy routing、task status mapping 的调用边界
移除或隔离 _phase31_output_guard 及相关固定知识 padding
将 phase41 ChatQualityExperienceService 改为通用 QualityPolicy + ResponseComposer 模板
scheduled task 自然语言创建从 chat.py 迁到 ScheduledTaskIntentCoordinator
隐私高恢复回复由 Safety/ResponseComposer 统一生成
为 coordinator 增加最小单测
```

## 本阶段不做

```text
不改变用户可见 API
不修改数据库 schema，除非拆分状态需要最小字段
不新增前端页面
不为了重构重写 Task Engine、Memory 或 Model Router
不删除真实历史会话或 data
```

## 小阶段总览

| 小阶段 | 名称 | 核心交付 |
|---:|---|---|
| 45.1 | 主链路调用图与测试冻结 | 标出 chat.py 内所有 phase 补丁和策略分支 |
| 45.2 | Coordinator 骨架 | 定义输入输出 dataclass/schema |
| 45.3 | 安全和隐私迁移 | 隐私分类、本地优先、恢复回复进入独立 coordinator |
| 45.4 | 任务和定时任务迁移 | task status、scheduled task intent 从 chat.py 拆出 |
| 45.5 | 记忆和质量回复迁移 | 记忆确认、遗忘边界、Persona 拒绝模板策略化 |
| 45.6 | phase31 补丁清理 | 删除生产调用，剩余逻辑进入 eval fixture 或 policy |
| 45.7 | 回归与性能 | SSE 一致性、trace、质量用例和真实模型 smoke |

## 验收标准

```text
chat.py 行数明显下降，主函数只做编排
生产路径没有 _phase31_output_guard 调用
phase35/41 已知安全和质量用例继续通过
隐私高输入仍不外发云模型，且有可恢复回复
任务 paused/failed/running 不被说成 completed
最新指令优先不是通过单一测试句硬编码实现
所有 coordinator 公共函数有最小测试
```

## 与后续阶段关系

第四十六阶段会补后台 worker，第四十七阶段会推进真实浏览器和外部平台执行器。如果聊天主链路仍堆积补丁，后续所有自然语言入口都会变得脆弱，因此第四十五阶段应优先完成。
