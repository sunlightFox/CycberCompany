# 第三十六阶段 - 长期定时任务与后台执行策略

## 阶段背景

前面阶段已经具备聊天触发任务、Task Engine、task_jobs、memory_jobs、trace、审批和恢复能力，但这些能力仍主要服务“用户当前发起的一次任务”。对标 OpenClaw 和 Hermes Agent 的真实使用反馈，用户会自然提出长期型请求：

```text
每天早上帮我看一次指定网页并总结。
每周整理一次下载目录，只生成报告，不自动删除。
持续监控某个账号或知识源，有变化再提醒我。
后台帮我定期跑一个检查，失败时告诉我原因。
```

这类需求不是普通 `task_jobs` 的简单重复。它必须有长期契约、执行窗口、审批策略、失败退避、离线恢复、通知策略、预算限制和审计证据。否则系统容易出现两类问题：

```text
任务只能当场执行，无法长期守约
后台任务在用户不在场时越权执行高风险动作
```

第三十六阶段要把长期任务和定时任务纳入后端一等契约，形成 `ScheduledTask` / `RecurringTask` / `BackgroundPolicy` 的稳定底座。当前阶段仍只做后端、schema、migration、repository、service、API、tests、evals 和文档，不新增任何前端页面、组件、样式、Tauri 窗口或桌面端交互代码。

## 核心目标

本阶段完成后，后端应支持：

```text
用户通过聊天或 API 创建长期/定时任务
系统把定时规则、执行窗口、预算、权限和审批策略结构化保存
后台 scheduler 能按 due time 创建普通 Task Engine 任务
每次运行都有独立 task、trace、tool_call、approval、artifact 和 replay
无人值守时高风险动作默认 deny 或等待明确策略，不自动执行
任务失败有 retry/backoff/dead_letter 机制
用户可暂停、恢复、取消、归档长期任务
release/eval 能验证长期任务不会绕过 Safety、Approval、Asset Broker 和 Capability Graph
```

## 阶段原则

1. 长期任务只是任务触发器，不直接执行工具。
2. 每次运行必须落到普通 Task Engine，不绕过 task 状态机。
3. 后台执行默认保守；没有用户在线确认时，高风险动作默认 fail-closed。
4. schedule 只描述“何时尝试执行”，不等于永久授权。
5. 永久允许只开放给低风险、只读、可解释、可撤销的动作模板。
6. 资产句柄必须按每次运行重新发放或校验，不复用过期 handle。
7. 每次模型调用、工具调用、审批和记忆写入都必须有 trace。
8. 失败必须可解释：错过执行窗口、后端离线、资产过期、权限不足、审批缺失都要有原因。
9. 不把 cron 字符串直接当 shell 或 prompt；所有定时规则先解析成结构化对象。
10. 不新增 UI；只提供后端契约和可供未来 UI/CLI 调用的 API。

## 对标结论

### OpenClaw 采用点

```text
用户希望 agent 能做长期后台任务，而不是每次都靠手动触发
长期任务需要清晰的 job state、下一次运行时间和失败记录
浏览器、文件、网页、知识库等工具可以作为定时任务步骤
```

对我们的启发：

```text
把 schedule 和 task execution 分离
把每次运行当作可回放 task
在任务计划阶段就写入预算和工具风险边界
```

### Hermes Agent 采用点

```text
cron / unattended execution 默认不能越权
危险动作在用户不在场时不能自动通过
会话级 approval 不能无限期带到后台任务
```

对我们的启发：

```text
新增 BackgroundExecutionPolicy
区分 interactive、semi_attended、unattended 三种运行场景
无人值守高风险动作默认 deny 或等待下一次用户确认
```

## 关键实现要求

### schedule parser

```text
支持 once、interval、daily、weekly、monthly-lite
所有 schedule 必须带 timezone，默认从 client_context 或系统配置取
拒绝模糊规则，例如“有空时”“经常”
计算 next_run_at 必须确定、可测试、无随机漂移
jitter 必须有上限，且只用于错峰，不改变语义
```

### due scanner

```text
只扫描 active 且 next_run_at <= now 的 scheduled_tasks
使用数据库锁或幂等 key 防止重复触发
每次 due 创建 scheduled_task_run，再创建普通 Task Engine task
如果系统离线错过窗口，根据 misfire_policy 处理 skip/run_once/catch_up_limited
```

### 安全策略

```text
unattended 场景下 R3+ 默认不执行
需要审批时 run 进入 waiting_approval 或 paused，不自动重试危险步骤
session approval 不允许跨 scheduled run 复用
always approval 只能用于低风险只读模板，且必须有 policy source
资产 handle 每次 run 重新 query/validate
```

### 聊天入口

```text
用户说“每天帮我...”时，Brain 识别 scheduled_task_request
如果时间、对象、风险策略不明确，追问最少必要问题
创建后回复自然语言摘要，不要求用户理解 cron
用户说“暂停每天那个任务”时，匹配唯一 scheduled task；多个候选则追问
```

## 必测用例

```text
创建 daily scheduled task，next_run_at 正确
创建 interval scheduled task，due scanner 只触发一次
manual trigger 创建 scheduled_task_run 和普通 task
pause 后 due scanner 不触发
resume 后重新计算 next_run_at
unattended 高风险工具进入 waiting_approval 或 deny，不自动执行
资产 handle 过期后下一次 run 重新发放
连续失败后进入 backoff，超过阈值 dead_letter
错过执行窗口按 misfire_policy 处理
run history 可跳转 task replay
聊天“每天...”可创建计划，聊天“暂停...”可暂停计划
```
