# 第一百一十六阶段 - 成熟度看板与发布门禁统一化

## 阶段定位

当前仓库已经有 readiness、release gate、task closure gate、extension runtime snapshot、deliverable proof 等多套成熟度信号。
问题不在于没有指标，而在于这些指标仍然分散在不同 phase、不同 summary 和不同脚本中，缺少统一决策面。

这一阶段要做的是把分散的阶段门禁收口成统一成熟度看板，让“能不能发布、哪里不成熟、下一步先修什么”一眼可见。

## 目标

```text
统一 phase109 / phase110 / phase111 / phase112 相关成熟度信号
形成面向发布的单一成熟度看板和 blocker 视图
把 readiness、release report、check report 三套输出收敛到同一口径
为后续长期运行治理和扩展生态治理提供统一优先级排序
```

## 重点

### 116.1 成熟度视图统一

```text
统一 stability、routing、delivery evidence、extension sync、quality gate 五类一级维度
每个维度要能看到 status、blockers、evidence refs、next owner
避免 readiness ready 但 release partial，或脚本通过但成熟度仍不透明
```

### 116.2 blocker 优先级统一

```text
把 blocker 分成 runtime_fix、evidence_gap、governance_gap、ecosystem_gap
按 P0 / P1 / P2 输出修复队列
保证 release 决策优先看真实 blocker，而不是只看阶段编号是否完成
```

### 116.3 发布门禁口径统一

```text
check.ps1、chat-mainline-readiness、release report 使用同一份 contract version 和成熟度汇总字段
同一个问题在不同报告里要有相同原因码和相同 owner 指向
让封版和日常开发看到的是同一套事实，而不是三套解释
```

## 验收

```text
形成统一成熟度看板，至少覆盖 stability、routing、delivery、extension、quality 五个一级维度
同一 blocker 在 readiness、release report、check report 中使用一致原因码和 owner
release 决策可以直接依据成熟度看板，而不需要人工拼接多个阶段报告
P0 / P1 / P2 修复队列可自动从门禁输出中提取
后续新增阶段文档可以挂接到同一成熟度框架，而不是再造一套 summary
```
