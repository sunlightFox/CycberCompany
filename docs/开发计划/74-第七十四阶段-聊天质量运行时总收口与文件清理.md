# 第七十四阶段 - 聊天质量运行时总收口与文件清理

## 阶段背景

第六十九到第七十三阶段会把聊天主链路、上下文、工具、浏览器、Skill、MCP、渠道运行时拆开。拆分完成后，如果不做一次总收口，仓库会出现：

```text
新旧文件并存
薄协调器残留
旧 route reason_code 残留
旧字段与新字段混用
chat.py / tools.py / memory.py 仍留大段历史逻辑
```

这会让后续维护再次退回“补丁式聊天系统”。

本阶段目标是做一次严格的运行时总收口：删除不再需要的桥接层，收敛新契约，明确哪些文件保留、哪些文件兼容、哪些文件可以删除。

## 核心目标

完成后，仓库应满足：

```text
聊天主链路文件边界稳定
上下文、工具、浏览器、Skill、MCP、渠道 runtime 各自独立
薄协调器不再成为长期负担
旧硬编码 route / reply / payload 残留最小化
新增文件树与后端服务边界对齐
```

## 破坏性调整

### 可删除或并回的薄协调器

在前置阶段完成后，评估删除或并回：

```text
apps/local-api/app/services/chat_context.py
apps/local-api/app/services/chat_model.py
apps/local-api/app/services/chat_memory.py
apps/local-api/app/services/chat_response.py
```

原则：

```text
只保留真正有独立边界价值的 service
纯转发、纯薄包装、纯单函数壳优先删除或并回
```

### 统一后端目录归属

聊天质量相关后端最终建议归属：

```text
apps/local-api/app/services/chat_*.py
apps/local-api/app/services/context_*.py
apps/local-api/app/services/tool_*.py
apps/local-api/app/services/browser_*.py
apps/local-api/app/services/skill_*.py
apps/local-api/app/services/mcp_*.py
apps/local-api/app/services/channel_*.py
apps/local-api/app/services/channels/providers/*.py
```

不再继续向 `chat.py`、`tools.py`、`channel_connectors.py` 这类总包型文件回填大块新逻辑。

## 实现要求

### 1. 文件清理清单

必须给出明确清单：

```text
保留文件
缩减文件
待删除文件
兼容层文件
新入口文件
```

并在文档中标明：

```text
谁是正式入口
谁只是兼容壳
谁不允许再新增逻辑
```

### 2. 新旧契约收口

要求：

```text
route taxonomy 统一
tool result 结构统一
session context metadata 统一
browser state 结构统一
skill runtime 返回结构统一
mcp bridge 返回结构统一
```

### 3. 静态残留扫描

新增或扩展检查项：

```powershell
rg "_handle_browser_read_page|_handle_terminal_readonly_command|_execute_browser_tool|_execute_terminal_tool" apps/local-api/app/services
rg "class ChatContextCoordinator|class ChatModelCoordinator|class ChatMemoryCoordinator|class ChatResponseCoordinator" apps/local-api/app/services
rg "route_semantics" apps/local-api/app/services/chat.py
```

目标：

```text
旧巨型文件中的历史实现只剩兼容入口，不再继续增长
```

### 4. 回归门禁

在这一阶段，聊天质量回归必须覆盖：

```text
闲聊
深度聊天
多轮聊天
记忆
工具调用
Skill
MCP
浏览器任务
系统命令
渠道聊天
```

## 验收标准

```text
聊天质量相关核心模块完成拆分并收口
历史巨型文件不再承担新增主逻辑
薄协调器完成删除或明确兼容定位
仓库文件边界与 docs/11 的后端服务职责更接近
后续新功能可以按模块扩展而不是继续堆入 chat.py
```
