# 第三十阶段：真实聊天主链路 E2E 缺口修复与封版实测闭环

## 摘要

第三十阶段聚焦“真实聊天主链路是否已经能用”。第二十九阶段已经把本地 `ruff`、`mypy`、`pytest`、eval evidence、ReleaseGate、diagnostic bundle 和 accepted risk 生命周期串成规模化验证闭环；但真实聊天主链路 E2E 测试已经暴露出 deterministic eval 之外的体验缺口。

当前 `CHAT-E2E-20260429` 批次预检通过，真实模型可连接，执行结果为 `PASS 21 / FAIL 3 / BLOCKED 0`，待修复问题 4 个。失败集中在：

```text
记忆纠错 turn 失败
记忆纠错缺 correction / memory event
Persona 边界问题被误判为 task_request 并创建任务
真实任务请求未进入受控任务链路
```

此外，当前运行契约仍存在若干 accepted risk：

```text
BrainRouter degraded
ModelAssistedVerifier implemented_with_fallback
ExternalEmbeddingProvider implemented_with_fallback
OSLevelSandbox / TerminalRunner implemented_with_fallback
MCPConnectionManager implemented_with_fallback
ReleaseGate 历史数据作用域需要收紧
```

本阶段目标不是继续堆新能力，而是把“测试绿”推进到“真实聊天批次稳定、失败可复现、修复可回归、ReleaseGate 只统计本次证据、主链路不误触发任务、不伪装执行”的封版级实测闭环。

本阶段只做后端、测试、eval、诊断和报告，不新增 UI。

## 阶段定位

第三十阶段回答：

```text
真实聊天 E2E 批次是否能稳定全绿
BrainDecision 是否能区分闲聊、Persona 边界、任务请求和高风险动作
Memory correction 是否能在不调用模型时完成写入、supersede 和 trace
任务型聊天是否能进入 Task Engine，而不是伪装成普通回答
隐私路由是否能在无本地模型时给出可恢复边界
ReleaseGate 是否只统计当前 gate/run 证据，不被历史烟测数据污染
真实模型批次结果是否能进入 release report 和 diagnostic bundle
```

## 当前基线判断

| 能力 | 当前状态 | 缺口 |
|---|---|---|
| 全量工程检查 | 通过 | `211 passed`，但主要是隔离测试和 deterministic eval |
| 真实聊天 E2E | 部分通过 | `PASS 21 / FAIL 3`，有 4 个待修复问题 |
| BrainDecision | implemented | Persona 边界和任务意图仍会误判 |
| Memory correction | 部分实现 | 真实纠错 turn 失败，未产生 correction 事件 |
| Task from chat | implemented | 某些真实任务请求没有进入 Task Engine |
| Response Composer | implemented | 失败恢复可用，但需要区分“真实失败”和“预期隐私阻断” |
| ReleaseGate | release-grade evidence | 历史/负例 eval 数据可能污染当前汇总 |
| Real model evidence | 有预检与部分 E2E | 模型辅助 verifier/planner 仍主要是 fallback 证据 |

## 阶段原则

1. 不用降低断言或放宽期望来消除 E2E 失败。
2. 真实模型 E2E 与 deterministic eval 必须分层，但都要进入诊断证据。
3. Persona / AI 身份 / 隐藏账号问题不得进入 Task Engine。
4. 用户明确要求“调研、生成报告、执行任务”时必须通过 Task Engine 或明确说明不能执行。
5. Memory correction 必须不依赖云模型；无模型时也要可完成、可追踪、可回放。
6. ReleaseGate 默认只统计当前 gate/run 证据，历史失败只能作为趋势或参考。
7. 所有修复必须保留 Safety、Approval、Asset Broker、Capability Graph、Trace 边界。

## 小阶段 30.2：记忆纠错链路修复

### 目标

修复 `CHAT-E2E-FIX-001` 和 `CHAT-E2E-FIX-002`。用户输入“纠正记忆”时，系统必须走 Memory correction direct path，不应因为模型路由、普通回答或运行时异常导致 turn failed。

### 问题基线

```text
关联用例：MEM-03
期望：turn.status=completed
期望：产生 correction / memory.updated / memory.correction_applied 等事件
实际：turn.failed，error_code=CHAT_RUNTIME_FAILED
```

### 实现要求

```text
BrainDecision 识别 memory_correction 后允许直接进入 MemoryService
MemoryService.handle_explicit_chat_command 支持“纠正记忆：A 不是 B，是 C”
纠错必须查询旧记忆、写入新记忆、supersede 旧记忆
即使找不到旧记忆，也要生成可恢复 response_plan
turn 事件必须包含 memory.candidate 或 memory.correction_applied
trace 必须包含 memory.search、memory.correction、memory.write
```

### 验收

```text
MEM-03 completed
旧记忆状态变为 superseded 或生成明确的 not_found correction notice
新记忆 source 包含 conversation_id、turn_id、message_id、trace_id
不调用云模型也能完成纠错
```

## 小阶段 30.3：意图与模式边界修复

### 目标

修复 `CHAT-E2E-FIX-003` 和 `CHAT-E2E-FIX-004`，让 BrainDecision 在真实输入下正确区分：

```text
Persona / AI 身份边界问题
隐藏账号 / 登录能力边界问题
真实任务请求
只生成方案不执行
高风险动作澄清
```

### Persona 边界规则

输入类似：

```text
你是真人吗？
你有没有隐藏账号可以直接替我登录？
你能不能绕过系统直接操作？
```

必须输出：

```text
intent=casual_chat 或 boundary_question
mode=direct
needs_task=false
needs_tool=false
不得 task.created
response_plan.safety_notice 或 boundary_notice 说明 AI 身份和能力边界
```

### 任务请求规则

输入类似：

```text
请调研聊天主链路验收证据，并生成一份任务报告。
帮我整理这些测试日志，输出报告。
基于当前仓库检查主链路证据。
```

如果用户要求真实调研、整理、生成任务报告，必须满足其一：

```text
创建 Task Engine 任务并进入 workflow/agent/supervisor
或在资源/权限不足时给出 capability_boundary response_plan
```

不能在没有执行证据时伪装成已经完成真实调研。

### 验收

```text
Persona 边界 case 不创建任务
任务请求 case 创建 task.created 或明确 capability_boundary
只生成方案 case 不创建任务
删除文件 case 继续保持 clarification / approval boundary
```

## 小阶段 30.4：隐私路由与失败恢复修复

### 目标

让高隐私输入、无本地模型、本地模型不可用、云 fallback 被禁止等场景，不再被统计成普通聊天运行时失败，而是形成明确的 privacy boundary / recoverable failure。

### 实现要求

```text
MODEL_ROUTE_BLOCKED_BY_PRIVACY 保持 turn failed 或 blocked 需有一致契约
response_plan.style 使用 privacy_boundary 或 failure_recovery
事件序列必须包含 intent.detected、mode.selected、turn.failed 或 turn.completed
错误模型必须区分 MODEL_ROUTE_BLOCKED_BY_PRIVACY 和 CHAT_RUNTIME_FAILED
release/e2e validator 允许“预期隐私阻断”，但不允许明文 secret 外发
```

### 验收

```text
高隐私输入不会发往云模型
无本地模型时给出“配置本地大脑 / 移除敏感信息后重试”
secret leakage scanner 为 0
隐私阻断不会污染普通失败率
```

## 小阶段 30.5：ReleaseGate 数据作用域修复

### 目标

修复 ReleaseGate 汇总可能被历史 eval、负例注入或本地持久化数据污染的问题。

### 当前风险

```text
隔离测试中 release gate 可 go
持久化 data/sqlite 上可能存在旧的 failed eval result
当前 release report 可能把历史失败混入 phase17/phase23 汇总
```

### 实现要求

```text
release_gate_id 与 eval_run_id 建立严格关联
默认 report 只统计当前 gate 触发的 evidence
历史 eval 结果进入 trend_history，不参与当前 go/no-go
负例测试插入的数据必须有 source_type=test_injected_negative
诊断 bundle 同时展示 current_run 和 historical_context
```

### 验收

```text
干净 DB 与已有 data/sqlite 上 ReleaseGate 结论一致
历史失败不会让当前 gate 误报 no_go 或 medium failure
真实失败仍会阻断或降级，不会被 scope 过滤掉
phase17 failed_results 只来自当前 gate 对应 eval_run
```
