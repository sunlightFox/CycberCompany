# 第二阶段：聊天主链路与模型路由

## 阶段定位

第二阶段的目标是把第一阶段预埋的聊天骨架接成真实可运行的主链路。

完成后，用户从聊天页输入一句话，系统必须能稳定完成以下闭环：

```text
聊天页输入
  -> /api/chat/turn
  -> Chat Runtime 创建 turn 与 trace
  -> 持久化用户消息
  -> Context Gateway 构建 ContextPacket
  -> Brain 识别意图、选择模式、选择模型
  -> Model Router 给出可解释路由
  -> Model Adapter 调用真实模型
  -> Response Composer 组织最终回复
  -> 事件流推送到聊天页
  -> 持久化助手消息
  -> 关闭 trace
```

这一阶段不是临时聊天能力，也不是只接一个模型接口。它是最终产品中“聊天主入口”的第一块真实能力切片。后续记忆系统、资产中心、任务引擎、Skill、MCP、多成员协作都会接在这条链路上，因此第二阶段必须把链路边界、事件语义、错误模型、trace 语义和路由规则一次立稳。

第二阶段只做一件大事：让聊天主链路真实工作。

但它必须克制：

```text
不做完整长期记忆
不做文件、浏览器、终端等工具执行
不做完整 Task Engine
不做 Skill 和 MCP 执行
不做多成员 supervisor 协作
不把聊天页做成工作台
```

这些能力必须保留最终契约和调用位置，但不能用会被推翻的临时实现假装完成。

## 第一阶段交接基线

第二阶段开始前，第一阶段必须已经交付以下基线：

```text
/api/chat/turn 路由占位
ChatTurnRequest
ChatTurnResponse
ChatEvent
ContextPacket
TraceSpan
messages 表
conversations 表
brains 表
traces 表
trace_spans 表
members 表
shell runtime
chatStore
聊天页最终骨架
管理页信息架构占位
TraceService

```

如果这些基线没有完成，第二阶段不能直接绕过补做，更不能在业务代码里重新发明一套聊天请求、事件流或模型调用结构。

## 第二阶段核心结论

| 结论 | 含义 |
|---|---|
| Chat Runtime 是唯一主链路入口 | 前端、模型路由、Response Composer 都不能绕过它 |
| Context Gateway 必须开始真实构建上下文包 | 但本阶段只纳入成员、会话、近期消息、基础安全信息和壳标签 |
| Brain 必须变成可运行决策层 | 至少完成意图识别、模式选择、模型路由三个子层 |
| Model Router 必须真实工作 | 能按隐私、模型状态、成本、上下文长度、任务类型选择模型 |
| 模型适配必须 OpenAI-compatible 优先 | 本地 Ollama、vLLM 和云端兼容 endpoint 用统一 adapter 接入 |
| Response Composer 必须成为输出闸门 | 不把模型原始输出直接作为唯一产品回复语义 |
| 流式事件必须接到聊天页 | 用户能看到真实 streaming、停止、失败、重试 |
| trace 必须覆盖每个 turn | 模型路由、模型调用、上下文构建、回复编排都可回放 |
| 任务和工具意图只能识别不能执行 | 需要工具时给出产品化说明，不创建假任务、不假装调用工具 |

## 必须遵守的最终态规则

第二阶段从第一行代码开始必须遵守：

```text
聊天页只显示当前成员的人名、头像、状态、消息、输入框和必要动作按钮
聊天页不显示组织、壳、部门树、资产数量、权限等级、复杂右侧面板
前端不直接调用模型 provider
模型调用只能由 Chat Runtime 通过 Brain / Model Router / Adapter 发起
任何模型调用必须有 trace span
任何云端模型路由必须经过隐私判断
高隐私输入禁止云端 fallback
api_key、secret、钱包助记词、本地敏感路径不得进入模型上下文和 trace 明文
壳只改变标签、菜单和文案，不改变底层数据
系统管理及其子菜单不随壳改名
大脑是模型资源，不等于成员人格
成员默认独立，不天然知道组织、资产、其他成员和系统秘密
```

## 阶段目标

| 目标 | 结果 |
|---|---|
| 真实聊天请求 | `/api/chat/turn` 可以接收真实输入并返回事件流 |
| 真实模型配置 | 用户可以配置本地或云端 OpenAI-compatible 模型 |
| 真实模型路由 | 系统可以按策略选择 primary 和 fallback 模型 |
| 真实 direct 回复 | 普通聊天、解释、写作、总结可以生成可用回复 |
| 基础上下文包 | ContextPacket 能包含成员、人格、近期消息、会话摘要、隐私标记 |
| 基础意图识别 | 能区分闲聊、问答、创作、总结、记忆询问、系统设置、任务执行意图 |
| 基础模式选择 | 能选择 direct、direct_with_memory、workflow、agent、supervisor，并知道当前哪些可执行 |
| Response Composer 输出 | 用户看到的是经过产品层整理的回复，而不是裸模型流 |
| 流式 UI | 聊天页支持 streaming、停止、失败、重试 |
| 消息持久化 | 用户消息、助手消息、turn_id、trace_id、路由元数据正确落库 |
| trace 闭环 | 每轮对话都能查看模型路由、耗时、token、错误和 fallback |
| 安全基线 | 高隐私输入不走云端，secret 不落日志，不进入 prompt |

## 小阶段 2.1：第一阶段契约复核与补齐

### 目标

在写真实聊天逻辑前，先确认第一阶段的契约能承载第二阶段真实模型调用。

第二阶段不允许出现这样的情况：

```text
ChatTurnRequest 不够用，于是在 handler 里临时加散字段
ChatEvent 不够用，于是在前端私下约定字符串
ContextPacket 不够用，于是在 prompt builder 里直接查数据库
TraceSpan 不够用，于是在日志里写不可回放文本
```

如果契约需要补齐，必须在共享类型层补齐，并同步更新后端、前端和测试。

### 任务

| 编号 | 任务 | 说明 |
|---:|---|---|
| 2.1.1 | 复核 `ChatTurnRequest` | 确认支持 session、conversation、member、input、attachments、client_context |
| 2.1.2 | 复核 `ChatTurnResponse` | 确认支持 turn_id、conversation_id、message_id、trace_id、status |
| 2.1.3 | 补齐 `ChatEvent` | 增加 intent、mode、route、fallback、cancel、error 相关事件 |
| 2.1.4 | 复核 `ContextPacket` | 确认能承载 member、persona、conversation、memories、capabilities、resource_handles、safety_notes |
| 2.1.5 | 复核错误模型 | 确认模型不可用、隐私拦截、超时、取消、流中断都有错误码 |
| 2.1.6 | 复核 trace span | 确认每个核心步骤都有 span 类型和 redaction 规则 |
| 2.1.7 | 建立契约测试 | 前后端共享 schema 变更必须触发测试 |

## 小阶段 2.2：大脑配置与模型资产接入

### 目标

把“大脑”从第一阶段占位变成真实可配置、可验证、可被成员绑定的模型资源。

本产品的核心原则是：

```text
成员不是模型
成员 = 人格、记忆、身份、工作状态
大脑 = 推理资源、endpoint、模型能力、成本和隐私策略
```

第二阶段只接入大脑资产。账号、钱包、硬件、知识库等其他资产留给第四阶段。

### 页面入口

前端可以从两个位置进入大脑配置：

```text
资产中心 / 大脑
系统管理 / 模型路由
```

规则：

```text
资产中心一级菜单名称可以受壳映射影响
资产二级分类“大脑”固定不改名
系统管理名称固定
系统管理子菜单名称固定
聊天页不得出现模型配置入口
聊天页只在模型未配置时显示轻量错误和去配置动作
```

### 后端数据源

`brains` 表是第二阶段大脑运行配置的主数据源。



## 小阶段 2.3：OpenAI-compatible 模型适配层

### 目标

建立统一模型调用接口，屏蔽 Ollama、vLLM、云端兼容 endpoint 的差异。

第二阶段不追求多供应商特性覆盖，而是先把最终产品需要的共同能力接稳：

```text
chat completion
streaming
timeout
cancel
fallback
token usage
错误归一化
trace 记录
```

### 文件目标

```text
services/brain/adapters/
  base.py
  openai_compatible.py
  ollama.py
  vllm.py
  errors.py
  types.py
  token_estimator.py
```

### 统一接口

```python
class ChatModelClient:
    async def stream_chat(
        self,
        request: ModelChatRequest,
        cancel_token: CancelToken,
    ) -> AsyncIterator[ModelStreamEvent]:
        ...

    async def complete_chat(
        self,
        request: ModelChatRequest,
        cancel_token: CancelToken,
    ) -> ModelChatResult:
        ...
```

### ModelChatRequest

必须包含：

```text
model
messages
temperature
max_output_tokens
top_p
timeout_seconds
stream
trace_id
turn_id
route_id
privacy_level
metadata
```

禁止包含：

```text
api_key 明文
secret 明文
本地敏感文件内容
未经 Context Gateway 标记的外部内容
```

### ModelStreamEvent

事件类型：

```text
started
delta
usage_delta
completed
failed
cancelled
```

模型 delta 不直接发给前端。它必须先进入 Response Composer 或 direct streaming composer，再由后者输出 `response.delta`。

### 错误归一化

| 原始问题 | 统一错误 |
|---|---|
| 401 / 403 | MODEL_AUTH_FAILED |
| 404 model not found | MODEL_NOT_FOUND |
| 连接失败 | MODEL_UNAVAILABLE |
| 超时 | MODEL_TIMEOUT |
| streaming 中断 | MODEL_STREAM_INTERRUPTED |
| 上下文超限 | MODEL_CONTEXT_TOO_LARGE |
| provider 返回格式不合法 | MODEL_PROTOCOL_ERROR |

### 超时与重试

默认策略：

```text
连接超时：10 秒
首 token 超时：30 秒
总超时：180 秒
网络错误可重试 1 次
认证错误不重试
上下文超限不重试，交给 Context Gateway 降采样后再走 fallback
用户取消不重试
```

### 取消生成

取消必须穿透：

```text
前端停止按钮
  -> Chat Runtime cancel token
  -> Model Adapter 关闭 streaming 请求
  -> Chat Runtime 发送 turn.cancelled
  -> 消息以 cancelled 状态记录到 content_json
  -> trace span 标记 cancelled
```

### 验收

```text
本地兼容 endpoint 可以 streaming 输出
远程兼容 endpoint 可以 streaming 输出
模型认证失败返回 MODEL_AUTH_FAILED
模型不可用返回 MODEL_UNAVAILABLE
模型超时返回 MODEL_TIMEOUT
用户点击停止后模型请求被取消
所有模型调用都有 model.call span
trace 中不出现 api_key 明文
前端不接触 provider SDK
```

## 小阶段 2.4：Model Router 路由策略

### 目标

让系统能够根据隐私、任务类型、模型健康、上下文长度、成本和延迟选择合适模型。

Model Router 是 Brain 的一部分，但它不是模型。它是可解释、可测试、可审计的策略层。

### 路由输入

```text
member_id
default_brain_id
conversation_id
intent
mode
privacy_level
sensitivity_hits
estimated_input_tokens
reserved_output_tokens
requires_strong_reasoning
requires_tool_calling
requires_vision
latency_budget
cost_budget
allow_cloud
available_brains
model_routing_config
client_context
```

### 路由输出

```json
{
  "route_id": "route_001",
  "primary_brain_id": "brn_local_main",
  "fallback_brain_ids": ["brn_cloud_strong"],
  "reason_codes": ["daily_chat", "privacy_medium", "local_first"],
  "privacy_policy": "allow_cloud_without_secrets",
  "context_budget": {
    "max_input_tokens": 12000,
    "reserved_output_tokens": 2000
  },
  "model_params": {
    "temperature": 0.5,
    "top_p": 0.9,
    "max_output_tokens": 2000
  }
}
```

### 路由规则

必须按顺序执行：

1. 过滤 disabled / unhealthy / not_configured 大脑。
2. 根据 `privacy_level` 过滤不允许的云端大脑。
3. 根据 `requires_vision`、`requires_tool_calling` 过滤不支持能力的大脑。
4. 根据 `estimated_input_tokens + reserved_output_tokens` 过滤上下文窗口不足的大脑。
5. 优先使用成员默认大脑。
6. 如果默认大脑不满足，按路由配置选择 local_fast、local_main、cloud_strong。
7. 为 primary 选择合法 fallback。
8. 生成 reason_codes。
9. 写入 trace。

### 隐私级别

| 级别 | 示例 | 云端策略 |
|---|---|---|
| low | 普通知识问答、公开资料改写 | 可使用云端 |
| medium | 项目方案、个人偏好、非敏感对话 | 用户允许时可使用云端 |
| high | 密码、密钥、钱包、身份证、敏感本地路径、私密文件内容 | 禁止云端 |

### 敏感命中规则

第二阶段至少实现规则检测：

```text
password / 密码 / 口令
api_key / token / secret / 私钥
助记词 / seed phrase / mnemonic
身份证 / 银行卡 / 支付密码
本地绝对路径且包含用户目录
ssh key / private key
cookie / session
```

命中高隐私后：

```text
privacy_level = high
allow_cloud = false
fallback 中移除所有非本地大脑
如果没有本地 healthy 大脑，返回 MODEL_ROUTE_BLOCKED_BY_PRIVACY
```

### 路由配置

`config/model-routing.yaml` 在第二阶段变成真实配置：

```yaml
routing:
  default: local_main
  privacy:
    high:
      allow_cloud: false
      route: local_main
    medium:
      allow_cloud: true
      route: local_main
      fallback: cloud_strong
    low:
      allow_cloud: true
      route: cloud_strong
  tasks:
    intent_classify: local_fast
    daily_chat: local_main
    writing: local_main
    summarization: local_main
    planning: cloud_strong
    response_compose: local_fast
```

配置规则：

```text
配置里的 route key 必须能映射到 healthy brain
找不到 route key 时使用成员 default_brain
成员 default_brain 不可用时使用 routing.default
所有 fallback 必须重新经过隐私过滤
用户关闭云 fallback 后，所有 fallback 云端项失效
```

### 验收

```text
普通闲聊可路由到 local_main
复杂写作在用户允许云端时可路由到 cloud_strong
包含 api_key 的输入不会路由到云端
本地模型不可用时可 fallback 到云端，但高隐私输入不允许
没有可用模型时返回可理解错误
route.selected 事件包含 reason_codes
trace span 中记录 primary、fallback、privacy_level、reason_codes
路由单测覆盖隐私、fallback、上下文超限、模型不可用
```

## 小阶段 2.6：Brain 意图识别与模式选择

### 目标

把 Brain 从目录占位变成可运行的决策层。

第二阶段 Brain 必须完成三件事：


这里的“当前阶段能否完成”不能变成产品里的阶段文案。产品表达应该是能力表达：

## 小阶段 2.7：Chat Runtime turn 生命周期

### 目标

实现 Chat Runtime 的主编排逻辑。

Chat Runtime 是聊天主链路唯一入口。它负责把请求、上下文、决策、模型、回复、持久化、trace 串成一个可回放的 turn。

### 文件目标

```text
services/chat-runtime/
  runtime.py
  events.py
  session.py
  turn_state.py
  cancel.py
  errors.py
```

### 生命周期

```text
1. 接收 ChatTurnRequest
2. 校验 member_id、conversation_id、input
3. 创建 turn_id
4. 创建 trace
5. 发送 turn.started
6. 保存用户消息
7. 发送 context.started
8. 调用 Context Gateway
9. 发送 context.ready
10. 调用 IntentClassifier
11. 发送 intent.detected
12. 调用 ModeSelector
13. 发送 mode.selected
14. 调用 ModelRouter
15. 发送 route.selected
16. direct 路径调用模型
17. Response Composer 输出 response.delta
18. 保存助手消息
19. 发送 response.completed
20. 更新会话标题和摘要
21. 关闭 trace
22. 发送 turn.completed
```

失败路径：

```text
任意步骤失败
  -> 记录错误 span
  -> 生成产品化失败回复或 turn.failed
  -> 保存必要状态
  -> 关闭 trace
```

取消路径：

```text
用户点击停止
  -> cancel token 置位
  -> 中断模型流
  -> 保存 partial assistant message
  -> 发送 turn.cancelled
  -> 关闭 trace
```

### 状态机

```text
created
validating
persisting_user_message
building_context
classifying_intent
selecting_mode
routing_model
calling_model
composing_response
persisting_assistant_message
completed
cancelled
failed
```

状态必须写入内存 turn state。是否落库可以放在后续任务系统中，但 trace 必须记录。

### direct 与非 direct

第二阶段真正执行：

```text
direct
direct_with_memory 的当前会话可见部分
```

第二阶段不执行：

```text
workflow
agent
supervisor
```

当模式不可执行时：

```text
不创建 task
不发送 task.created
不调用 tool
不调用 Skill
Response Composer 生成“可先给计划、需要能力和确认才能执行”的回复
```

### 验收

```text
每次聊天都有 turn_id 和 trace_id
turn.started 一定早于 response.delta
response.completed 一定早于 turn.completed
用户消息先于模型调用落库
助手消息在完成或取消时落库
失败时 trace 能看到失败步骤
非 direct 请求不会创建 task
非 direct 请求不会调用 tool
```

## 小阶段 2.8：Direct Answer 执行路径

### 目标

实现普通聊天、问答、写作、总结的真实生成路径。

Direct Answer 是第二阶段的主要用户价值。它必须足够稳定，因为后续所有复杂任务也会在阶段性反馈、总结、失败解释中复用这条生成链路。

### Prompt 构建层

文件目标：

```text
services/response-composer/

  direct_answer.py
  style_policy.py
```

Prompt 构建顺序：

```text

会话摘要
近期消息
当前用户输入
输出格式要求
```

禁止：


### Direct Answer 类型

| 类型 | 处理 |
|---|---|
| chat | 自然对话，保持成员人格和边界 |
| question_answer | 解释、回答、推理 |
| creative_writing | 方案、文档、文案、结构化输出 |

### 模型参数

默认参数按 intent 调整：

| intent | temperature | 输出策略 |
|---|---:|---|
| chat | 0.6 | 自然、简洁 |
| question_answer | 0.3 | 准确、结构化 |
| creative_writing | 0.5 | 可展开、保持主题 |
| summarization | 0.2 | 忠于上下文 |
| memory_query | 0.2 | 明确可见范围 |

这些参数由 Model Router 输出，Response Composer 可以提出建议，但最终由 Model Router 合并。

### 长输出控制

规则：

```text

```

### 验收

```text

direct 回复通过 response.delta 流式展示
最终助手消息与 UI 展示内容一致
```

## 小阶段 2.9：Response Composer 基础实现

### 目标

让 Response Composer 成为最终输出闸门。

第二阶段不能把模型原始流直接等同于产品回复。即使 direct 路径中模型输出占主要内容，也必须经过 Response Composer 管理：

```text
语气
结构
失败解释
安全提示
模式不可执行时的产品化回复
最终 response.completed 元数据
```

### 输入

```text
用户原始输入
ContextPacket
intent result
mode result
model route
model stream
safety result
persona summary
heart suggestion
```

### 输出

```text
response.delta
response.completed
assistant_message_content
response_metadata
```

### 基础策略

| 场景 | 策略 |
|---|---|
| direct 成功 | 模型内容经轻量整理后流式输出 |
| 模型不可用 | 说明模型配置或连接问题，给配置入口动作 |
| 隐私拦截 | 说明不能把敏感内容发往云端，建议配置本地模型 |
| 需要工具 | 说明需要对应能力和确认，先给计划 |
| 需要长期记忆 | 说明当前只能看见本会话内容，避免编造历史 |
| 用户取消 | 保留已生成内容，显示已停止 |
| provider 错误 | 转成人能理解的失败原因 |

### 语气规则

```text
结论先行
不暴露内部 prompt
不暴露 trace 细节
不把 provider 错误原文整段贴给用户
不使用过度拟人和操控性表达

```

### response.completed payload

```json
{
  "message_id": "msg_assistant_001",
  "finish_reason": "stop",
  "usage": {
    "input_tokens": 1200,
    "output_tokens": 860
  },
  "route": {
    "primary_brain_id": "brn_local_main",
    "fallback_used": false
  },
  "mode": "direct"
}
```

### 验收

```text
所有用户可见文本都来自 Response Composer

取消生成后 UI 展示部分内容和停止状态

```

## 小阶段 2.10：事件流与聊天页接入

### 目标

把后端事件流接到聊天页，让用户获得真实、稳定、克制的聊天体验。

第二阶段采用 SSE 作为首个事件传输实现。事件抽象必须保持 transport-neutral，后续如果接 WebSocket，不改变 ChatEvent 语义。

## 小阶段 2.11：会话、消息与标题持久化

### 目标

让聊天记录成为后续记忆、任务、trace、评测的可信来源。

第二阶段必须保证：

```text
用户消息一定可追溯
助手消息一定可追溯
每条助手消息能关联 trace
每轮 turn 能关联路由和模型调用
会话标题可用
重试不会覆盖历史
取消不会丢失已生成内容
```

### 写入规则

用户消息：

```text
收到请求并通过基础校验后立即写入
author_type = user
author_id = local_user
turn_id = 当前 turn
trace_id = 当前 trace
content_text = 用户文本
content_json = 输入结构、附件元数据、client_context 摘要
```

助手消息：

```text
response.completed 后写入完整内容
turn.cancelled 后写入 partial 内容
turn.failed 且有产品化失败回复时写入失败回复
author_type = member
author_id = 当前 member_id
trace_id = 当前 trace
content_json 包含 finish_reason、usage、route、mode、status
```

不得写入：

```text
api_key 明文
模型完整原始响应对象
未脱敏 provider header
secret_ref 对应真实值
```

### 会话创建

规则：

```text
如果 request 没有 conversation_id，则创建 single conversation
primary_member_id = 当前聊天成员
participant_json 至少包含用户和当前成员
status = active
conversation_type = single
```

### 标题生成

标题生成策略：

1. 第一条用户消息少于 30 个中文字符时，直接截断生成标题。
2. 第一条用户消息较长时，用规则提取前 12 到 18 个字。
3. 如果配置了 local_fast，可以异步生成更好标题。
4. 标题生成失败不影响 turn。
5. 标题不得包含 secret、api_key、钱包助记词。

### 重试规则

```text
重试创建新 turn
不覆盖原用户消息
不覆盖原助手消息
新 turn 的 client_context.retry_of_turn_id 指向原 turn
trace 中记录 retry_of_turn_id
UI 可将重试结果显示在原失败消息之后
```

### 验收

```text
新会话能自动创建
用户消息和助手消息都能落库
消息按 created_at 正确排序
刷新页面后能恢复会话
重试不会覆盖历史消息
取消后 partial 消息仍可见
助手消息 content_json 包含 route、usage、finish_reason
messages_fts 能检索 content_text
```

## 第二阶段风险与缓解

| 风险 | 后果 | 缓解 |
|---|---|---|
| 只接一个模型 endpoint | 后续本地/云混合要重写 | 第二阶段必须实现 Model Adapter + Model Router |
| 把模型配置写进前端 | secret 泄露、不可审计 | 前端只拿 masked 配置和 secret_ref |
| 让 direct 路径绕过 Response Composer | 后续任务回复风格分裂 | 所有输出都经 Response Composer |
| 过早接工具 | 安全边界未成型 | 工具意图只识别，不执行 |
| 过早做长期记忆 | 记忆写入和冲突规则混乱 | 第二阶段只做会话摘要，长期记忆放第三阶段 |
| trace 只写日志 | 不能回放、不能评测 | trace_spans 结构化写入 |
| cloud fallback 太激进 | 敏感信息外发 | privacy_level high 强制本地，无本地则失败 |
| 聊天页暴露模型细节 | 极简体验被破坏 | 模型细节放系统管理和 trace，不放聊天主界面 |
