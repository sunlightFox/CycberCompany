# 第七十二阶段 - 浏览器任务会话连续性与页面状态重构

## 阶段背景

当前浏览器能力分散在：

```text
apps/local-api/app/services/browser_workflows.py
apps/local-api/app/services/browser_executor.py
apps/local-api/app/services/browser_sessions.py
apps/local-api/app/services/tools.py
apps/local-api/app/services/chat.py
```

虽然已经具备自动浏览器工作流雏形，但在聊天质量上仍有明显短板：

```text
浏览器任务与普通聊天 turn 的状态连续性不够统一
页面状态、会话句柄、登录态、证据链没有形成独立 runtime
聊天入口的 browser read 仍像旁路
执行中、等待确认、已观察、需登录、需人工接管这些语义还不够稳
```

本阶段目标是对齐 Hermes 的“同一 task_id 复用浏览器/终端状态”思路，把浏览器任务整理成会话连续、可恢复、可回放的后端执行底座。

## 核心目标

完成后，浏览器系统应满足：

```text
同一浏览器任务拥有稳定 session runtime
页面观察、页面摘要、页面动作、回放证据分层
聊天入口 browser read 与浏览器工作流共用底层 runtime
登录态、challenge、approval boundary、人工接管点语义清晰
浏览器任务恢复不依赖 chat.py 大文件中的局部补丁
```

## 破坏性调整

### 新浏览器子模块

将 `browser_workflows.py` 拆为：

```text
apps/local-api/app/services/browser_workflow_runtime.py
apps/local-api/app/services/browser_intent_resolver.py
apps/local-api/app/services/browser_plan_runtime.py
apps/local-api/app/services/browser_session_runtime.py
apps/local-api/app/services/browser_page_state.py
apps/local-api/app/services/browser_replay_store.py
```

### 新页面状态语义

统一页面状态：

```text
observed
actionable
login_required
approval_required
challenge_detected
handoff_required

```

## 实现要求

### 1. 建立 BrowserSessionRuntime

主接口建议：

```python
class BrowserSessionRuntime:
    async def get_or_create(self, task_id: str, member_id: str, ...) -> BrowserSessionRef:
        ...

    async def snapshot_page(self, ...) -> dict[str, Any]:
        ...

    async def record_state(self, ...) -> None:
        ...
```

要求：

```text
同一 task_id 下复用会话句柄
snapshot / click / fill / navigate / download 使用统一上下文
登录页、挑战页、提交页、结果页状态独立记录
```

### 2. 拆开“观察”和“动作”

要求：

```text
browser.snapshot / page summary 属于 observe
click / fill / submit / upload / download 属于 act
observe 默认低风险
act 需结合 Safety、Approval、Capability、Asset Broker 决策
聊天入口 browser read 只能走 observe
```

### 3. 建立页面状态摘要

新增 `browser_page_state.py`，至少抽象：

```text
current_url
page_title
primary_forms
primary_actions
login_required
challenge_markers
download_candidates
safe_summary
```

要求：

```text
供 chat、task、response composer 共用
不要在多个模块里各自临时拼页面摘要
```

### 4. 建立 Replay 与 Evidence Store

要求：

```text
浏览器任务执行链路有统一 evidence refs
页面观察证据、截图证据、DOM/AX 摘要证据分开
response_plan 只引用 evidence，不泄露内部细节
恢复时可读取最近页面状态而不是重头猜测
```
