# 第九十四阶段 - 失败经验沉淀与自增长治理闭环

## 阶段定位

前面阶段主要靠：

```text
发现问题
补测试
修主链
加 gate
```

这条路线有效，但越来越依赖人工持续盯盘。  
第九十四阶段要补的是更高一层成熟度：**让系统把失败经验结构化回收，并用于后续判断、回复和执行治理。**

## 阶段目标

```text
把失败、回退、误拦截、误完成、渠道异常等经验转成结构化经验记录
建立 experience -> memory / policy hint / regression candidate 的闭环
让系统能利用过去失败经验降低重复犯错概率
同时限制自增长边界，避免把坏经验写成新规则
```

## 核心问题

当前项目已经有大量 release gate 和诊断，  
但“知道出了什么问题”和“下次少犯同类问题”之间还没完全打通。

```text
失败经验主要留在报告、文档、测试和人工记忆里
系统对 recurring failure 缺少自动归档与召回
经验沉淀还没有稳定区分临时异常、真实模式、长期策略
```

## 本阶段范围

### 必须完成

```text
建立 structured failure experience contract
让关键失败可沉淀成经验记录，并可被 ContextGateway / BrainDecision 消费
建立经验写入审批与质量门槛
把 recurring failure 自动转成 regression candidate
```

### 明确不做

```text
不让模型自由修改核心策略文件
不把单次偶发错误直接升级成长期规则
不绕过人工审查去自动发布 skill / prompt / policy 变更
```

## 实施拆解

### 94.1 failure experience contract

目标：

```text
把失败经验从自由文本报告变成结构化对象
```

交付：

```text
failure experience schema
reason_code / evidence_ref / impact_scope / recurrence_count 字段
```

### 94.2 经验沉淀与召回

目标：

```text
让系统在相似新问题出现时，能参考过去经验而不是重新踩坑
```

交付：

```text
experience memory write path
experience recall path
policy advisory hints from prior failures
```

### 94.3 regression candidate 自动化

目标：

```text
让重复问题自动进入测试或门禁候选清单
```

交付：

```text
recurring failure detector
regression candidate queue
release summary linkage
```

### 94.4 自增长边界治理

目标：

```text
防止系统把噪音、错判或偶发脏数据沉淀成坏规则
```

交付：

```text
experience write safety checks
minimum evidence threshold
human-review required classes
rollback and tombstone semantics
```
