# 第五十二阶段 - 聊天驱动项目部署与软件安装执行闭环

## 阶段背景

前五十一阶段已经具备聊天主链路、任务引擎、工具执行、安全审批、终端沙箱、浏览器工具、外部平台动作和质量回归的基础能力。但当前“帮我部署 GitHub 项目”“帮我安装 npm”“帮我安装桌面软件”这类真实用户请求仍没有产品级闭环：

```text
聊天意图没有把安装、部署、GitHub 项目、运行时缺失识别为一等执行任务
任务规划不会稳定生成 clone、detect、install、build、run、health check、stop 等步骤
terminal.run 已存在，但仍是裸命令工具，不是项目部署工作流
默认 sandbox allowlist 未覆盖 node/npm/pnpm/yarn/docker/winget/choco 等生态命令
没有项目工作区、运行时 toolcache、端口租约、后台进程、部署日志和健康检查模型
没有区分“项目内依赖安装”和“改本机全局环境/安装桌面软件”
```

本阶段目标是参考 OpenClaw 的 host exec/approval 绑定思路和 Hermes Agent 的多执行后端思路，把本项目的聊天主链路打通为：

```text
用户一句话
-> 创建受控项目/安装任务
-> 选择执行后端
-> 生成计划并展示影响范围
-> 用户确认
-> 执行 clone/install/build/run 或 host install
-> 返回访问地址、日志、版本、失败原因和可继续动作
```

本阶段仍只做后端、schema、migration、repository、service、API、tests、evals 和文档；不新增前端页面、组件、样式、Tauri 窗口或桌面端交互代码。

## 参考结论

| 参考 | 值得学习 | 不直接照搬 |
|---|---|---|
| OpenClaw | `exec` host/sandbox/node 分层、审批绑定 command/cwd/env/argv，防止确认后换命令 | 不默认开启 YOLO，不把裸 shell 暴露为主要产品能力 |
| Hermes Agent | local/docker/ssh/modal/daytona 等多终端后端，适合部署类任务体验 | 不默认让 local backend 无隔离直接拥有用户文件系统权限 |
| 本项目 | Task/Safety/Approval/Trace/Asset Broker/Capability Graph 已有底座 | 需要新增项目部署和软件安装的业务语义层 |

阶段核心取舍：

```text
默认：在项目级沙箱或受控项目工作区里尽量自动完成
降级：没有容器/WSL 时使用本机受控工作区 + portable toolcache
强确认：需要改本机全局环境、安装桌面软件、写系统目录、改环境变量时走 host executor
拒绝：驱动、内核扩展、钱包插件、浏览器扩展、未知来源安装包、绕过 UAC/验证码/安全提示
```

## 核心目标

本阶段完成后，后端应支持：

```text
聊天识别“部署 GitHub 项目、安装依赖、启动项目、安装运行时、安装桌面软件”
自动创建 ProjectWorkspace，并固定工作目录和文件边界
选择执行后端：container / wsl / local_workspace / host_executor
无容器时降级到本机受控项目工作区，不直接污染用户目录
缺 Node/npm/Python 等运行时时，优先安装 portable toolchain 到系统托管 toolcache
桌面软件和全局运行时安装走 host install plan，必须强审批
项目部署支持 clone、detect stack、install deps、build、test、run、health check、read logs、stop
后台进程有 process_id、port lease、log artifact、status、stop/restart 语义
审批绑定命令、cwd、执行后端、目标路径、环境变量、下载来源、hash 和影响范围
所有模型调用、工具调用、审批、安装、进程、端口、日志、记忆写入有 trace
所有 secret、token、cookie、password、private key、本地敏感路径继续脱敏
```

## 阶段原则

1. 聊天入口可以自然，但执行入口必须是后端受控工作流，不让模型直接拼任意 shell。
2. 项目内依赖安装和宿主机软件安装是两个风险等级，不能混在一个 `terminal.run` 概念里。
3. 默认执行位置是 `ProjectWorkspace`，不是用户桌面、下载目录、系统盘或仓库根目录。
4. 没有容器不是阻断条件，但必须明确降级为 `local_workspace_policy_guard`。
5. 能用 portable toolchain 就不改全局系统环境。
6. host executor 只用于用户明确要求改本机环境或安装桌面软件的场景。
7. host install 必须展示来源、命令、路径、签名/hash、权限需求、回滚建议和不可回滚风险。
8. 不自动点击 UAC，不绕过安装器安全提示，不绕过软件许可、验证码或平台风控。
9. 执行成功必须有证据：日志、exit code、版本、端口、health check、artifact 或进程状态。
10. 不新增 UI；后端返回未来聊天、CLI、通知或管理页可消费的结构化状态。

## 用户体验目标

### 部署 GitHub 项目

用户输入：

```text
帮我部署 https://github.com/example/app 这个项目，跑起来给我地址。
```

目标回复流程：

```text
我会在隔离项目工作区里克隆这个仓库、识别技术栈、安装依赖、构建并启动预览。
这不会改你的系统全局环境。需要联网下载依赖，可能占用本地端口。

你确认后我继续。
```

确认后目标结果：

```text
项目已经启动。

访问地址：http://127.0.0.1:5173
执行结果：依赖安装成功，构建成功，健康检查通过
日志摘要：...

你可以继续让我查看页面、停止服务或修复启动报错。
```

### 缺少 npm

用户输入：

```text
帮我安装 npm。
```

目标回复流程：

```text
我可以优先安装一个本项目托管的 portable Node.js/npm，不改系统全局环境。
如果你想装到整台电脑，也可以走本机安装，但需要强确认，可能触发管理员权限。

你想用哪种方式？
```

后端策略：

```text
portable toolcache：默认推荐，R3/R4，写入 data/toolchains
host global install：强确认，R5/R6，使用 winget/choco/msi/exe 安装
```

### 安装桌面软件

用户输入：

```text
帮我安装 VS Code。
```

目标回复流程：

```text
这是本机软件安装，会修改系统应用列表和可能的环境变量。
我会先查找可信安装来源，展示安装命令、来源和影响范围。你确认后再执行。
```

强制边界：

```text
未知下载站 exe 不静默安装
签名或 hash 不可验证时提高风险或停止
安装器要求人工确认/UAC 时停止等待用户人工处理
安装完成后记录版本和安装路径摘要
```

## 执行后端模型

| 后端 | 场景 | 默认风险 | 说明 |
|---|---|---:|---|
| `container` | Docker 可用的项目部署 | R3-R4 | 最推荐，隔离强，依赖可自由安装 |
| `wsl` | Windows 上 Linux 项目部署 | R3-R4 | 适合 Node/Python/Go/Rust 项目 |
| `local_workspace` | 无容器/WSL 时的项目部署 | R4-R5 | 固定 cwd、最小 env、allowlist、policy guard |
| `toolcache` | 安装 portable Node/Python 等运行时 | R3-R4 | 写入系统托管目录，不改全局 PATH，任务内引用 |
| `host_executor` | 全局安装软件、桌面软件、系统运行时 | R5-R6 | 必须强审批，可能需要用户人工 UAC |
| `manual_only` | 驱动/内核/钱包/浏览器扩展/未知来源安装器 | R6-R7 | 默认只给人工步骤或拒绝自动执行 |

执行后端选择规则：

```text
项目部署优先 container，其次 wsl，其次 local_workspace
缺运行时优先 toolcache，不优先 host_executor
明确要求“装到电脑全局/桌面软件/系统 PATH”才进入 host_executor
涉及驱动、系统服务、自启动、安全软件、钱包、浏览器扩展默认 manual_only
```

## 安全与审批模型

### 项目部署审批

以下动作必须审批：

```text
首次 clone 外部仓库
下载并执行依赖安装脚本
运行 package scripts、postinstall、setup.py、install.sh
启动长期进程或打开本地端口
访问私有 registry 或需要资产 handle 的依赖源
```

审批绑定字段：

```text
source_uri
commit_sha 或 ref
workspace_id
backend_type
working_dir_policy
network_policy
install_commands_redacted
run_command_redacted
port_range
toolchain_ids
```

### host 安装审批

host install 永远强审批，审批绑定：

```text
requested_software
installer source
package manager
command preview
publisher/signature/checksum
target scope
requires_admin
writes_system_locations
modifies_path_or_registry
rollback suggestion
```

### 默认拒绝项

```text
未知来源 exe/msi 且无法校验签名或 hash
驱动、内核扩展、虚拟网卡、系统安全软件
钱包、浏览器扩展、密码管理器扩展自动安装
要求绕过 UAC、关闭杀毒、禁用防火墙、绕过安全提示
安装盗版、破解、keygen、外挂、恶意用途工具
在用户目录、系统目录、浏览器 profile、钱包目录执行写入
```

## 本机无容器时的降级方案

无容器/WSL 时仍支持项目部署，但必须显式降级：

```text
backend_type = local_workspace
workspace_root = data/workspaces/projects/{workspace_id}
env_policy = minimal
filesystem_policy = workspace_root_only
network_policy = external_read_requires_approval
allowed_executables = git,node,npm,pnpm,yarn,python,py,pip,curl,wget
denied_executables = powershell,cmd,bash,sh,rm,del,reg,msiexec unless host_executor
```

需要注意：

```text
local_workspace 不是强隔离沙箱，release report 必须标注 degraded isolation
install_deps 可写 node_modules/.venv 等项目目录
不允许写全局 npm prefix、系统 PATH、用户 home 配置
如果依赖安装脚本尝试访问敏感路径，应阻断并生成失败证据
```

## portable toolcache

本阶段应优先支持 portable Node.js/npm：

```text
data/toolchains/node/{version}/
data/toolchains/python/{version}/
data/toolchains/ffmpeg/{version}/
```

规则：

```text
下载来源必须是可信 manifest 或用户确认的来源
记录 checksum、version、source_uri
任务运行时临时注入 PATH，不修改系统 PATH
toolchain 可被多个 ProjectWorkspace 复用
toolchain 输出和下载日志进入 artifact
```

## 小阶段 52.1：契约冻结与威胁边界

### 目标

把“部署项目”和“安装软件”的能力边界写入后端契约，避免后续把所有事情塞进 `terminal.run`。

### 必须完成

```text
定义 ProjectDeployRequest / ProjectDeployPlan / HostInstallPlan / ToolchainInstall schema
定义执行后端枚举和降级语义
定义默认拒绝项和 manual_only 场景
定义审批绑定字段
定义聊天可见文案中的状态语义：未确认、执行中、成功、失败、需人工
```

### 验收

```text
“部署 GitHub 项目”不会被当成普通闲聊
“只解释部署方案，不要执行”不会创建任务
“安装桌面软件”不会进入项目工作区路径
“跳过确认安装”被安全拒绝
```

## 小阶段 52.2：Schema、migration、repository

### 目标

建立可恢复、可审计、可回放的数据层。

### 必须完成

```text
新增 migration：project_workspaces、project_deployments、toolchain_installs、host_install_plans、host_install_executions、managed_processes、port_leases
新增 Pydantic schema，不把裸 dict 作为长期接口
新增 repository，支持 create/get/list/update_status/append_evidence
索引 task_id、workspace_id、deployment_id、status、created_at
```

### 验收

```text
空库 migration 可初始化
重复运行 migration 不破坏旧表
repository 返回结构化类型
敏感字段只保存 redacted 或 secret_ref/handle
```

## 小阶段 52.3：ExecutionBackendSelector

### 目标

根据本机能力和任务风险选择执行后端。

### 必须完成

```text
检测 docker 是否可用
检测 WSL 是否可用
检测本机 portable toolcache 是否可用
检测 host package manager：winget/choco/msiexec
输出 backend_candidates、selected_backend、degraded_reason、blocked_reason
把选择结果写入 trace 和 deployment plan
```

### 验收

```text
Docker 可用时优先 container
Docker 不可用但 WSL 可用时选择 wsl
二者都不可用时选择 local_workspace，并标注 degraded isolation
host install 不被误选为项目部署默认后端
```

## 小阶段 52.4：ProjectWorkspaceService

### 目标

让源码、依赖、日志和运行产物都有固定边界。

### 必须完成

```text
工作区根目录固定为 data/workspaces/projects/{workspace_id}
clone 支持公开 GitHub HTTPS URL
记录 source_uri、ref、commit_sha、clone_log_artifact_id
拒绝 file://、UNC、敏感本地路径、路径穿越
工作区可以 archive，但不自动删除用户数据
```

### 验收

```text
公开 GitHub fixture 可 clone 到工作区
非 GitHub 普通 HTTPS git URL 可按策略处理或澄清
敏感路径和本地用户目录被拒绝
clone 失败有日志 artifact 和可恢复提示
```

## 小阶段 52.5：StackDetector 与部署 planner

### 目标

让项目部署计划稳定，不依赖模型临场猜命令。

### 支持栈优先级

```text
Node: package.json / lockfile / scripts
Python: pyproject.toml / requirements.txt / uv.lock
Static: index.html / vite config / package scripts
Go/Rust: go.mod / Cargo.toml 可先识别并给受限计划
Unknown: 只生成分析报告和询问，不执行 install
```

### 必须完成

```text
读取 manifest 时只在 workspace 内
识别 package manager：npm/pnpm/yarn
识别 install/build/dev/start/test scripts
生成计划前对 scripts 做风险扫描
postinstall、prepare、install.sh 等脚本标记高风险
```

### 验收

```text
Vite/React fixture 生成 npm install + npm run dev 计划
无 package.json 不执行 npm
危险 script 被标为 approval_required 或 deny
未知栈不伪装可部署
```

## 小阶段 52.6：ToolchainService

### 目标

让缺少 npm/Node/Python 时仍能继续，但默认不改宿主全局环境。

### 必须完成

```text
托管 portable Node.js/npm runtime
runtime manifest 包含 version、source_uri、checksum、platform、archive_type
下载、解压、校验、注册均写 trace
任务运行时临时 PATH 注入 toolchain bin
host 全局安装作为显式替代路径，不作为默认
```

### 验收

```text
缺 npm 时可以生成 runtime.ensure 计划
toolcache 安装不修改系统 PATH
checksum 不匹配时拒绝
下载失败时给可恢复错误
```

## 小阶段 52.7：ProjectDeploymentService

### 目标

完成项目部署的主执行链路。

### 必须完成

```text
按计划执行 clone、detect_stack、runtime.ensure、install_deps、build、test、run、health_check
每步使用 ToolRuntime/ExecutionBoundary，不直接 subprocess
install/build/run 日志落 artifact
失败时停止后续步骤，生成 failure_reason 和 retry_plan
成功时写 endpoint_url、health_status、managed_process_id
```

### 验收

```text
本地 fixture 项目可从聊天创建任务并跑到可访问端口
install_deps 失败不会进入 completed
build 失败给日志摘要和下一步
health check 未通过不能声称成功
```

## 小阶段 52.8：ManagedProcess 与 PortLease

### 目标

把“项目跑起来”从同步命令升级为可管理进程。

### 必须完成

## 验收标准

本阶段完成时必须满足：

```text
聊天可识别并创建部署 GitHub 项目的受控任务
公开 fixture 项目可被 clone、install、build、run、health check，并返回 endpoint_url
无容器时可以通过 local_workspace 降级完成简单项目部署，且报告明确 degraded isolation
缺 npm/Node 时优先走 portable toolcache，不默认改系统环境
安装桌面软件走 host install plan，未审批不执行
host install 审批绑定 command/cwd/env/source/hash/impact，篡改即拒绝
未知来源安装包、驱动、钱包、浏览器扩展、绕过 UAC 请求默认拒绝或 manual_only

所有部署/安装步骤有 trace、audit、artifact、replay 证据
执行失败、等待确认、用户拒绝时不得声称已完成
secret/token/cookie/password/private key、本地敏感路径不进入模型上下文、trace、audit、report

```

## 与其他阶段关系

```text
第二十一阶段提供工具和执行边界策略，本阶段扩展项目部署/安装 action category
第二十七阶段提供终端沙箱和本地执行隔离，本阶段使用其 backend/degraded evidence
```

## 最终验收定义

第五十二阶段完成时，应能回答：

```text
用户能否一句话让系统部署一个公开 GitHub 项目
系统是否知道该在容器、WSL、本机工作区、toolcache 还是 host executor 中执行
没有容器时是否仍能受控运行，而不是直接污染用户电脑
缺 npm/Node 时是否能优先准备 portable runtime
安装桌面软件时是否先展示来源、命令、影响和风险，并等待确认
项目跑起来后是否有访问地址、日志、进程、端口和停止能力
失败时是否有明确失败步骤、日志摘要和可恢复建议
所有高风险动作是否经过 Safety、Approval、Trace 和审计
系统是否从产品上区分“项目内执行”和“改我的电脑”
```
