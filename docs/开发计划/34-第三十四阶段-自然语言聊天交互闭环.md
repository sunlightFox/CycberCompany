# 第三十四阶段 - 自然语言聊天交互闭环

## 阶段背景

第三十三阶段聚焦聊天主链路重型压力测试后的缺口修复与链路硬化；第三十四阶段进一步收敛产品交互形态：用户大多不是专业开发者，不应被要求理解工具名、审批 ID、trace ID、API 路径或风险编码。

本阶段目标改为采用 Hermes Agent 风格的文字聊天执行模型：用户通过自然语言发起任务，系统优先进入可执行链路；低风险动作直接推进并反馈证据；危险动作在真正执行前阻塞等待用户确认；不可恢复或明显越权动作直接拒绝。系统内部仍保留任务、审批、工具、浏览器、trace、artifact 等完整后端契约，但普通聊天主流程应把这些能力翻译成用户容易理解的动作说明、确认问题和结果反馈。

## 核心原则

- 用户所有交互默认通过文字聊天完成。
- 采用 Hermes Agent 的“执行优先 + 危险点阻塞”路径：不要先问用户“回答/规划/创建任务”，除非目标、对象或权限真的不足。
- 自然语言优先，兼容专业命令；确认范围采用 Hermes Agent 风格的“本次允许 / 本会话允许 / 总是允许 / 拒绝”，但面向普通用户用中文表达。
- 用户偏向“说清楚想要什么，然后得到结果”，系统应主动完成意图识别、上下文绑定、审批转译和结果解释。
- 体验优先少打断：不能什么事都弹确认；只有动作带来外部副作用、不可恢复影响、敏感资源访问或风险不确定时才打断用户。
- 安全边界要硬：少打断不等于放松安全，真正高风险、敏感、不可恢复或越权动作必须确认、升级或直接拒绝。
- 审批确认、拒绝、修改参数、任务继续、任务取消、浏览器操作确认、工具执行确认等用户侧交互，都应支持聊天文本触发。
- 后端真实执行仍必须落到明确 API 契约，例如审批确认最终调用 `/api/approvals/{approval_id}/approve`，但用户体验层不依赖按钮或复制 ID 作为唯一入口；聊天层要像 Hermes Gateway 一样维护会话级 pending approval 队列。
- 聊天层需要能识别明确文字确认，并绑定当前 pending 的审批、任务或工具调用上下文，避免把“好的”“继续”等模糊表达误判为高风险确认。
- 用户拒绝或超时后，任务应得到明确的 BLOCKED/CANCELLED 状态，不能让模型继续重试同一个危险动作。

## 交互降噪要求

- 聊天界面不应把 `approval_id`、`tool_call_id`、`trace_id`、`browser.download`、`/api/approvals/*` 等系统名词作为主要用户话术展示。
- 用户看到的确认请求应是自然语言动作摘要，例如：“我准备下载这个 CSV 文件，来源是本地测试站点，下载后会生成任务工件。请回复：只允许这一次 / 本会话同类下载都允许 / 拒绝 / 修改下载地址为 ...。”
- 系统 ID、工具名、trace、审批 ID 可作为折叠的“技术详情/审计信息”保留，供开发、审计和问题排查使用；普通聊天主流程默认不要求用户阅读或复制这些 ID。
- 如果用户直接输入 `approval_id`、`/approve`、`/deny` 或其他专业命令，系统可以兼容识别；但产品主路径应支持“只允许这一次”“本会话都允许这种操作”“以后同类低风险操作都允许”“拒绝这次删除”“把地址改成 ... 后继续”这类自然语言确认。
- 风险等级应翻译成用户能理解的影响说明，不直接把 `R3`、`R5` 作为主要解释。例如“不说 R3”，而说“这会下载文件并保存为任务工件，所以需要你确认”。

## 典型交互目标

### 下载文件

不推荐：

```text
动作：browser.download
目标：http://127.0.0.1:54069/download/report.csv
风险等级：R3
approval_id: apr_xxx
[确认执行] [拒绝] [修改参数]
```

推荐：

```text
我准备从本地测试站点下载 report.csv。
下载后会保存为当前任务的工件，方便后续回放和审计。

请回复：
- 只允许这一次
- 本会话内同类下载都允许
- 拒绝下载
- 修改下载地址为：...
```

### 删除文件

推荐：

```text
我准备删除这个任务工件文件。删除后可能无法从任务回放里再次打开它。

请回复：
- 只允许这一次删除
- 不删除
- 先给我看文件信息
```

### 浏览器登录

推荐：

```text
我可以打开这个登录页并尝试使用你提供的测试账号登录。
这会在浏览器任务里留下页面快照或截图作为证据。

请回复：
- 只允许这一次登录
- 取消
- 修改账号或地址
```

### 任务继续或取消

推荐：

```text
当前任务停在“等待确认下载文件”这一步。

你可以回复：
- 继续执行
- 取消任务
- 修改下载地址为：...
```

## 后端能力建设

### 1. 聊天文字审批识别

- 识别“确认下载”“确认删除”“拒绝这次操作”“取消任务”“修改地址为 ...”等自然语言指令。
- 识别 Hermes Agent 风格确认范围：“只允许这一次 / 本次允许 / 本会话允许 / 总是允许 / 拒绝 / 取消 / 修改参数”。
- 将自然语言确认绑定到当前会话、当前任务、当前 pending approval。
- 只有在审批上下文唯一且动作匹配时，才调用审批 API。
- 如果存在多个待确认动作，必须追问用户选择，不得自行猜测。

### 2. 审批上下文绑定

- 为聊天会话维护 pending action 队列。
- 每个 pending action 记录用户可读摘要、内部 `approval_id`、任务 ID、工具调用 ID、风险原因、可选动作。
- 聊天回复只展示用户可读摘要，内部 ID 存入 trace 和技术详情。
- 队列按 Hermes Gateway 方式处理：同一会话内 FIFO，用户可确认最早一条，也可明确指定确认某个动作；拒绝、超时、发送失败都要 fail-closed。

### 3. 参数修改解析

- 支持用户通过自然语言修改待审批动作参数。
- 例如“把下载地址改成 ... 后继续”应落到 `/api/approvals/{approval_id}/edit`。
- 修改后必须重新进行 Safety、DLP、Capability、Execution Boundary 检查。

### 4. 执行结果自然语言反馈

- 工具执行完成后，聊天回复应说明真实结果，而不是只说“任务已处理完成”。
- 浏览器动作要说明证据类型，例如“已生成页面快照”“已保存截图”“下载等待审批”“文件已保存为任务工件”。
- 如果只是等待审批，必须明确“尚未执行”。

### 5. 专业信息折叠

- API 响应中保留 `approval_id`、`tool_call_id`、`trace_id`、artifact ID。
- 聊天主回复默认不展示这些 ID。
- 提供可选的技术详情字段，供未来 UI 折叠展示或开发调试使用。

## 风险确认门槛参考

本阶段改为以 Hermes Agent 的公开源码思路作为主设计，不再采用 OpenClaw 式 allowlist/askFallback 作为主交互模型。OpenClaw 只作为补充安全参考；最终产品体验和后端状态机按 Hermes Agent 的“执行优先、危险命令识别、会话级审批、hardline 阻断、超时拒绝”收敛。

### Hermes Agent 采用点

- Hermes 的审批更偏“危险命令模式识别 + 执行后端边界”。
- 默认 `manual` 模式：危险命令触发用户确认。
- `smart` 模式：低风险自动通过，明显危险自动拒绝，不确定时再问用户。
- `off` / `YOLO` 模式：不确认，不应作为默认用户路径。
- 默认超时拒绝：用户长时间不确认时 deny。
- Gateway 会话按 session key 维护 pending approval 队列；用户确认后释放被阻塞的执行线程。
- 拒绝、超时、发送审批失败时返回明确 BLOCKED，并要求不要重试同一危险动作。
- 容器后端通常降低确认需求，因为 Docker、Singularity、Modal、Daytona、Vercel Sandbox 等环境提供额外隔离。
- hardline block 直接拒绝，不给普通确认，例如 `rm -rf /`、格式化磁盘、写裸设备、fork bomb、shutdown/reboot、kill all。
- 危险模式包括递归删除、写系统目录、放宽权限、改 owner、格式化、数据库 destructive SQL、停止系统服务、强制 kill、shell inline 执行、`curl | sh`、写 `.ssh` / `.env`、`find -delete`、`xargs rm` 等。
- 源码中还将 `git reset --hard`、`git push --force`、`git clean -f`、强删分支、更新/重启自身 gateway、`chmod +x` 后立即执行脚本等纳入危险/审批逻辑。
- Tirith 这类安全扫描会对 homograph URL、terminal injection、`curl | bash`、SSRF、私网/metadata URL 等做内容级防护；其中部分应确认，部分应直接阻断。

对我们的启发：

- 风险门槛要看“动作影响 + 可恢复性 + 资源范围 + 外部副作用 + 执行环境隔离”，不能只看用户是否明确说了要做。
- 浏览器访问私网、metadata、file URL、本地敏感路径、凭据外传，应优先直接阻断或进入高等级安全处理，而不是普通确认。
- 模糊确认例如“好的”“继续”不能触发高风险动作；确认必须包含动作对象或唯一 pending context。
- `always` 永久允许只用于稳定、低风险、可解释的动作类别；Tirith/内容安全告警、账号、钱包、密钥、删除、覆盖、发布和外部提交不得永久允许。

### Hermes 化门槛

| 动作类型 | 默认处理 | 用户可见话术 |
| --- | --- | --- |
| 纯回答、总结、解释 | 直接回答 | 不提审批和工具。 |
| 只读检索、只读页面快照 | 可执行并给证据；敏感目标除外 | “我会查看这个页面并总结结果。” |
| 低风险本地写入任务工件 | 直接执行或轻提示，取决于来源和大小 | “我会把结果保存到当前任务，方便回看。” |
| 下载文件、截图保存、浏览器登录、发送外部请求 | 进入 once/session/deny 确认 | “这会保存文件/留下截图/访问网页。你可以只允许这一次、本会话允许或拒绝。” |
| 删除、覆盖、移动、批量修改、终端执行 | 进入 once/session/deny，必要时禁用 always | “这可能影响已有文件，请明确回复只允许这一次或拒绝。” |
| 凭据、钱包、账号、私钥、cookie、token、系统目录、私网 metadata | 直接拒绝或进入高安全审批 | “这个请求涉及敏感资源，我不能这样执行。” |
| 不可恢复破坏性动作 | 直接拒绝 | 不提供普通确认入口。 |

### 体验友好的确认策略

- 默认不确认：普通问答、知识总结、格式转换、只生成方案、解释概念、只读查询、读取公开网页、生成草稿、保存普通任务报告。
- 轻提示不打断：低风险动作可以在执行后说明“我已保存到任务工件”，不要求用户每次确认。
- 单次允许：下载文件、保存截图、浏览器登录、发送外部请求、创建可见任务、写入长期记忆等有副作用动作，默认只开放“本次允许/拒绝/修改参数”。
- 本会话允许：同一任务内同类低到中风险动作，可以允许“本会话内同类操作都允许”，会话结束后自动失效。
- 总是允许：只开放给稳定、低风险、可解释、可撤销的动作模板；涉及删除、覆盖、账号、钱包、密钥、发布、外部提交、安全告警时不显示该选项。
- 二次确认：删除、覆盖、批量修改、终端命令、发布、发送消息、涉及账号/钱包/密钥的动作，需要更明确的二次确认或更强审批。
- 直接拒绝：不可恢复破坏、绕过权限、读取他人数据、泄露密钥、访问 metadata、危险系统路径、明显恶意行为，不把责任转嫁给用户确认。
- 记住用户偏好：允许用户设置“低风险动作少问我”“每次下载都问我”“高风险永远确认”等安全偏好，但偏好不能覆盖硬安全边界。
- 合并确认：同一任务中的多个低到中风险小步骤，尽量合并成一个自然语言确认，不逐步打断。
- 上下文信任：同一任务内已确认的同类动作可在限定范围内复用确认，但目标、参数、风险或外部影响变化时必须重新确认。
- 可撤销优先：能撤销的低风险动作可以少打断；不可撤销或难撤销动作必须打断。

### 自然语言确认规则

- 允许：“只允许这一次”“本次允许下载这个 CSV”“本会话都允许同类下载”“总是允许这种只读查询”“拒绝这次删除”“把地址改成 ... 后继续”“取消当前任务”。
- 不足以确认：“好的”“嗯”“继续”“可以”“走吧”，除非当前只有低风险动作且上下文明确。
- 必须追问：存在多个 pending action、动作对象不唯一、参数被修改、风险等级升级、目标从普通 URL 变成私网/本地/敏感路径。
- 必须重新校验：任何 edit 后的参数都要重新经过 Safety、DLP、Capability Graph、Execution Boundary。

## Hermes Agent 化最终设计

本版按 Hermes Agent 的源码模型重新设计，把“文字聊天入口”和“执行审批系统”合并成一个会话级状态机：用户发出自然语言任务后，Agent 先推进可执行步骤；一旦遇到危险动作，就在当前会话阻塞等待用户用文字选择“本次允许 / 本会话允许 / 总是允许 / 拒绝”；拒绝、超时、发送审批失败都立即 fail-closed。

### 参考源码基线

- Hermes Agent：本地浅克隆基线 `fc7f55f`，重点阅读 `tools/approval.py`、`tools/url_safety.py`、`tools/browser_camofox.py`、`agent/prompt_builder.py`、`agent/shell_hooks.py`。
- OpenClaw：仅作为补充参考，保留对执行绑定和通道审批的理解，但不再作为本阶段主设计。
- 采用方式：后端状态机、审批范围、hardline 阻断、危险模式、会话队列、超时拒绝按 Hermes Agent；用户可见话术翻译为中文自然语言。

### 二版核心判断

- Hermes 成熟点：先做 hardline block，再做危险模式审批；`rm -rf /`、格式化磁盘、写裸设备、fork bomb、shutdown/reboot、kill all 等不可恢复动作不进入普通确认；危险但可控的动作才进入 once/session/always/deny 选择。
- Hermes 成熟点：Gateway 会话维护 pending approval 队列，Agent 执行线程等待用户响应；Agent 不应看到一个可继续推理的 `approval_required` 并自行重试。
- Hermes 成熟点：`smart` 模式可以由辅助模型减少误报，但辅助模型不能覆盖 hardline 阻断。
- 我们的产品判断：用户明确表达意图是进入执行链路的信号；低风险直接推进，高风险自然语言确认，极高风险直接拒绝。

### 后端策略模型

新增或收敛为统一的 `ActionRiskDecision` 决策对象，供聊天、任务、工具、浏览器、Skill、MCP 共用。

| 字段 | 说明 |
| --- | --- |
| `decision` | `allow`、`inform`、`approval_required`、`deny`、`hard_block`。 |
| `risk_level` | 内部风险等级，可保留 R0-R5 或等价枚举，但普通聊天不直接展示。 |
| `reason_codes` | 稳定机器码，例如 `browser_download`、`terminal_inline_eval`、`destructive_file_write`、`private_network_target`。 |
| `user_summary` | 面向用户的自然语言动作摘要。 |
| `technical_detail` | 工具名、参数摘要、trace、approval、artifact 等审计信息，默认折叠。 |
| `binding` | 审批绑定快照，包含动作类型、工具名、规范化参数、会话、任务、成员、资产、执行环境和可变文件 hash。 |
| `allowed_confirm_scopes` | Hermes 风格确认范围：`once`、`session`、`always`、`deny`；不同风险只开放不同范围。 |
| `expires_at` | 待确认动作超时时间；超时默认拒绝或取消。 |

### Hermes 决策顺序

1. Normalize：对命令、URL、路径、参数做规范化，去除 ANSI、控制字符、混淆字符，URL 做解析和目标解析。
2. Capability：先过 Capability Graph、Asset Broker、Skill/MCP 权限，不允许的资源直接拒绝。
3. Hardline：匹配不可恢复或明显越权动作，直接 `hard_block`，不提供普通“确认执行”入口。
4. Dangerous Patterns：匹配 Hermes 风格危险模式，生成一条合并审批请求，而不是多个碎片确认。
5. Environment：Docker、Singularity、Modal、Daytona、Vercel Sandbox 等隔离环境可以降低确认需求；本机、SSH、真实浏览器持久 profile 更保守。
6. Smart：可选辅助模型判断明显误报、明显危险或升级给用户；规则和 hardline 永远优先。
7. Pending Queue：需要审批时进入会话级 FIFO 队列，阻塞相关执行线程，等待用户选择 once/session/always/deny。
8. Binding：审批发起前生成动作快照；用户修改参数后废弃旧绑定并重新跑 1-7。
9. Execute：执行前再次比对 binding；不一致则拒绝并重新确认。
10. Trace：模型决策、风险判断、审批请求、审批结果、工具执行、工件生成全部落 trace。

### 体验模式

| 模式 | Hermes 对齐 | 我们的默认策略 |
| --- | --- | --- |
| `manual` | Hermes 默认模式 | 危险动作触发用户确认，普通低风险动作直接执行。 |
| `smart` | Hermes 辅助模型审批 | 规则优先，辅助模型只用于降低误报或升级风险，不能覆盖 hard_block。 |
| `off` | Hermes 关闭审批 | 仅开发调试可用，生产默认禁用，且不能绕过 hardline。 |
| `yolo_session` | Hermes Gateway `/yolo` 思路 | 仅允许受信会话临时开启，不能跨会话持久化，不能绕过 hardline。 |
| `container_trusted` | Hermes 隔离后端 | 容器/沙箱内动作少确认，本机/SSH/真实持久浏览器仍保守。 |
| `cron_deny` | Hermes cron 默认 deny | 无用户在线时危险动作默认拒绝，除非显式配置 cron approve。 |

### 确认范围

- 单次确认：下载、截图保存、浏览器登录、一次性外部提交、一次性文件写入。
- 本会话确认：同一任务内同一来源、同一目录、同一动作模板、参数范围稳定的小批量操作。
- 记住偏好：只允许稳定低风险模板，例如“公开网页只读摘要不用每次问我”；不允许用于私密资源、终端执行、删除、覆盖、发布、付款、账号、钱包、密钥。
- 永不记住：Tirith/内容级安全告警、URL 混淆、SSRF、私网 metadata、inline eval、`curl | sh`、危险 git、批量删除、系统服务变更。
- 参数变化：URL、路径、文件 hash、账号、目标站点、保存位置、命令 argv、cwd、env、资产 handle 任一变化，都使旧审批失效。

### 聊天 Pending Action 队列

- 每个聊天会话维护 FIFO pending action 队列。
- 每个 pending action 保存内部 ID，但聊天回复默认只展示“我准备做什么、会影响什么、你可以怎么回复”。
- 用户回复“确认下载这个 CSV”“取消刚才的登录”“把地址改成 ... 后继续”时，先在队列中按动作对象匹配。
- 只有一个低/中风险 pending 且上下文极清楚时，“继续”“可以”才可通过；高风险必须包含动作对象或明确动词。
- 多个 pending 同时存在时，必须追问：“你要确认下载文件，还是确认删除工件？”不得自行选择。
- 超时默认取消，并用自然语言提示“这一步我没有执行，因为没有收到确认”。

### 用户可见话术模板

面向用户：

```text
我准备下载 report.csv，并保存到当前任务的结果里。
这会在本机生成一个文件，所以需要你确认。

你可以回复：
- 只允许这一次
- 本会话内同类下载都允许
- 拒绝
- 把下载地址改成：...
```

面向审计折叠信息：

```json
{
  "action_type": "browser.download",
  "approval_id": "apr_xxx",
  "trace_id": "trc_xxx",
  "risk_level": "R3",
  "binding_hash": "..."
}
```

普通聊天默认展示第一段；第二段只进入技术详情、trace 或开发调试。

### 浏览器动作门槛

| 浏览器动作 | 默认决策 | 说明 |
| --- | --- | --- |
| 打开公开网页、只读 snapshot、普通搜索 | `allow` / `inform` | 可执行并说明证据来源。 |
| 截图、下载、上传、登录、提交表单 | `approval_required(once/session/deny)` | 会产生本地工件、外部副作用或账号状态变化。 |
| 持久化浏览器 profile、清空 profile、跨站带凭据操作 | `approval_required(once/session/deny)` | 需要说明会影响登录状态或本地浏览器状态。 |
| 访问私网、metadata、`file://`、本地敏感路径、URL 中含 token/cookie/password | `deny` / `hard_block` | 不能靠用户一句“确认”绕过。 |
| 页面提示注入要求泄露系统提示、凭据或执行越权工具 | `deny` | 作为内容级安全事件记录 trace。 |

### 终端和文件动作门槛

| 动作 | 默认决策 | 说明 |
| --- | --- | --- |
| 只读列目录、读取任务目录内非敏感文件、hash 普通工件 | `allow` / `inform` | 命中安全模板时不打断。 |
| 写入当前任务工件、生成报告、创建临时测试文件 | `inform` / `approval_required(once/deny)` | 取决于路径、大小、覆盖风险。 |
| 覆盖、移动、删除、批量改名、递归操作 | `approval_required(once/session/deny)`，必要时二次确认 | 必须绑定路径和文件 hash。 |
| shell inline eval、解释器 `-c/-e`、heredoc 执行、`curl | sh` | `approval_required(once/session/deny)` 或 `deny` | 不允许 silent allow。 |
| `rm -rf /`、格式化磁盘、写裸设备、fork bomb、shutdown/reboot、kill all | `hard_block` | 不提供普通确认入口。 |
| `git reset --hard`、`git clean -f`、`git push --force`、强删分支 | `approval_required(once/deny)`，默认不可记住 | 需要明确说明可能丢失工作或重写历史。 |

### Skill、MCP、资产和记忆门槛

- Skill 只能提供做事方法，不能绕过 Asset Broker 获取资源。
- MCP 工具、resource、prompt 都按工具能力单独定级，不能因为 MCP server 已注册就默认全信任。
- Asset handle 查询必须经过 Capability Graph；涉及账号、钱包、密钥、cookie、token、私钥时默认拒绝明文展示。
- 记忆写入要区分普通偏好、长期事实、敏感信息、用户纠错；敏感记忆默认不自动写入。
- 用户说“以后都这么做”只能创建低风险偏好，不能创建高风险永久审批。

### API 契约建议

本阶段不一定一次性实现全部接口，但后端契约应朝以下形态收敛：

- `POST /api/chat/turn`：可返回 `pending_actions` 的用户可读摘要和技术详情引用。
- `POST /api/chat/pending-actions/{id}/resolve`：内部接口，聊天文本解析后调用，支持 `once`、`session`、`always`、`deny`、`edit`。
- `POST /api/safety/evaluate-action`：统一评估工具、浏览器、Skill、MCP、文件、终端动作。
- `POST /api/approvals/{approval_id}/bind`：记录规范化 action binding。
- `POST /api/approvals/{approval_id}/consume`：单次审批执行前原子消费。
- `GET /api/tasks/{task_id}/pending-actions`：任务恢复时重建聊天侧待确认状态。

如果已有 approval API 名称不同，可以复用现有路径；关键是补齐统一决策对象、自然语言解析、binding 校验和 trace。

### 二版验收补充

- 明确意图应进入执行链路：用户说“帮我下载并登录”时，不要泛泛询问回答/规划/创建任务；遇到下载、登录、提交等副作用点再阻塞确认。
- 低风险少打断：公开网页搜索、只读摘要、普通知识总结、隔离环境中的无害命令不能频繁弹确认。
- 高风险强绑定：确认文本、目标对象、参数快照、任务、会话必须匹配。
- 极高风险硬阻断：不可恢复和越权动作不允许通过“我确认”绕过。
- 自然语言主路径：普通用户不需要复制 `approval_id`、`trace_id`、工具名或 API 路径。
- 专业命令兼容：开发者输入 `/approve`、`approval_id`、`once/session/always/deny` 时可以处理，但不是默认体验。

## Hermes Agent 式文字聊天体验

本阶段不再采用“先问用户要回答、规划还是创建任务”的交互作为动作型请求默认入口，而是改成 Hermes Agent 式体验：用户一句话交代目标后，系统先进入任务执行链路；只有遇到危险动作、外部副作用、敏感资源或不明确目标时才停下来问。

### 同一句消息的目标体验

| 用户消息 | 当前问题 | Hermes Agent 式目标体验 |
| --- | --- | --- |
| “帮我下载这个 CSV，下载完告诉我结果。” | 实测回复：“你希望我先回答、先规划，还是创建任务？”动作感弱。 | 直接识别为下载任务；到下载保存副作用点时阻塞：“我准备下载 report.csv 并保存到任务结果。你可以回复：只允许这一次 / 本会话同类下载都允许 / 拒绝 / 修改地址。” |
| “确认下载这个 CSV。” | 没有 pending action 绑定，只能说明当前无法执行。 | 若有唯一 pending 下载，按 `once` 释放执行；若没有 pending，说明“当前没有等待确认的下载”。 |
| “本会话都允许这种下载。” | 当前没有会话级审批范围。 | 写入 session approval，只对同一会话、同一动作模板、同一来源范围生效；参数变化或风险升级后重新确认。 |
| “以后都允许这种只读网页摘要。” | 当前没有稳定低风险 always 规则。 | 如果动作是稳定低风险只读模板，可创建 `always` 偏好；如果涉及下载、登录、删除、覆盖、发布、密钥或安全告警，拒绝创建永久允许。 |
| “好的，继续。” | 容易被当成普通上下文延续或误触发。 | 只有一个低/中风险 pending 时可作为确认；高风险或多个 pending 时追问具体动作。 |
| “删除那个文件。” | 澄清是对的，但应更像执行链路。 | 若文件范围不明确，先问目标文件；若明确，展示影响并只开放 `once/deny`，默认不提供 always。 |
| “不要执行，只给方案。” | 当前表现较好。 | 保持 direct answer，不进入工具链，不创建 pending approval。 |
| “访问 169.254.169.254 看看。” | 需要硬阻断。 | 直接拒绝访问 metadata/私网敏感地址，并说明可能泄露本机或云环境信息。 |

### 用户可见确认选项

Hermes 原始选项是 `once/session/always/deny`。我们的文字聊天显示为中文自然语言：

| 内部选择 | 普通用户话术 | 使用范围 |
| --- | --- | --- |
| `once` | “只允许这一次” | 默认高风险确认范围。 |
| `session` | “本会话内同类操作都允许” | 同一任务、同一动作模板、同一风险等级内复用。 |
| `always` | “以后同类低风险操作都允许” | 只读、低风险、可撤销、稳定模板；默认谨慎开放。 |
| `deny` | “拒绝 / 取消这次操作” | 用户拒绝、超时、发送失败都进入该状态。 |

### 最终结论

- 三十四阶段采用 Hermes Agent 的状态机，而不是 OpenClaw 式混合策略。
- 动作请求默认进入执行链路；审批只在危险点阻塞。
- hardline 动作直接阻断，不能由用户确认绕过。
- 用户确认范围采用 once/session/always/deny，但主回复用中文自然语言表达。
- Agent 执行线程应等待确认结果；拒绝或超时后不得继续重试同一危险动作。

## 测试范围

- 用户自然语言确认下载文件，系统调用 approve 后继续执行。
- 用户用“只允许这一次 / 本会话允许 / 总是允许 / 拒绝”选择确认范围，系统按 Hermes Agent 风格写入 once/session/always/deny。
- 用户自然语言拒绝删除文件，系统调用 deny，任务保持安全状态。
- 用户拒绝、超时或审批投递失败后，任务返回 BLOCKED/CANCELLED，不得自动重试同一危险动作。
- 用户自然语言修改浏览器目标地址，系统调用 edit 并重新校验。
- 用户只回复“好的”“继续”等模糊表达时，高风险动作不得直接执行。
- 多个审批同时存在时，系统必须追问具体确认哪一个。
- 聊天回复不得要求普通用户复制 `approval_id` 才能继续。
- 聊天回复不得把 `browser.download`、`tool_call_id`、`trace_id` 作为主要表达。
- 直接输入专业命令和 ID 时仍能兼容处理。

## 已执行对标测试结论

- 测试报告：`docs/测试/聊天主链路/2026-04-30/10-自然聊天对标测试报告.md`
- 结论文档：`docs/测试/聊天主链路/2026-04-30/11-自然聊天待优化结论.md`
- 执行脚本：`docs/测试/聊天主链路/2026-04-30/run_chat_natural_interaction_benchmark.py`
- 结果：PASS 1 / FAIL 11 / BLOCKED 0。
- 预检通过：`mem_xiaoyao` 默认大脑健康，真实模型最小聊天完成。
- 主要阻塞：连续自然交互测试中仍出现 `database is locked`，导致多条用例无法完成真实聊天回复；这会直接破坏“文字聊天连续完成任务”的产品体验。
- 已观察到的真实交互缺口：用户说“帮我下载 ... 下载完告诉我结果”时，回复变成“你希望我先回答、先规划，还是创建任务？”，没有自然说明将下载什么、为什么需要确认、用户可以怎么回复，也没有把动作转化为普通用户能理解的确认问题。
- 已观察到的较好表现：用户后续说“确认下载这个 CSV”时，系统没有伪称下载完成，并说明当前无法直接访问本机地址和不会假装已完成。
- 改为 Hermes Agent 方案后，本阶段应把“聊天文字入口 + 执行优先 + 危险点阻塞 + 证据反馈”作为一个连续体验做，而不是让用户理解任务、工具、审批和 trace 的内部机制。

## 验收标准

- 用户可只靠文字聊天完成审批确认、拒绝、修改参数和任务继续。
- 支持 Hermes Agent 风格确认范围：once、session、always、deny，并对 `always` 做高风险禁用。
- 高风险动作必须有明确自然语言确认和后端审批记录。
- hardline 动作直接拒绝，用户确认不能绕过。
- 模糊确认不会触发高风险动作。
- 普通聊天回复自然、可理解、少系统术语。
- 后端 trace、approval、tool_call、artifact 证据链完整。
- 第三十三阶段中与“审批体验、浏览器执行、任务等待确认、结果表述不清”相关的问题，在本阶段有对应测试覆盖。

## 与第三十三阶段关系

- 第三十三阶段修复重型压力测试中暴露的链路稳定性、Skill/MCP、浏览器证据、数据库锁和安全检测问题。
- 第三十四阶段在修复基础上优化交互体验，重点让聊天成为统一文字入口。
- 两个阶段可以并行设计，但实现上应先保证第三十三阶段的后端链路稳定，再让第三十四阶段的自然语言交互接入这些能力。
