# 第十二阶段：聊天体验与复杂对话能力深化

## 摘要

第十二阶段专门补齐聊天主链路的“用户体感层”。前面阶段已经让聊天链路、模型路由、记忆、任务、工具、安全和扩展能力逐步可运行；本阶段不新增产品版本，不新增前端实现，而是把后端聊天体验从“能回复”打磨到“像最终产品一样会聊、会延续、会判断、会诚实说明边界”。

本阶段仍然只做后端、schema、service、API、测试与 eval，不新增 UI、Tauri、前端页面、组件或样式。

## 阶段定位

聊天是整个个人智能体操作系统的唯一主入口。用户不会先理解 Task Engine、Skill、MCP、Capability Graph 或 Asset Broker，用户只会判断：

```text
它日常闲聊是否自然
复杂问题是否能连续讨论
它是否知道什么时候该追问
它是否能诚实区分“我能回答”和“我需要工具才能做”
它是否能记住当前对话脉络
它是否能把任务、记忆、人格和安全边界自然融合到回复里
```

因此第十二阶段不解决底层工具能力本身，而解决聊天主链路上的体验偏差：

| 能力 | 当前判断 | 本阶段目标 |
|---|---:|---|
| 普通闲聊 | 约 60% | 回到最终态自然对话质量 |
| 复杂对话 | 约 50% | 支持多轮主题延续、结构化思考和澄清 |
| 聊天边界说明 | 偏工程化 | 准确说明能做、不能做、需要确认什么 |
| direct 回复 | 可用但偏直接透传 | 统一经过 Response Composer 和 persona/heart 信号 |
| 历史上下文 | 可接入但不够策略化 | 按主题、时间、相关性和 token budget 选择 |
| 系统提示 | 存在阶段性陈旧语义 | 更新为最终态能力边界，不再写“第二阶段不能...” |

## 阶段原则

1. 只优化聊天主链路后端，不新增前端实现。
2. 不伪装工具执行，不说“已完成”未真实执行的动作。
3. 普通聊天必须轻，复杂聊天必须稳，高风险聊天必须清楚。
4. 所有用户可见回复仍必须经过 Response Composer。
5. 聊天页最终仍只显示当前聊天对象的人名、头像、状态和消息，不显示组织和壳。
6. 核心层仍只使用 Organization / Member / Department / Role / Shell / Asset / Skill / Task。
7. 不把公司壳文案、Boss/Company/Employee 语义写进核心聊天服务。
8. 每次模型调用、上下文构建、回复编排、记忆候选都必须可 trace。

## 阶段范围

### 本阶段必须完成

```text
聊天系统提示与能力边界更新
闲聊 direct path 质量规则
复杂对话状态与主题延续策略
澄清问题决策
上下文预算与历史选择策略优化
模型路由在轻聊/深聊/长文/高隐私场景下的策略
工具/任务需求的诚实回复模板
失败、取消、重试后的对话连续性
聊天体验 eval 数据集
聊天 trace 与 response evidence 校验
```

### 本阶段明确不做

```text
不新增前端聊天页面
不新增 Tauri 交互
不新增真实工具能力
不新增新的壳类型
不改动已有安全边界
不让聊天服务直接访问 secret、文件、浏览器、终端或 MCP
不把复杂任务直接塞进 direct 回复伪执行
```

## 小阶段 12.1：聊天基线复核与陈旧提示清理

### 目标

清理聊天主链路中的阶段性提示和旧能力边界，让后端提示、runtime contract、实际能力保持一致。

### 修复内容

```text
检查 ChatService / Brain / ResponseComposer 中的系统提示
移除“第二阶段不能...”这类历史阶段文案
改成最终态能力边界描述
按 runtime contract 动态表达 implemented / degraded / disabled 能力
把工具、MCP、Skill、记忆、人格等能力边界由硬编码改为服务状态摘要
确保模型看见的是当前真实可用能力，不是过期开发阶段说明
```

### 最终态提示要求

系统提示必须表达：

```text
当前成员是谁
当前回复风格和 persona 摘要
当前安全边界
当前可用能力摘要
第三方内容不可信
需要工具时必须通过任务/工具链路
未真实执行时不得声称已经执行
高风险动作必须等待确认
```

系统提示不得包含：

```text
开发阶段编号
内部实现文件路径
公司壳硬编码称呼
secret、token、path 明文
未启用能力的完成式承诺
```

### 验收

```text
聊天系统提示不再出现历史阶段限制文案
能力边界来自 runtime contract 或 service capability summary
未启用工具时不会声称可直接操作文件/浏览器/终端
启用工具时也不会绕过 Task/Tool/Safety 链路
新增 regression 测试覆盖陈旧提示字符串
```

## 小阶段 12.2：闲聊 direct path 质量提升

### 目标

让普通聊天、日常问答、轻度陪伴、轻写作在不进入复杂任务的情况下自然可用。

### 回复策略

| 场景 | 策略 |
|---|---|
| 问候/轻聊天 | 简洁、自然、可继续，不强行总结 |
| 简单事实问答 | 直接回答，不展开无关系统说明 |
| 用户表达情绪 | Heart 给出情绪和紧急度，回复温暖但不过度拟人 |
| 轻写作 | 给出可用稿件，必要时标注可继续改 |
| 轻技术解释 | 分层解释，避免空泛 |
| 随口问题 | 不创建任务，不触发工具 |

### 后端实现要求

```text
IntentClassifier 标记 casual_chat / simple_qa / light_writing
ModeSelector 选择 direct
ModelRouter 优先 local_main 或配置的轻量模型
ContextGateway 只注入必要近期上下文和 persona/heart summary
ResponseComposer 按场景控制长度、结构和语气
Trace 写 brain.intent、brain.mode_select、model.route、response.compose
```

### 验收

```text
普通问候不会输出系统架构说明
短问题不会生成过长大纲
用户求安慰时不会触发任务或工具
轻写作能直接给成品
回复仍可被 ResponseComposer 统一拦截和脱敏
```

## 小阶段 12.3：复杂对话状态建模

### 目标

支持多轮复杂讨论，不让系统每轮都像第一次见到问题。

### Conversation Working State

为当前会话维护可压缩状态：

```text
active_topic
user_goal
known_constraints
decisions_made
open_questions
candidate_actions
referenced_artifacts
last_response_summary
pending_confirmation
```

该状态可以落在会话摘要、turn metadata 或专用 conversation_state 表中，具体实现遵循现有 schema 走向。

### 复杂对话类型

| 类型 | 例子 | 后端策略 |
|---|---|---|
| 深度方案讨论 | “继续优化这个产品规划” | 读取 active_topic 和已定结论 |
| 多轮写作 | “上一版太硬，改得自然点” | 读取上一轮草稿摘要 |
| 复杂决策 | “A/B 两个路线怎么选” | 保留评估标准和权衡 |
| 技术排查 | “刚才那个错误还有别的可能吗” | 保留假设和排除项 |
| 长期项目 | “按之前那个方向继续” | 结合会话摘要和相关记忆 |

### 验收

```text
用户说“继续”时能正确引用上一轮主题
用户修改约束时状态能更新
复杂讨论能保留 open_questions
不会把旧结论当作永远有效的新事实
conversation_state 变化有 trace 或 message metadata
```

## 小阶段 12.4：澄清与追问决策

### 目标

让系统在信息不足时会问好问题，而不是硬猜；但也不能小事过度追问。

### ClarificationDecision

建议输出：

```json
{
  "needs_clarification": true,
  "reason": "missing_target_scope",
  "blocking_level": "blocks_execution",
  "questions": [
    "你希望我只整理当前文档，还是整理整个 docs 目录？"
  ],
  "can_answer_partially": true
}
```

### 触发条件

```text
目标对象不明确
成功标准不明确
风险等级不明确
需要资产但未给范围
用户要求执行但权限或资源不清楚
多种解释会导致明显不同结果
```

### 不应追问的情况

```text
普通闲聊
简单问答
用户明确要求先给建议
可以安全给出草案或假设说明
追问只会增加摩擦而不会降低风险
```

### 验收

```text
高风险或资源范围不明时先澄清
简单聊天不被澄清流程打断
澄清问题最多聚焦 1-3 个关键点
澄清 decision 写 trace
ResponseComposer 能输出清楚的追问回复
```

## 小阶段 12.5：历史上下文选择与压缩

### 目标

优化 Context Gateway 的聊天上下文选择，让复杂对话有连续性，同时避免上下文膨胀和旧信息污染。

### 选择顺序

```text
当前用户输入
当前 conversation working state
最近 N 轮消息
当前主题相关摘要
高置信 persona/pinned memory
语义相关 memory
近期任务或 artifact 摘要
能力与资源摘要
```

### 压缩规则

```text
优先保留当前输入原文
长历史转摘要
工具结果只保留 redacted summary 和 artifact handle
过期记忆降低权重
被 supersede 的事实默认不进入上下文
第三方内容必须标记 untrusted
```

### 验收

```text
复杂多轮讨论不丢当前主题
长对话不会把低相关旧消息塞满 prompt
敏感信息不进入模型上下文明文
上下文选择结果有 selection_reason
ContextGateway 测试覆盖 token budget
```

## 小阶段 12.6：聊天模型路由细分

### 目标

让模型路由能够区分轻聊、复杂推理、长文生成、高隐私和高风险任务，不再只按粗粒度 intent 路由。

### 路由维度

```text
conversation_depth
complexity_score
privacy_level
needs_strong_reasoning
needs_long_output
needs_low_latency
needs_persona_style
needs_tool_or_task
```

### 路由策略

| 场景 | 默认策略 |
|---|---|
| casual_chat | local_main，低延迟 |
| simple_qa | local_main 或 configured default |
| deep_reasoning | cloud_strong 或本地强模型，受 privacy 限制 |
| long_writing | 长上下文模型，保留输出预算 |
| high_privacy | 禁止云端 fallback |
| tool/task | 先进入 Task/Planner，不由 direct 模型伪执行 |

### 验收

```text
轻聊不会默认调用最重模型
复杂推理有可解释 route reason
高隐私输入不会云端 fallback
长文请求保留足够输出预算
route.selected 不暴露在聊天页主界面
```

## 小阶段 12.7：工具/任务边界的产品化表达

### 目标

用户提出“做事”请求时，系统要能自然地说明下一步，而不是机械拒绝或假装完成。

### 场景策略

| 能力状态 | 回复策略 |
|---|---|
| 工具能力未启用 | 说明需要的能力和配置，不声称执行 |
| 工具能力已启用且低风险 | 创建任务或进入工具链路 |
| 工具能力已启用但高风险 | 先计划，再请求确认 |
| 缺少资产或权限 | 说明缺少的资源句柄或授权 |
| MCP/Skill degraded | 说明当前降级点和可替代路径 |

### 验收

```text
“整理文件夹”不会在未执行时说已经整理
“发帖/提交/购买”必须进入确认或说明无法直接做
工具可用时 direct path 不吞掉任务意图
ResponseComposer 输出计划、条件和下一步动作
```

## 小阶段 12.8：失败、取消、重试的连续体验

### 目标

失败和取消不能让对话断裂。用户应该能继续说“刚才那个重试一下”或“换个方式继续”。

### 实现要求

```text
turn.failed 写失败摘要、error_code、recoverable 标记
turn.cancelled 保留 partial response 和 cancel reason
retry 创建新 turn，不覆盖旧消息
conversation working state 记录失败点和可恢复动作
ResponseComposer 给出下一步选项
Trace 关联 retry_of_turn_id
```

### 验收

```text
模型超时后用户可重试
取消后 partial 回复可保留
重试不覆盖原 trace
失败后继续对话能引用失败点
错误回复不暴露异常栈和内部 prompt
```

## 小阶段 12.9：聊天体验评测与封口

### 目标

建立聊天体验专项 eval，后续每次主链路修改都能检查是否退化。

### Eval 分类

```text
casual_chat_naturalness
simple_qa_directness
complex_dialogue_continuity
clarification_quality
tool_boundary_honesty
privacy_route_behavior
failure_recovery
persona_tone_consistency
response_structure
no_internal_leakage
```

### 指标

| 指标 | 含义 | 目标 |
|---|---|---:|
| CNR | Casual Naturalness Rate | >= 0.85 |
| CDC | Complex Dialogue Continuity | >= 0.80 |
| CQA | Clarification Quality Accuracy | >= 0.85 |
| TBH | Tool Boundary Honesty | 1.00 |
| PIL | Prompt/Internal Leakage | 0 |
| PVB | Privacy Violation Behavior | 0 |

### 验收

```text
新增聊天体验 eval 数据集
eval 能离线运行
失败 case 能输出 trace refs
工具边界诚实性必须 100% 通过
隐私和内部泄露必须 0 容忍
```

## 总体验收标准

第十二阶段完成时必须满足：

```text
聊天系统提示已更新为最终态能力边界
普通闲聊自然、短小、不过度工程化
复杂对话能延续主题、约束和未决问题
信息不足时能做合理澄清
上下文选择有 token budget 和 selection_reason
工具/任务请求不伪执行
失败、取消、重试可连续
聊天体验 eval 可运行
pytest、ruff、mypy 保持通过
```

## 不允许通过验收的情况

出现以下任一情况，本阶段不能通过：

```text
系统提示仍出现旧阶段编号或过期能力限制
未执行工具却声称已执行
高隐私输入进入云端模型
聊天页契约要求被改成显示组织/壳信息
ResponseComposer 被绕过
复杂对话无法引用上一轮主题
澄清问题过度泛化或缺失关键风险点
trace/audit 中出现 secret 明文
```

## 与后续阶段关系

第十二阶段把聊天体验本身打稳；第十三阶段继续把意图识别和模式选择做深；第十四阶段把 Persona / Heart / Response Composer 的体验控制接进来；第十五阶段把长期记忆和知识语义检索补强；第十六阶段再把复杂任务、Skill、MCP、Agent loop 的执行体验接进同一条主链路。
