# 第十八阶段：复杂对话语义理解与低置信度决策增强

## 摘要

第十八阶段聚焦聊天主链路中“听懂用户”的质量上限。当前第十二到第十七阶段已经让闲聊、复杂聊天、意图识别、上下文选择、记忆、人格、任务和工具链路形成后端闭环；但从完成度分析看，复杂对话和意图识别仍以 deterministic / rule-first 为主。

本阶段不重写现有 BrainDecisionService 和 ChatExperienceService，而是在已有稳定规则链路上增加语义状态抽取、低置信度模型复核、跨轮目标跟踪、多意图拆解和可解释决策证据。

本阶段只做后端，不新增 UI。

## 阶段定位

第十八阶段回答以下问题：

```text
用户说得含糊时，系统是否知道自己没听懂
用户一个请求包含聊天、记忆、工具、任务时，系统是否能拆开
用户继续上一轮主题时，系统是否能跟住目标和约束
用户改变目标时，系统是否能识别转向
用户触发高风险执行时，系统是否能先澄清或审批
规则判断低置信度时，是否能用模型做受控复核
所有决策是否有 trace、reason_codes 和 eval evidence
```

## 当前基线判断

| 能力 | 当前完成度 | 主要问题 |
|---|---:|---|
| 闲聊识别 | 约 82% | 基础稳定，但对暗含任务/记忆的轻量输入识别不足 |
| 复杂对话状态 | 约 76% | working state 已有，语义抽取偏关键词 |
| 意图识别 | 约 80% | intent/mode/context 已成链，model_assist=false |
| 澄清策略 | 约 78% | 有 1-3 个问题机制，但缺少分层澄清策略 |
| 上下文选择 | 约 82% | 能选记忆/资源/人格，但缺少语义证据和冲突处理 |

## 阶段原则

1. 规则链路继续作为默认稳定路径。
2. 模型辅助只在低置信度、复杂多意图、高风险或上下文冲突时触发。
3. 模型复核只能提出建议，最终仍由 BrainDecisionService 生成可审计决策。
4. 高风险、工具、MCP、Skill、资产、记忆写入仍必须经过原有 Safety、Approval、Asset Broker 和 Capability Graph。
5. 所有语义抽取结果都要保存来源、置信度、reason_codes 和 trace refs。
6. 不把模型复核失败当作主链路失败，必须有规则 fallback。

## 小阶段 18.1：DialogueState 契约升级

### 目标

把当前 working state 从“轻量会话状态”升级为可被复杂对话、意图识别和上下文网关共同消费的 DialogueState。

### Schema 字段

```text
dialogue_state_id
conversation_id
member_id
active_topic
user_goal
goal_status
goal_history
known_constraints
soft_preferences
hard_constraints
decisions_made
open_questions
pending_confirmation
topic_shift
last_user_action
candidate_next_actions
referenced_memories
referenced_artifacts
confidence
source_turn_id
trace_id
created_at
updated_at
```

### 目标状态

```text
new
active
paused
changed
completed
abandoned
unclear
```

### 验收

```text
复杂对话每轮都能生成或更新 DialogueState
用户切换主题时 topic_shift=true
用户否定上一轮决定时 decisions_made 可被 supersede
DialogueState 不包含 secret 明文
DialogueState 写入必须有 trace span
```

## 小阶段 18.2：多意图拆解与冲突识别

### 目标

解决“一句话里同时有闲聊、记忆、任务、工具、约束和风险”的情况。

### Intent 拆解字段

```text
primary_intent
secondary_intents
actionable_intents
non_actionable_intents
risk_intents
memory_intents
tool_intents
skill_intents
mcp_intents
conversation_intents
conflicts
confidence
reason_codes
```

### 冲突类型

```text
memory_vs_privacy
tool_vs_missing_asset
execution_vs_missing_approval
casual_vs_action_request
old_goal_vs_new_goal
persona_request_vs_safety_boundary
high_risk_vs_ambiguous_destination
```

### 验收

```text
混合输入不会被粗暴归为 casual_chat
记忆写入和任务执行能拆成两个候选意图
高风险动作即使包在闲聊里也能被识别
冲突必须进入 clarification 或 safe fallback
```

## 小阶段 18.3：低置信度模型复核

### 目标

在规则判断不稳时启用模型辅助复核，但不破坏可测试性和本地降级能力。

### 触发条件

```text
intent confidence < threshold
mode confidence < threshold
同一输入命中三个以上意图
需要工具/Skill/MCP 但能力不可用
高风险动作目的地不明确
上下文与用户当前输入冲突
用户引用“刚才/那个/继续”但 working state 不足
```

### Verifier 输出

```text
candidate_primary_intent
candidate_secondary_intents
candidate_mode
missing_information
risk_notes
context_notes
clarification_candidates
confidence
explanation_summary
```

### 安全边界

```text
Verifier 不返回工具调用
Verifier 不解析 asset handle
Verifier 不读取 secret
Verifier 输出必须经过 schema validation
Verifier 失败时回退规则路径
```

### 验收

```text
model_assist 可通过配置关闭
关闭模型辅助时测试仍通过
低置信度复核写 trace span
模型复核只影响建议，不直接越过 Safety/Approval
```

## 小阶段 18.4：澄清策略分层

### 目标

让系统在不知道、不能做、需要确认、风险不明时问不同类型的问题。

### 澄清类型

```text
missing_goal
missing_scope
missing_asset
missing_destination
ambiguous_reference
conflicting_context
risk_confirmation
approval_precheck
tool_unavailable_fallback
```

### 问题规则

```text
最多 3 个问题
优先问阻断执行的问题
能部分回答时先给部分回答
高风险动作必须明确等待确认
不要暴露内部模块名称
不要把 MCP/Skill/Tool 术语强加给用户
```

### 验收

```text
澄清 response_plan 包含 clarification_type
高风险缺目的地不会直接执行
工具不可用时给出安全替代路径
复杂对话澄清后能回到原目标
```

## 阶段总验收标准

第十八阶段完成时必须满足：

```text
复杂对话有 DialogueState 结构化状态
多意图输入能拆解并写入决策证据
低置信度场景可选模型复核，且有规则 fallback
澄清策略按缺失信息和风险分层

```
