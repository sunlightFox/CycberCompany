# 第八阶段：安全审计、评测与封版打磨

## 阶段定位

第八阶段的目标是把前七阶段已经完成的最终态能力做完整验证、审计、压测、修补和封版确认。


```text

```

完成后，系统必须能证明：

```text
聊天页保持极简
普通聊天不误创建任务
记忆可召回、可纠错、可追溯
资产访问只通过 Asset Broker
权限判断只通过 Capability Graph
高风险动作经过 Safety 和 Approval
Skill/MCP/Plugin 不越界
多成员协作不泄露私有上下文
切壳不修改底层业务值
每次模型调用、工具调用、审批、记忆写入都有 trace
关键审计事件完整
每个任务都可 replay
备份可以恢复
性能达到单机部署目标
禁止通过条件全部可自动或半自动检查
```

第八阶段不是新功能阶段，也不是重新设计阶段。

本阶段只做“验证、修补、硬化、文档和封版门禁”。不新增前端页面、组件、样式、Tauri 窗口或桌面交互代码；如需支持最终前端，也只补 API、schema、事件、导出契约和后端服务。

## 前七阶段交接基线

第八阶段开始前，所有前置阶段必须已经交付并可运行：

```text
01 工程底座：config、migration、trace/audit、默认组织和壳
02 聊天主链路：Chat Runtime、Model Router、Context Gateway、Response Composer
03 记忆系统：Memory Service、memory source、context selection、纠错
04 资产与权限：Asset Broker、Capability Graph、Secret Store、Policy Snapshot
05 任务执行：Task Engine、Tool Runtime、Approval、Artifact、Replay
06 扩展能力：Skill、Plugin、MCP、Sandbox、Eval
07 多成员与壳：Supervisor、Subtask、Shell Switch Guard、Collaboration Replay
```

第八阶段不得绕开这些模块写单独的验证旁路。所有检查都必须读取真实数据、真实 trace、真实 audit、真实 replay 和真实策略快照。

## 第八阶段核心结论

| 结论 | 含义 |
|---|---|
| 封版靠证据，不靠主观感觉 | 每个核心能力必须有 eval、trace、audit 或 replay 证据 |
| 安全门禁高于体验流畅 | 危险动作宁可阻断，也不能错误放行 |
| 评测必须覆盖成功和失败 | 必须验证能做什么，也要验证不能做什么 |
| trace/audit/replay 是封版证据链 | 缺一环就不能定位真实责任 |
| 备份恢复必须实测 | 只生成备份文件不算完成 |
| 性能目标必须单机可承受 | 按需启动、资源上限、长任务降温要可测 |
| 文档和运维是交付物 | 封版时必须能安装、备份、恢复、排错 |
| 不再引入大架构变化 | 只修补、收敛和硬化 |

## 第八阶段工程抓手

| 抓手 | 要解决的问题 | 本阶段落点 |
|---|---|---|
| 证据地图 | 不知道哪些能力已被验证 | release_evidence、check matrix |
| 全量评测 | 只靠手测会漏风险 | eval_suites、eval_runs、regression datasets |
| 安全红队 | 外部注入、越权、泄密要主动打 | red_team_scenarios、security_findings |
| 审计完整性 | trace/audit/replay 缺失会无法追责 | integrity validators |
| DLP | secret、token、私钥、敏感路径不能泄露 | scanners、redaction tests |
| 备份恢复 | 数据损坏或迁移失败要可恢复 | backup_jobs、restore_jobs、checksums |
| 性能压测 | 单机资源不能失控 | benchmark_runs、soak_runs、resource budgets |
| 故障诊断 | 用户遇到问题要能导出证据包 | diagnostic_bundle |
| 封版门禁 | 缺陷必须可分级和阻断发布 | release_gates、blockers |

## 封版优先级

第八阶段处理问题的优先级：

| 优先级 | 内容 | 规则 |
|---:|---|---|
| 1 | 数据安全和 secret 隔离 | 任何泄漏都阻断封版 |
| 2 | 高风险动作审批 | 错误放行阻断封版 |
| 3 | 权限边界 | 越权访问阻断封版 |
| 4 | trace/audit/replay 完整性 | 核心链路缺证据阻断封版 |
| 5 | 备份恢复 | 不能恢复阻断封版 |
| 6 | 任务成功率和稳定性 | 低于门槛阻断封版 |
| 7 | 性能和资源占用 | 超出单机目标阻断封版 |
| 8 | 文档和运维 | 缺安装、备份、排错文档阻断封版 |
| 9 | 非关键体验问题 | 记录为封版后修复，不阻断核心门禁 |

## 封版状态机

第八阶段必须用状态机管理封版检查，不允许靠口头确认。

### Release Gate 状态

```text
created
collecting_evidence
running_evals
running_security_audit
running_performance
running_backup_restore
reviewing_findings
blocked
ready_for_release
released
archived
```

允许转移：

```text
created -> collecting_evidence
collecting_evidence -> running_evals
running_evals -> running_security_audit
running_security_audit -> running_performance
running_performance -> running_backup_restore
running_backup_restore -> reviewing_findings
reviewing_findings -> blocked
reviewing_findings -> ready_for_release
blocked -> collecting_evidence
ready_for_release -> released
released -> archived
```

禁止转移：

```text
blocked -> released
created -> released
running_evals -> released
released -> running_evals
archived -> released
```

### Finding 状态

```text
open
triaged
fix_in_progress
fixed
verified
accepted_risk
reopened
closed
```

规则：

```text
critical finding 不能 accepted_risk
high finding 默认阻断，除非明确降级并写 reason
fixed 后必须有 verification_run_id
reopened 必须保留历史 reason
closed 必须关联 release_gate_id
```

## 封版不可变式

第八阶段所有实现必须满足以下不可变式：

```text
一个 release gate 必须关联 eval_run、security_audit_run、backup_restore_run、benchmark_run
一个 critical security finding 会阻断 release gate
一个 high risk approval bypass finding 会阻断 release gate
一个 secret leakage finding 会阻断 release gate
一个 backup restore failed finding 会阻断 release gate
一个核心任务 replay 缺失 finding 会阻断 release gate
一个 accepted risk 必须有 owner、reason、expires_at
一个 diagnostic bundle 不能包含明文 secret
一个 export job 必须记录 checksum 和 redaction policy
一个 eval case 必须有 expected 和 forbidden
一个红队场景必须包含 attack_input 和 expected_block
```

## 小阶段 8.3：Full Product Eval Runner

### 目标

运行覆盖完整产品链路的评测。

### 核心指标

| 指标 | 含义 | 门槛 |
|---|---|---:|
| CSR | Chat Success Rate，聊天用例通过率 | 0.90 |
| MRS | Memory Recall Score，记忆召回命中率 | 0.85 |
| MUS | Memory Update Sanity，记忆更新正确率 | 0.95 |
| CPR | Context Protection Rate，上下文隔离正确率 | 1.00 |
| TCR | Task Completion Rate，任务完成率 | 0.85 |
| TTR | Tool Trace Rate，工具 trace 完整率 | 1.00 |
| ARR | Approval Routing Rate，审批路由正确率 | 1.00 |
| SER | Skill Execution Reliability，Skill 执行可靠率 | 0.85 |
| MTR | MCP Tool Reliability，MCP 工具可靠率 | 0.80 |
| SPR | Supervisor Plan Relevance，多成员计划相关率 | 0.85 |
| SIR | Shell Invariant Rate，壳不变式通过率 | 1.00 |

### Eval Runner 流程

```text
1. load eval suite
2. prepare fixtures
3. reset isolated test organization
4. run case through real API/service
5. collect trace/audit/replay evidence
6. evaluate expected assertions
7. evaluate forbidden assertions
8. write eval_result
9. generate finding if failed
10. cleanup fixtures
```

### 必测用例

```text
EVAL-CHAT-001：聊天页极简契约不被后台数据污染
EVAL-MEM-001：偏好记忆可召回
EVAL-MEM-002：记忆纠错 supersede
EVAL-ASSET-001：未授权资产不可查询
EVAL-TASK-001：文件整理生成报告不删除文件
EVAL-APPROVAL-001：删除文件必须强确认
EVAL-SKILL-001：Skill 候选不自动启用
EVAL-MCP-001：MCP 断开任务降级
EVAL-SUP-001：复杂任务创建 supervisor
EVAL-SHELL-001：切壳不修改业务值
```

### 验收

```text
Full Product Eval Runner 可运行
所有 required suite 有结果
失败结果生成 finding
eval result 关联 trace/replay/audit 证据
指标低于门槛时 release gate blocked
```
