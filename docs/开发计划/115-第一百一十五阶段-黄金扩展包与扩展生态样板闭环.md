# 第一百一十五阶段 - 黄金扩展包与扩展生态样板闭环

## 阶段定位

phase101 到 phase112 已经把扩展兼容层、运行时同步、diagnostics、deliverable proof 这条底层链路搭了起来。
现在的主要问题不再是“能不能导入扩展”，而是“导入后有没有高价值样板、能不能形成重复可用的真实生态”。

这一阶段聚焦少数高价值执行域，做出可以反复复用的黄金扩展包，不再继续泛化兼容格式。

## 目标

```text
沉淀 3 到 5 个高价值黄金扩展包
让扩展生态从兼容层存在，升级为真实任务可复用
统一 install / enable / diagnostics / plan-run / deliverable proof 的样板实践
降低新扩展开发和接入成本
```

## 重点

### 115.1 黄金扩展包选型

```text
优先做内容平台、代码仓协作、办公生产力、浏览器登录态、渠道消息闭环五类样板
每个样板都必须绑定真实 task domain，而不是只停在 demo tool
优先覆盖当前仓库已经有基础测试和能力积累的执行域
```

### 115.2 样板闭环统一

```text
每个黄金扩展包都要具备 preview-import、install、enable、bind、diagnostics、plan-run、task delivery proof
同一扩展的 skill / mcp / external platform contribution 要共享统一 runtime snapshot
扩展 disable / rollback 后，运行时贡献和工具状态要同步撤出
```

### 115.3 开发者体验收口

```text
提供最小 manifest 模板、最小 runtime 插件模板、最小 diagnostics 样例
统一常见失败原因：missing_bindings、runtime_sync_missing、external_runtime_required
让新扩展作者优先复用黄金样板，而不是重新摸索契约
```

## 验收

```text
至少 3 个黄金扩展包具备完整 install -> enable -> task delivery 闭环
每个黄金扩展包都有 diagnostics、runtime snapshot、deliverable proof 样板
扩展 disable / rollback 后运行时注册表、tool 状态、plan-run 结果同步更新
新增扩展接入时可以直接复用黄金样板文档和模板
extension_ecosystem 域的 final_deliverable_rate 稳定维持在高位
```
