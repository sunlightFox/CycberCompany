# 第十一阶段：封版前能力补齐与深度硬化

## 摘要

第十一阶段承接第十阶段后的真实运行状态。第十阶段已经把工程健康恢复到可继续推进的状态；本阶段不再处理基础回归，而是把当前仍然 `degraded` / `placeholder` 的后端能力补到可封版、可审计、可长期维护的程度。

本阶段仍然只做后端，不新增 UI、Tauri、前端页面、组件、样式或桌面交互代码。

本阶段目标：

```text
把待优化项拆成可执行阶段
把 degraded 能力补成 implemented 或明确 accepted risk
把 placeholder 能力补成最小可用后端契约
把封版证据链从 smoke 推进到真实链路验证
```

## 阶段原则

1. 不做前端实现，只稳定未来 UI 可消费的 API、schema、事件和错误模型。
2. 不扩大产品边界，不引入社区市场、多设备同步、云 SaaS 或新壳类型。
3. 不把 degraded 能力标成 implemented，除非有测试、eval 和 runtime contract 证据。
4. 所有涉及工具、MCP、终端、浏览器、文件系统、网络动作的能力必须经过 Safety、Approval、Trace。
5. 资产访问仍必须经过 Asset Broker，权限判断仍必须经过 Capability Graph。
6. 记忆写入必须保留 source，模型调用、工具调用、审批和记忆写入必须有 trace。
7. 每个小阶段完成后都要更新 runtime contracts / design gaps，避免文档和代码漂移。

## 小阶段 11.1：Response Composer 结构化闭环

### 目标

让最终回复不再只是模型文本或固定文案透传，而是所有用户可见路径都输出稳定的 `ResponsePlan`，方便未来极简聊天 UI 消费。

### 修复内容

```text
扩展 ResponsePlan 字段使用范围
ChatService 完成/失败/取消路径都记录 response_plan
任务创建、任务完成、approval.required、artifact.created 都输出结构化 payload
隐私阻断、模型未配置、工具不可用、MCP 失败等错误路径都通过 composer
response.delta 保留纯文本 fallback，但 completed 事件携带结构化摘要
```

### 关键契约

`ResponsePlan` 至少覆盖：

```text
summary
sections
task_status
approval_prompt
artifact_refs
safety_notice
memory_notice
follow_up_options
plain_text
structured_payload
```

### 验收

```text
普通 direct 聊天有 plain_text 和 summary
任务路径有 task_status 和 artifact_refs
approval 路径有 approval_prompt
安全阻断路径有 safety_notice
失败路径说明失败点和下一步
测试覆盖所有用户可见出口不绕过 composer
```

## 小阶段 11.2：Settings API 最小可用契约

### 目标

把 `SettingsAPI` 从 placeholder 补成最小可用后端能力，为未来系统管理页提供稳定契约。

### 修复内容

```text
新增 settings schema
新增 settings migration
新增 SettingsRepository
新增 SettingsService
实现读取当前设置
实现安全更新部分设置
更新设置时写 audit 和 trace
敏感设置只保存 secret_ref，不返回明文
```

### 设置范围

首批只覆盖后端必需项：

| 设置域 | 内容 |
|---|---|
| model_routing | cloud fallback、privacy policy、token budget |
| safety | risk thresholds、approval policy、terminal policy profile |
| vector | provider、enabled、degraded fallback |
| mcp | allowed stdio commands、default unknown tool status |
| memory | implicit extraction、candidate review threshold |

### 验收

```text
GET /api/settings 返回真实设置
PATCH /api/settings 只允许白名单字段
secret/token/password 不出现在响应、trace、audit
非法设置返回项目统一错误模型
runtime contract 中 SettingsAPI 不再是 placeholder
```

## 小阶段 11.3：VectorStore 真检索接入

### 目标

把 VectorStore 从“状态记录和 degraded refs”推进到 Memory/Knowledge 实际检索链路。

### 修复内容

```text
明确 Chroma provider 初始化方式
实现 embedding provider 抽象
Memory 写入后尝试 embedding + vector upsert
Knowledge chunk 写入后尝试 embedding + vector upsert
Memory search 融合 vector hits + FTS hits
Knowledge search 融合 vector hits + FTS hits
检索结果记录 selection_reason、provider、degraded_reason
向量不可用时保持 FTS fallback，并写 degraded trace
```

### 验收

```text
Chroma 可用时 vector refs 不再是 skipped
Chroma 不可用时明确返回 degraded，不伪装成功
Memory/Knowledge 检索能解释结果来自 vector、FTS 或融合排序
eval 覆盖 provider available/unavailable 两种路径
runtime contract 中 VectorStore 状态与真实 provider 一致
```

## 小阶段 11.4：Terminal OS Sandbox 深度硬化

### 目标

把当前 task artifact cwd + 命令策略的软隔离，推进到可封版接受的终端执行边界。

### 修复内容

```text
定义 TerminalExecutor interface
实现 LocalArtifactExecutor 作为当前默认降级执行器
预留 WindowsSandboxExecutor / DockerExecutor / WslExecutor
终端命令从字符串策略升级为结构化 command policy
R3-R5 命令走 approval_required
R6-R7 命令 deny
禁止任意 cwd、环境变量注入和敏感路径访问
stdout/stderr 统一 DLP/redaction 后写 artifact
终端日志写 checksum 和 policy snapshot
```

### 验收

```text
terminal.run 必须绑定 task
自定义 cwd 被拒绝
敏感路径和破坏性命令被拒绝
可变更命令必须审批
终端日志不包含明文 secret
trace/audit/replay 能看到 policy snapshot 和 sandbox profile
TerminalRunner 如果没有 OS sandbox，必须继续标 degraded 或 accepted risk
```

## 小阶段 11.5：MCP 策略与资源边界深化

### 目标

让 MCP 不只是“能登记、能同步、能调用”，而是符合最终态资源边界和权限策略。

### 修复内容

```text
MCP server policy 支持 organization/member/role/department scope
sync 后 unknown/high-risk tool 默认 disabled 或 approval_required
MCP resource 和 prompt 标记 untrusted
MCP prompt 不允许直接提升为系统指令
MCP tool call 前合并 capability、safety、skill policy
env_refs 必须从 SecretStore 解析，禁止明文 env
MCP 输出统一 redact，写 mcp_call 与 audit
```

### 验收

```text
未知 stdio command 被拒绝
明文 env 被拒绝
未授权成员不能调用 MCP tool
MCP resource/prompt 进入上下文时标记 untrusted
高风险 MCP tool 不会自动 active
MCP 断开时任务 failed/degraded，不伪成功
```

## 小阶段 11.6：Task Agent Loop 最小闭环

### 目标

把 `agent` 模式从“模式字段和关键词分流”推进到最小真实 agent loop，但仍受预算、权限和审批约束。

### 修复内容

```text
新增 AgentLoopRunner
新增 agent_loop_steps / observations 存储或复用 task_events
实现 observe -> plan -> act -> evaluate -> revise -> stop
每轮写 trace span 和 task event
工具调用仍走 ToolRuntime
高风险动作仍走 Approval
缺资源或缺权限时 safe fail 或 ask clarification
任务结束后生成 reflection candidate，不自动启用 Skill
```

### 预算约束

```text
max_loop_steps
max_tool_calls
max_model_calls
max_runtime_seconds
max_total_cost
stop_reason 必填
```

### 验收

```text
固定步骤任务继续走 workflow
探索型任务可进入 agent loop
agent loop 超预算会停止并写 stop_reason
高风险动作不能被 loop 绕过审批
replay 能还原每轮 observe/act/evaluate
失败任务保留可恢复状态
```

## 小阶段 11.8：Heart 与 Persona 策略化

### 目标

保持当前 deterministic 能力可用，同时把 Heart / Persona 推进到可配置、可审计、可被 Response Composer 使用的后端策略层。

### 修复内容

```text
Persona profile 支持 tone_policy、disclosure_policy、risk_tone_policy
Heart state 支持 urgency、mood、relationship_temperature、deescalation_boundary
高风险/高影响事务降低拟人强化
Context Gateway 注入 persona/heart 摘要，不注入后台组织结构
Response Composer 使用 persona/heart 调整表达，但不能改变事实和安全判断
```

### 验收

```text
persona/heart 只影响表达姿态，不影响权限和安全决策
紧急、焦虑、愤怒等输入可产生不同 heart summary
高风险审批文案保持清楚、克制、可审计
聊天页约束仍成立：不显示组织和壳
```

## 第十一阶段验收标准

第十一阶段通过必须满足：

```text
ruff/mypy/pytest 全部通过
SettingsAPI 不再是 placeholder
ResponseComposer 全路径输出结构化 ResponsePlan
VectorStore 真实接入检索链路或明确 accepted risk
```

## 不允许通过验收的情况

出现以下任一情况，本阶段不能通过：

```text
ruff/mypy/pytest 任一失败
SettingsAPI 仍返回占位能力
高风险工具可绕过 Safety 或 Approval
MCP 明文 env 可登记
终端可访问敏感路径或任意 cwd
VectorStore 伪装成功但实际只写 skipped refs
ResponseComposer 被用户可见路径绕过
release diagnostic/trace/audit 出现明文 secret
runtime contracts 继续夸大能力状态
```

## 与第八、九、十阶段关系

第八阶段是封版门禁，第九阶段是设计偏差修复，第十阶段是工程健康恢复。

第十一阶段的定位是：

```text
在工程健康全绿之后
把剩余 degraded/placeholder 能力补齐
并为第八阶段最终封版提供更深证据链
```

完成第十一阶段后，必须回到第八阶段执行完整封版流程；如仍发现 blocker，则回流到本阶段对应小阶段继续修复。
