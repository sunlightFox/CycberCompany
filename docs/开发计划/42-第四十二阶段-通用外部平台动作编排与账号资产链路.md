# 第四十二阶段 - 通用外部平台动作编排与账号资产链路

## 阶段背景

第三十四阶段已经把 Hermes Agent 式自然语言聊天交互作为主入口，第三十七阶段规划了持久浏览器会话与网页登录资产化，第四十阶段补齐外部消息渠道，第四十一阶段继续修复聊天主链路质量缺口。下一步需要把“用户用自然语言要求在某个外部平台完成动作”收敛成通用后端能力。

典型输入不是技术指令，而是自然表达：

```text
帮我在某个平台发一篇文章
用我们的运营账号登录后发布这段内容
把这条动态发到指定平台
去后台把今天的公告同步一下
```

系统不应为了某个示例平台写死分支，也不应让 Skill 或工具直接读取账号密码。正确形态应接近 OpenClaw / Hermes Agent 的通用执行模型：先理解用户目标，再解析平台和动作，再通过系统资产能力拿到受控账号句柄，必要时反问选择账号，最后在 Safety、Approval、Trace 约束下执行。

本阶段仍只做后端、schema、migration、repository、service、API、tests、evals 和文档。不新增产品前端页面、组件、样式、Tauri 窗口或桌面端交互代码。测试需要模拟外部平台时，优先使用后端 mock provider、测试 fixture 或受控本地测试服务；若后续确需真实测试网页，也必须明确标记为测试资源，不进入产品 UI。

## 核心目标

本阶段完成后，后端应支持：

```text
从自然语言中识别外部平台动作意图
通过通用 Platform Resolver 解析平台、动作、内容、目标对象和约束
通过 Asset Broker 查询 account/browser_session/API credential 等资产候选
多个账号候选时生成自然语言澄清问题，而不是自行猜测
缺少账号或授权时给出可恢复回复，不伪造登录成功
生成通用外部平台动作计划，并交给受控工具、浏览器或 provider 执行
高风险动作经过 Safety、Capability Graph 和 Approval
每一步都有 trace、audit、redaction 和可回放证据
平台和账号信息只来自资产与配置，不写死在聊天、任务或 Skill 代码里
```

## 阶段原则

1. 平台是可配置 target，不是代码里的固定分支。
2. 账号是 Asset，不是聊天上下文里的明文参数。
3. Skill 负责做事方法，不负责绕过系统查询账号、密钥或 session。
4. 工具只能拿到 Asset Broker 发放的受控 handle，不能直接读取 secret。
5. 用户自然表达可以触发任务规划，但不能跳过权限、安全和审批。
6. 多账号、多平台、多 pending action 时必须反问，不能静默选择。
7. 发布、发送、修改外部状态等动作默认比只读动作风险更高。
8. trace 记录平台、动作、账号资产引用、审批和证据，不记录明文 secret、token、cookie、密码或私钥。
9. 模拟平台只用于测试通用链路，不把模拟平台逻辑写成产品特例。
10. 壳系统只改变 UI 标签、模板和文案，不改变核心 Organization / Asset / Task 语义。

## 对标结论

### OpenClaw 采用点

```text
用户说自然语言目标，而不是调用具体 API
Agent 通过环境和工具完成网页、文件、任务等操作
执行过程需要可观察、可恢复、可解释
```

对我们的启发：

```text
外部平台动作应落到通用 action plan
平台适配器和工具执行要可替换
执行证据应进入 trace 和 task replay
```

### Hermes Agent 采用点

```text
低风险步骤可自动推进
危险步骤必须停下来等待用户确认
用户确认释放的是已绑定的 pending action
```

对我们的启发：

```text
发文章、发消息、提交表单等外部状态变更必须有 pending action binding
确认时要绑定平台、账号、动作和内容摘要
用户一句“好的”不能释放多个候选动作
```

## 小阶段 42.1：schema、migration 与 repository

### 目标

建立外部平台动作编排的持久化底座，让后续 resolver、账号选择、执行计划和证据都能被审计和回放。

### 实现要求

```text
新增 external_platform_targets
新增 external_platform_action_intents
新增 external_platform_action_plans
新增 external_platform_executions
新增 external_platform_plan_events
所有表包含 organization_id、member_id、trace_id、status、created_at、updated_at
敏感字段只保存 asset_id、secret_ref 或 redacted summary，不保存明文 secret
repository 不返回裸 dict，使用明确 schema
```

### 验收

```text
migration 可重复初始化测试库
repository 可创建、查询、更新 plan 状态
redacted payload 不含 token、password、cookie、private key
```

## 小阶段 42.2：Platform Resolver

### 目标

从用户自然语言中识别“外部平台动作”，但只输出通用结构，不把某个平台写成业务分支。

### 实现要求

```text
解析 platform_hint、action_type、content、target_hint、constraints
platform_hint 通过配置 target aliases 匹配 platform_key
action_type 使用通用枚举，例如 publish_content、send_message、update_profile、read_status
低置信度时返回 clarification_needed
缺内容、缺目标、缺平台时返回自然语言追问所需字段
```

### 验收

```text
“帮我在某平台发一篇文章”可解析为 publish_content
平台别名来自 target 配置，不来自 hardcoded if/else
低置信度不会创建执行计划
resolver trace 记录输入摘要、匹配结果和置信度
```

## 小阶段 42.3：Asset Broker 账号候选链路

### 目标

把“公司账号里的某平台账号”落成核心层可接受的资产查询：根据 platform_key、asset_type、capability 和权限范围，从 Asset Broker 获取候选账号 handle。

### 实现要求

```text
不得从数据库或 secret store 直接查询明文账号密钥
Asset Broker 接收 platform_key、asset_type、required_capabilities、member_id、organization_id
Capability Graph 判断当前成员是否可使用该账号执行目标动作
返回 AccountAssetCandidate，只含显示名、能力、风险、asset_id、脱敏摘要
候选账号数量为 0、1、多个时分别进入不同决策分支
```

### 验收

```text
无账号时回复不能声称已登录
单账号时可自动进入计划创建，但高风险发布仍要审批
多账号时必须反问用户选择哪个账号
候选结果和 trace 不出现明文 password、token、cookie
```

## 小阶段 42.4：多账号澄清与聊天衔接

### 目标

让用户感觉流程自然顺畅：系统知道缺什么就问什么，用户回答后能接上原来的任务，而不是重新开始。

### 实现要求

```text
clarification state 绑定 conversation_id、intent_id、candidate asset ids 和过期时间
多账号时给出用户可理解的账号显示名和用途，不暴露 secret
用户选择后校验候选仍有效
回答模糊或候选变化时继续澄清
澄清完成后继续创建 action plan
```

### 目标回复形态

```text
我找到了 2 个可用于这个平台的账号：

1. 运营账号 A
2. 品牌账号 B

你想用哪一个发布？我会先准备草稿，真正提交发布前再停下来给你确认。
```

### 验收

```text
多账号不会自动选第一个
用户回复“用品牌账号”能绑定到对应 asset_id
用户改口换账号时旧选择被 superseded
澄清事件有 trace 和 redaction
```

## 小阶段 42.5：Action Orchestrator

### 目标

把已解析的 intent 和已选账号转换成可执行、可审批、可回放的通用动作计划。

### 实现要求

```text
计划状态包含 draft、awaiting_clarification、awaiting_approval、running、completed、failed、cancelled
每个步骤声明 step_type、risk、required_capability、executor、input_redacted
执行前重新校验 asset handle 和 capability
失败时记录 failed_step、reason、recoverable、next_action
取消或用户拒绝时不继续执行后续步骤
```

### 验收

```text
publish_content 计划包含登录状态检查、内容准备、提交前审批
计划状态变化可追踪
失败不会伪装为完成
取消后不能继续执行旧 pending action
```

## 小阶段 42.6：Provider / Browser Executor interface

### 目标

定义通用执行器接口，让同一套计划既可以由平台 provider，也可以由浏览器执行器完成。测试阶段使用 fake provider 或受控本地测试服务验证完整链路。

### 实现要求

```text
Executor 输入 action plan step 和 asset handle
Executor 不接收明文 secret，只接收 Asset Broker 发放的临时 handle
Browser executor 产出 URL、title、snapshot、screenshot ref、form submit evidence 等证据摘要
Provider executor 产出 provider request id、status、redacted response
Fake provider 覆盖登录、填内容、提交发布、失败恢复等路径
```

### 验收

```text
fake provider 可完成完整“账号选择-登录-发布”E2E
browser executor 契约不依赖具体平台名称
执行证据进入 external_platform_executions 和 trace
secret 不出现在 executor log、trace、test report
```

## 小阶段 42.7：Safety、Approval、Trace 集成

### 目标

让外部平台动作成为受控高风险链路，而不是自然语言一触发就直接发布。

### 实现要求

```text
Safety 根据 action_type、platform sensitivity、account sensitivity、content risk 评估风险
Capability Graph 校验成员是否能使用对应资产执行该动作
Approval 绑定 plan_id、asset_id、platform_key、action_type、content_summary
用户确认释放的是唯一 pending action
Trace 覆盖 resolver、asset candidate、clarification、plan、approval、execution、result
```

### 验收

```text
发布内容默认 requires_approval
多个 pending publish action 时，用户一句“确认”不会执行
审批拒绝后计划进入 cancelled 或 denied
trace 不记录明文密钥、cookie、token、正文敏感值
```

## 小阶段 42.8：模拟平台 E2E 与回归门禁

### 目标

用模拟平台验证完整流程：用户自然语言提出发布请求，系统解析平台，取账号，必要时反问，准备发布，审批后执行，并返回证据。

### 推荐 E2E 场景

```text
单账号：用户要求发布文章 -> 自动选择唯一账号 -> 等待发布确认 -> 确认后 fake provider 发布成功
多账号：用户要求发布文章 -> 返回账号选择澄清 -> 用户选择账号 -> 等待发布确认 -> 发布成功
无账号：用户要求发布文章 -> 返回缺少账号资产的恢复提示 -> 不创建执行成功记录
审批拒绝：用户拒绝发布 -> plan cancelled -> 不调用 submit
执行失败：fake provider 返回登录失效 -> plan failed recoverable -> 提示重新授权或换账号
敏感内容：文章含 token/password -> Safety 阻断或脱敏确认 -> 不泄漏敏感内容
平台别名：同一 target 多个 alias 均可匹配，但代码无平台 if/else
```

## 与其他阶段关系

```text
第三十四阶段提供自然语言聊天入口，本阶段把外部平台动作接入该入口
第三十七阶段提供网页登录资产化基础，本阶段复用账号和浏览器 session 资产
第四十阶段提供外部通知，本阶段的审批和失败恢复可通过通知网关触达

```

## 最终验收定义

第四十二阶段完成时，应能回答：

```text
用户说“帮我在某平台发文章”时，系统是否能识别为通用 publish_content 动作
平台信息是否来自 target 配置，而不是代码硬编码
公司/组织账号是否通过 Asset Broker 取得候选，而不是工具直接读密钥
多个账号时是否自然反问，并能接上用户选择继续执行
```
