# 第一百一十三阶段 - 质量门禁执行恢复与检查矩阵跑通

## 阶段定位

phase104 已经把检查脚本、报告契约和 gate signal plane 的骨架补齐，仓库里也有对应测试。
但从真实执行角度看，`scripts/check.ps1` 仍然存在「测试能过、实际 smoke/profile 跑不稳」的问题，导致封版门禁、阶段回归和日常开发自检之间仍有断层。

这一阶段不再扩新能力，先把质量门禁恢复成真正可依赖的工程入口。

## 目标

```text
修复 check.ps1 在真实 profile 执行中的失败路径
统一 smoke / fast / api / security / release 五类 profile 的输入、输出、日志和报告行为
让阶段门禁从“契约存在”升级为“日常可跑、失败可定位、报告可追溯”
为后续真实稳定性治理和 release gate 提供可信入口
```

## 重点

### 113.1 检查脚本成功路径恢复

```text
修复日志文件创建、读取、慢测统计和报告写入之间的时序问题
确保 profile 成功、失败、部分跳过三种路径都能稳定生成报告
避免出现测试覆盖了失败路径，但成功路径在真机仍会报错
```

### 113.2 检查矩阵统一

```text
收口 ruff / mypy / pytest / issue gate / chat docs runner 的调用矩阵
统一各 profile 对 signal suite、输出目录、报告命名、退出码语义的约束
保证 smoke 是最小可用门禁，release 是最大闭环门禁
```

### 113.3 报告与诊断可读性

```text
报告里要明确看到 profile、命令矩阵、失败阶段、日志路径、耗时和 gate 结果
失败时优先暴露根因，而不是只给脚本异常
让开发者可以只看 check report 就知道该回哪个模块修复
```
