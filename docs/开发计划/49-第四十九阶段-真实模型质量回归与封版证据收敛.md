# 第四十九阶段 - 真实模型质量回归与封版证据收敛

## 阶段背景

第三十五到第四十八阶段分别补齐安全一致性、后台能力、浏览器/平台执行、治理闭环和媒体能力。接下来需要把这些能力用真实模型、真实 data、真实默认成员和完整 release gate 做一次集中收敛，避免只靠单元测试和 mock provider 得到“看起来完成”的结果。

第四十九阶段是后端封版前的质量和证据阶段：目标不是新增能力，而是证明能力在真实聊天主链路、任务链路、后台链路和安全链路里稳定、诚实、可回放。

## 核心目标

```text
真实模型聊天质量批次稳定重跑
35-48 阶段契约全部进入 release gate summary
质量用例不依赖生产硬编码补丁
长任务、浏览器、Skill、通知、checkpoint、外部平台、媒体形成组合 E2E
所有报告、trace、diagnostic 经过 secret 泄漏扫描
accepted risk 有明确 owner、阶段归属和关闭条件
```

## 阶段原则

1. 不用假回复掩盖真实模型问题。
2. 真实执行结果必须有 task/tool/artifact/trace 证据。
3. 没有执行就不能说完成。
4. 安全阻断也要有用户可恢复回复。
5. 每个 accepted risk 必须有关闭条件。
6. 后端封版证据优先，UI 不在本阶段范围内。
7. 不清空 data，不删除历史报告。

## 本阶段必须完成

```text
更新 CHAT-E2E quality runner 覆盖 35-48 新能力
新增组合 E2E：scheduled -> browser/media/skill -> approval -> notification -> replay
真实模型 smoke：direct、memory、task、privacy、tool boundary、quality reply
release gate phase35-48 summary 全部可读
diagnostic bundle 增加 leakage scan 和 accepted risk matrix
移除生产路径内测试 case id 依赖
生成封版报告：通过、失败、accepted risk、后续阶段
```

## 本阶段不做

```text
不新增 UI
不新增大型业务功能
不接入真实生产外部平台
不把 mock provider 结果说成真实外部平台成功
不降低 Safety、Approval、Asset Broker 和 Capability Graph 约束
```

## 小阶段总览

| 小阶段 | 名称 | 核心交付 |
|---:|---|---|
| 49.1 | 质量 runner 升级 | 覆盖新能力和真实模型回复质量 |
| 49.2 | 组合 E2E 设计 | 长期任务、浏览器、Skill、通知、媒体组合链路 |
| 49.3 | 真实模型 smoke | 默认成员、默认大脑、隐私和降级路径 |
| 49.4 | Release gate 收敛 | phase35-48 summary、evidence、diagnostic |
| 49.5 | 泄漏扫描 | report、trace、artifact metadata、diagnostic |
| 49.6 | accepted risk 管理 | owner、关闭条件、后续阶段绑定 |
| 49.7 | 封版报告 | 后端封版证据和下一轮开发入口 |

## 验收标准

```text
phase35-48 阶段测试全部通过或仅剩有 owner 的 accepted risk
真实模型质量批次全部落报告，不出现空白回复或伪完成
组合 E2E 至少覆盖 scheduled task、browser evidence、Skill governance、notification、checkpoint、media
release report 中每个阶段都有 suite_id、registered_cases、tables/contracts、evidence
diagnostic bundle 不含明文 secret/token/password/cookie/private key/local path
生产聊天路径无测试 case id 固定回复
后端封版报告可作为下一轮开发输入
```

## 与后续阶段关系

第四十九阶段完成后，后续阶段可以继续推进 UI、真实 provider、更多媒体生成和桌面原生能力。若第四十九阶段发现核心安全或质量缺口，应优先回补，而不是继续堆新能力。
