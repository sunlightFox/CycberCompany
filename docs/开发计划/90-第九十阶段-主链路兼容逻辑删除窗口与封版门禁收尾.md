## 删除策略

### 允许删除的前提

旧逻辑只有在满足以下条件后才能删除：

```text
新 runtime 已接管真实主路径
对应 acceptance matrix 全绿
诊断与 replay 已覆盖旧证据需求
旧逻辑不再是任何生产路径的唯一实现
渠道可靠性回归全绿
completed_with_evidence 相关 gate 全绿
```

### 典型删除对象

```text
chat.py 中 route-specific 旧 reply builder
已废弃的 finalize helper
只为旧 topology 保留的 compat branch
不再被主链使用的 heuristic shortcut
已被统一状态机替代的散落 pending / running / completed 文案桥
```

### 暂不删除对象

```text
仍被外部 API 或测试夹具依赖的 facade
仍承载数据迁移兼容责任的桥接层
尚未有替代 replay 读路径的诊断 helper
```

## 实施拆解

### 90.1 compat cleanup window 固化

目标：

```text
把“什么时候能删”从口头约定变成代码与文档中的明确窗口
```

交付：

```text
compat_cleanup_window 诊断细化
每个 compat shell 的 removal gate 列表
删除优先级清单
```

### 90.2 最小 release suite 固定

目标：

```text
把主链路封版最小测试集固定下来
```

交付：

```text
runtime topology
chat runtime closure
channel semantics
context gateway enhancement
tool loop
response visibility
ledger / memory
hook contract
acceptance matrix
```

### 90.3 删除动作的 fail-safe 流程

目标：

```text
删除兼容逻辑时，不因误删导致主链路失守
```

交付：

```text
删除前 checklist
删除后最小 smoke suite
删除回退策略
```

### 90.4 封版门禁收尾

目标：

```text
把聊天成熟度核心风险变成统一 gate
```

交付：

```text
visible leakage gate
false completion gate
no_turn gate
duplicate inbound gate
false interception gate
prompt / context contract gate
```
