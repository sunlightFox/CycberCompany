# 第十阶段：工程健康修复与封版前硬化

## 摘要

第十阶段用于承接第九阶段后的实际代码检查结果。它不是新产品版本，也不是继续扩张功能范围，而是把当前已经进入仓库、但尚未稳定通过测试和最终态约束的后端实现修到可继续封版的状态。

本阶段的核心任务是：

```text
先恢复工程健康
再收敛剩余偏差
最后为第八阶段封版门禁准备稳定证据
```

第十阶段完成后，仓库必须重新达到：

```text
pytest 全部通过
ruff check . 全部通过
mypy . 全部通过
后端主链路无已知阻断失败
runtime contracts 不夸大能力状态
生成物和本地数据不污染仓库
README 与当前能力一致
```

## 阶段定位

第九阶段已经把设计偏差修复方向列清楚；第十阶段专门处理“落地后发现的工程健康、回归失败、测试缺口、仓库卫生和封版前硬化问题”。

当前检查快照：

```text
pytest：84 个用例，70 通过，14 失败
ruff check .：9 个问题
mypy .：5 个问题
根目录缺少 .gitignore
存在 data / data_tmp_phase5* / __pycache__ / cache / egg-info 等生成物
README 停留在早期后端底座描述
```

本阶段优先级非常明确：

| 优先级 | 内容 | 规则 |
|---:|---|---|
| P0 | 让后端主链路和检查命令重新全绿 | 不全绿不进入新增硬化 |
| P1 | 修复最终态仍偏浅的安全、MCP、终端、向量、ReleaseGate 能力 | 不扩大产品范围，只补齐设计承诺 |
| P2 | 仓库卫生、文档、脚本、占位文案清理 | 为长期开发和封版降低噪声 |

## 阶段原则

1. 不新增前端页面、组件、样式、Tauri 窗口或桌面交互代码。
2. 不用修改测试断言来掩盖真实回归。
3. 不把 smoke 级实现标成最终态完整实现。
4. 优先修 runtime 错误，再修 lint/type，再补测试。
5. 所有安全相关修复必须覆盖 allow、deny、approval 三条路径。
6. 所有工具、MCP、Skill 修复必须保留 trace/audit。
7. 不删除用户数据；只增加 `.gitignore` 和清理规则，不自动移除未确认文件。

## 小阶段 10.1：P0 回归阻断修复

### 目标

先修会导致主链路直接失败的代码错误，让测试失败从“系统性崩溃”恢复到可定位的业务断言。

### 修复项

#### 10.1.1 修复 `tool_calls.risk_level` 更新

当前 `insert_tool_call` 已写入 `risk_level`，但 `update_tool_call` 不允许更新该字段，导致 ToolRuntime 安全决策后更新风险级别时报错。

修复要求：

```text
将 risk_level 纳入 TOOL_CALL_UPDATE_COLUMNS
确认 update_tool_call 可安全更新 risk_level
补测试覆盖 safety 决策提高 risk_level 后可更新
确认 tool_call replay 仍能读取最终 risk_level
```

验收：

```text
MCP 工具调用不再因 Unsupported tool_calls update columns: ['risk_level'] 崩溃
任务执行链路不再因 risk_level 更新失败而 failed
Skill run 不再被该问题连锁影响
```

#### 10.1.2 修复 `AssetHandleValidateRequest` 未定义

修复要求：

```text
在 tools.py 正确导入 AssetHandleValidateRequest
确认文件中所有 Asset Broker schema import 完整
补最小导入/执行测试
```

验收：

```text
ruff 不再报 F821
mypy 不再报 Name AssetHandleValidateRequest is not defined
asset.validate_handle 工具路径可运行
```

#### 10.1.3 修复 Context Gateway persona 失败

当前成员模板中的 `persona` 是壳模板 ID，如 `reliable_warm`；PersonaHeartService 默认兜底却偏向 `persona_mem_xiaoyao`。两者不一致会导致 Context Gateway 构建失败。

修复要求：

```text
PersonaHeartService 支持按 member.persona_profile_id 懒创建默认 profile
对 shell 模板 persona id 生成稳定默认 persona profile
Context Gateway 取不到 persona 时不得直接让 turn failed
失败时可降级为 deterministic persona，并记录 degraded trace
```

验收：

```text
普通聊天事件重新出现 context.ready / intent.detected / mode.selected
显式记忆命令可继续走 memory.candidate / memory.updated
trace 中保留 context.build 和 brain.intent
```

## 小阶段 10.4：Runtime Contracts 降噪校准

### 目标

让系统运行契约真实表达当前能力，不把浅层、smoke、deterministic、fallback 能力标成最终完整实现。

### 修复要求

将运行契约状态按真实能力调整：

| 模块 | 建议状态规则 |
|---|---|
| SafetyService | 测试全绿且接入 Tool/Task/Skill/MCP 后为 `implemented`，否则 `degraded` |
| ResponseComposer | 如果仅基础 ResponsePlan，为 `degraded`；全路径结构化后为 `implemented` |
| HeartService | deterministic 规则可标 `degraded`，完整策略/配置后为 `implemented` |
| PersonaEngine | 支持 profile CRUD 和 Context 注入后至少 `degraded` |
| VectorStore | Chroma 可用且 Memory/Knowledge 真检索后为 `implemented`，否则 `degraded` |
| ReleaseGate | smoke 级门禁为 `degraded`，全证据链为 `implemented` |
| MCPConnectionManager | 无策略 allowlist 前为 `degraded` |
| TerminalRunner | 无 sandbox 前为 `degraded` |

### 验收

```text
/api/system/runtime-contracts 返回的状态和真实能力一致
/api/system/design-gaps 能列出仍需修复的偏差
```

## 与第八、九阶段关系

第九阶段解决“设计偏差如何修”；第十阶段解决“当前落地实现如何恢复工程健康并继续硬化”。

第十阶段结束后，必须回到第八阶段执行完整封版流程：

```text
1. 创建 release gate
2. 收集 evidence
3. 运行 eval
4. 运行 security audit
5. 运行 secret scan
6. 运行 permission boundary validator
7. 运行 backup/restore
8. 运行 benchmark
9. 导出 diagnostic bundle
10. 生成 release report
```

如果第八阶段仍发现 blocker，则 blocker 回流到第九阶段或第十阶段对应小阶段继续修复。
