# 第四阶段：资产中心与能力权限图

## 阶段定位

第四阶段的目标是把系统里的“资源世界”做成最终态可管理、可授权、可审计、可被安全感知的能力层。

完成后，系统必须能回答并执行这些资源层问题：

```text
系统里有哪些资产
每个资产属于哪一类
每个资产的风险和敏感等级是什么
谁能看到这个资产摘要
谁能对这个资产做哪些动作
哪些动作需要确认
哪些动作被拒绝
模型能拿到什么资源摘要
模型不能拿到什么 secret
工具或任务未来如何申请资源句柄
资产、授权、句柄、检索、审计如何回放
```

第四阶段不是任务执行阶段。它不负责“真的发帖、签名、转账、控制设备、读写任意文件”。它负责把资源登记、权限判断、句柄发放、知识库索引、上下文资源摘要这些底座立起来。

最终链路目标是：

```text
Asset Center 管理资产
  -> Secret Store 保存敏感引用
  -> Capability Graph 管理授权和策略
  -> Asset Broker 接收资源请求
  -> Capability Graph 给出 allow / deny / approval_required
  -> Asset Broker 发放短期 AssetHandle
  -> Context Gateway 注入资源摘要和句柄
  -> Chat Runtime / Response Composer 给用户透明说明
  -> Trace / Audit 记录全过程
```

第四阶段必须延续前三阶段边界：

```text
不改变聊天主链路
不改变记忆系统主结构
不让模型直接读 assets 表
不让模型直接看到 secret
不让 Skill 自己发现资源
不让 MCP 默认拥有全部资产
不执行真实高风险动作
不把资产中心塞进聊天页
```

本阶段完成后，第五阶段任务引擎与工具执行才能安全地申请资源句柄；第六阶段 Skill/MCP 才能声明依赖并被权限系统约束。

## 第三阶段交接基线

第四阶段开始前，第三阶段必须已经交付：

```text
稳定的 Chat Runtime
稳定的 ContextPacket
ContextPacket.resource_handles 空位
Context Gateway memory_selector
Memory Service
asset_scope
Asset Memory layer 占位
memory kind: asset_usage
memory kind: knowledge_fact
MemoryPolicy 对 asset_scope 的默认拒绝进入上下文规则
Trace redaction

```

第四阶段接入时，不能重写：

```text

Memory Service 写入和检索主流程
Response Composer 输出闸门
聊天页布局
ChatEvent 顶层语义
```

资产能力必须通过 Asset Broker 和 Capability Graph 接入。

## 第四阶段核心结论

| 结论 | 含义 |
|---|---|
| 资产中心负责“有什么” | 它登记资源、状态、配置、敏感等级和可见摘要 |
| Capability Graph 负责“谁能做什么” | 它给出 allow、deny、approval_required、risk_level 和原因 |
| Asset Broker 负责“怎么给模型/任务看” | 它只发放短期句柄和脱敏摘要，不发 secret |
| Secret Store 负责“敏感值在哪里” | API key、账号 token、钱包凭证、连接密钥只存 secret_ref |
| 知识库是资产，不是 prompt 文件夹 | 文件、笔记、网页先索引为片段，再经权限和检索进入上下文 |
| 大脑资产继续沿用第二阶段 | 本阶段把大脑纳入统一资产中心和授权视图，不重写模型路由 |
| 账号、钱包、硬件只完成资源层 | 可登记、验证、授权、发句柄，不执行发布、签名、控制 |
| 资产二级分类固定 | 大脑、账号、钱包、硬件、知识库不随壳改名 |
| 聊天页不暴露资产管理 | 聊天页只在需要时用自然语言说明可用资源和确认需求 |
| 所有资源感知可审计 | 资产查询、授权决策、句柄发放、知识库检索都有 trace/audit |

## 第四阶段优化后的工程抓手

| 抓手 | 要解决的问题 | 本阶段落点 |
|---|---|---|
| 资产规范化 | 五类资产字段不同但需要统一生命周期 | `assets` + 类型化 `config_json` + validator |
| Secret 隔离 | 账号、钱包、连接信息不能进模型上下文 | `secret_refs`、masked API、redaction |
| 权限图 | 成员、部门、角色、Skill、任务授权要可解释 | `capability_edges`、policy priority、decision logs |
| 句柄化 | 模型只拿摘要和 handle，不拿真实资源 | `asset_handles`、TTL、allowed_actions |
| 知识索引 | 知识库要可检索但不能全量塞 prompt | `knowledge_sources`、`knowledge_chunks`、vector refs |
| 上下文接入 | 资源摘要必须经过 Context Gateway | `resource_handles`、trusted_resource_summary |
| 审计回放 | 资源访问是安全核心，必须能追踪 | `asset.audit`、`capability.decision`、`asset.handle.issue` |
| 决策确定性 | 同样输入必须得到同样权限结果 | policy snapshot、decision_id、context_hash |
| 撤销传播 | 资产禁用或授权收紧后旧句柄不能继续用 | handle revoke、active handle sweep |

### 资源层优先级

资源访问判断必须遵循：

| 优先级 | 来源 | 规则 |
|---:|---|---|
| 1 | 系统安全策略 | 永远最高，不可被资产授权放宽 |
| 2 | 用户显式禁用 | 用户说不能用，就不能用 |
| 3 | 资产自身策略 | 钱包、账号、高敏知识库可强制确认或拒绝 |
| 4 | 成员个人授权 | 可扩大或收紧该成员能力 |
| 5 | 部门授权 | 成员可继承部门能力 |
| 6 | 角色默认授权 | 角色模板能力 |
| 7 | 组织默认授权 | 最宽泛的默认策略 |
| 8 | 记忆和偏好 | 只能影响推荐，不能放宽权限 |

示例：

```text
记忆里写“用户喜欢少问确认”，不能绕过账号发布确认。
部门授权允许草稿，但资产自身策略拒绝发布，则不能发布。
角色默认允许查询知识库，但用户显式禁用某知识库，则不能查询。
```

### 资产生命周期

所有资产必须有显式生命周期：

```text
draft
  -> configured
  -> verifying
  -> active
  -> degraded
  -> disabled
  -> archived
  -> deleted
```

状态含义：

| 状态 | 含义 |
|---|---|
| draft | 已创建但配置不完整 |
| configured | 配置完整，尚未验证 |
| verifying | 正在验证连接或可用性 |
| active | 可被授权和查询 |
| degraded | 可见但部分能力不可用 |
| disabled | 用户禁用，不可发句柄 |
| archived | 归档，不再参与默认查询 |
| deleted | 逻辑删除，保留必要审计 |

### 资源句柄生命周期

资产句柄必须是短期、可撤销、可追踪的，不是永久 token。

```text
requested
  -> issued
  -> active
  -> used
  -> expired
  -> revoked
  -> denied
```

状态含义：

| 状态 | 含义 |
|---|---|
| requested | Asset Broker 收到资源请求 |
| issued | 已创建句柄记录 |
| active | 未过期、未撤销，可被后续 Tool/Task 校验 |
| used | 已被一次执行链路消费，是否继续可用取决于策略 |
| expired | TTL 到期 |
| revoked | 资产禁用、授权收紧、用户撤销或安全策略触发 |
| denied | Capability Graph 拒绝，没有发放可用句柄 |

撤销触发：

```text
资产 disabled / archived / deleted
secret 被删除或旋转后需要重新验证
授权边从 allow 改为 deny
用户显式禁用某资产
风险策略升级
任务或会话结束导致 scoped handle 失效
```

### 资产可见性矩阵

资产有“管理可见”和“上下文可见”两种可见性，不能混为一谈。

| 场景 | 可见内容 | 禁止内容 |
|---|---|---|
| 资产管理页 | 完整脱敏配置、状态、策略、审计摘要 | secret 明文 |
| 授权管理页 | 主体、对象、动作、效果、原因 | secret 明文 |
| 决策模拟器 | allow/deny/approval、继承链、policy_sources | secret 明文 |
| ContextPacket | 相关资产摘要、短期 handle、允许动作 | 完整 config、secret、全部资产列表 |
| 聊天页 | 自然语言说明可用资源或缺失能力 | 资产表、权限图、句柄详情 |
| Tool Runtime | handle_id 和受控执行所需的最小引用 | 绕过 handle 的 secret_ref |

## 必须遵守的最终态规则

第四阶段从第一行代码开始必须遵守：

```text
资产二级分类固定：大脑、账号、钱包、硬件、知识库
资产管理一级菜单可以随壳映射，二级分类不能随壳改名
系统管理及其子菜单固定，不随壳改名
聊天页不显示资产列表、权限图、资产数量、句柄列表
模型不能直接查询 assets / asset_grants / secret_refs
模型只能通过 ContextPacket.resource_handles 看到脱敏摘要和短期句柄
secret 明文不进入数据库普通字段、prompt、trace、audit、前端响应
Asset Broker 必须调用 Capability Graph
Capability Graph 决策必须可解释
deny 优先于 allow
approval_required 不能被记忆、Skill 或模型输出绕过
钱包签名、支付、转账不在本阶段执行
账号发布、私信、资料修改不在本阶段执行
硬件控制不在本阶段执行
知识库只读取用户显式添加或授权的来源
外部网页和文档内容进入上下文时必须标记为资料，不是指令
```

### 目标

把基础 assets / asset_grants 扩展成可支撑五类资产、策略、句柄、知识库索引和审计的资源底座。

### secret_refs

```sql
CREATE TABLE secret_refs (
  secret_ref_id TEXT PRIMARY KEY,
  organization_id TEXT NOT NULL,
  ref_uri TEXT NOT NULL,
  secret_type TEXT NOT NULL,
  provider TEXT NOT NULL,
  status TEXT NOT NULL,
  metadata_json TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  rotated_at TEXT,
  expires_at TEXT
);

CREATE UNIQUE INDEX idx_secret_refs_uri ON secret_refs(ref_uri);
```

说明：

```text
secret_refs 只保存引用和元数据
真实 secret 存在本地加密存储或系统凭据管理器
前端和普通 API 永远不返回明文
```

### asset_policies

资产自身策略从 `assets.policy_json` 拆出可审计版本。


## 小阶段 4.3：Secret Store 与脱敏响应

### 目标

建立本地 secret 引用体系，保证任何账号、钱包、连接配置、模型密钥都不以明文进入普通数据流。

### Secret Store 规则

```text
写入 secret 时只返回 secret_ref
更新 secret 时生成 rotated_at
删除 secret 时 asset 进入 degraded 或 disabled
前端查询资产详情时只返回 masked_secret_summary
trace 和 audit 只记录 secret_ref_id
```

### 存储边界

Secret Store 可以先使用本地加密存储或系统凭据管理器，但服务接口必须保持稳定。

必须抽象为：

```python
class SecretStore:
    async def put(self, ref_uri: str, secret_type: str, value: SecretValue) -> SecretRef:
        ...

    async def rotate(self, ref_uri: str, value: SecretValue) -> SecretRef:
        ...

    async def delete(self, ref_uri: str) -> None:
        ...

    async def resolve_for_executor(self, ref_uri: str, purpose: str, trace_id: str) -> SecretLease:
        ...
```

规则：

```text
普通 API 不能调用 resolve_for_executor
只有受控执行器在第五阶段执行工具时才允许按 handle 和 action 解析 secret
第四阶段 verify 可以使用短期 SecretLease 做连通性检查
SecretLease 不写数据库
SecretLease 不进入 trace
SecretLease 必须有极短 TTL
```

### 威胁点

必须防住：

| 威胁 | 防护 |
|---|---|
| 前端回显 secret | masked response |
| trace 记录 secret | redaction + 禁止完整请求落 trace |
| audit 记录 secret | 只记录 secret_ref_id |
| 知识库索引读到 secret 文件 | 路径授权 + 敏感检测 + 索引阻断 |
| 句柄兑换 secret | AssetHandle 不提供 secret 解析能力 |
| 日志打印连接串 | error_summary 脱敏 |

### secret_ref URI

格式：

```text
secret://asset/{asset_id}/{purpose}
secret://brain/{brain_id}/api_key
secret://account/{asset_id}/credential
secret://wallet/{asset_id}/credential
secret://hardware/{asset_id}/connection
```

### 前端响应示例

```json
{
  "asset_id": "account.xhs_main",
  "asset_type": "account",
  "display_name": "小红书主账号",
  "secret": {
    "configured": true,
    "masked": "••••••••",
    "rotated_at": "2026-04-26T10:00:00+08:00",
    "expires_at": null
  }
}
```

### Redaction

必须覆盖：

```text

trace output_summary
audit metadata
error_summary
knowledge indexing logs
capability decision logs
```

### 验收

```text

更新 secret 后前端不回显
trace 不出现 secret 明文
audit 不出现 secret 明文
```


## 小阶段 4.6：Capability Decision Engine

### 目标

实现可解释、可测试、可审计的权限决策。

### 输入

```json
{
  "subject": {"type": "member", "id": "member.mobai"},
  "object": {"type": "asset", "id": "account.xiaohongshu_main"},
  "action": "publish_post",
  "context": {
    "conversation_id": "conv_001",
    "task_id": null,
    "skill_id": null,
    "risk_hint": "R4",
    "client_time": "2026-04-26T10:00:00+08:00"
  }
}
```

### 输出

```json
{
  "decision_id": "dec_001",
  "allowed": true,
  "decision": "approval_required",
  "risk_level": "R4",
  "approval_required": true,
  "reason": "账号发布属于外部发布动作，需要用户确认。",
  "policy_sources": [
    "system.external_post",
    "asset.account.xiaohongshu_main",
    "member.member.mobai"
  ]
}
```

### 决策语义

Decision Engine 输出必须区分三个概念：

| 字段 | 含义 |
|---|---|
| allowed | 是否存在继续推进的可能 |
| decision | allow、deny、approval_required 中的最终结论 |
| approval_required | 是否必须进入审批 |

语义规则：

```text
decision = deny 时 allowed = false
decision = approval_required 时 allowed = true，但不得执行真实动作
decision = allow 时 allowed = true，可继续发放允许动作的句柄
approval_required 只能被 Approval Flow 消解，不能被模型文字消解
```

### Policy Snapshot

每次决策必须记录当时参与计算的策略快照摘要，避免未来策略变化后无法解释历史结果。

```json
{
  "policy_snapshot": {
    "system_policy_hash": "sha256:...",
    "asset_policy_ids": ["pol_001"],
    "capability_edge_ids": ["edge_001", "edge_002"],
    "ignored_edge_ids": ["edge_expired_001"],
    "context_hash": "sha256:..."
  }
}
```

不得记录 context 原文中的敏感内容，只记录 hash 和脱敏摘要。

### 决策算法

```text
1. 加载系统安全策略
2. 加载资产自身策略
3. 加载用户显式禁用策略
4. 展开 subject 继承：member -> role / department / organization
5. 加载 subject 直接授权
6. 加载 task / skill 临时上下文授权
7. 过滤过期、禁用、不匹配条件的边
8. deny 优先
9. approval_required 次优先
10. allow 最后生效
11. 合并最高 risk_level
12. 生成 reason 和 policy_sources
13. 生成 policy_snapshot
14. 写 decision log 和 trace
```

### 冲突处理

| 冲突 | 结果 |
|---|---|
| allow + deny | deny |
| allow + approval_required | approval_required |
| role allow + member deny | deny |
| department allow + asset deny | deny |
| task temporary allow + system deny | deny |
| memory preference + policy deny | deny |

### 验收

```text
成员继承部门授权后可 draft_post
成员个人 deny 后不可 draft_post
publish_post 返回 approval_required
钱包 sign_transaction 返回 deny 或强确认策略
过期授权不生效
每次决策有 capability_decision_logs
decision 输出包含 reason 和 policy_sources
decision log 包含 policy_snapshot
approval_required 不会被当作 allow 执行动作
```

### 发放流程

```text
1. 解析资产查询
2. 按 asset_type、status、visibility、keywords 过滤候选
3. 对每个候选调用 Capability Graph
4. 过滤 deny
5. 合并 allowed_actions、blocked_actions、approval_required_actions
6. 生成脱敏 summary
7. 创建短期 asset_handle
8. 写 trace: asset.query / capability.decision / asset.handle.issue
9. 返回句柄给 Context Gateway 或调用方
```

### 查询幂等

同一次 turn 内，相同资源查询不应重复制造大量句柄。

幂等 key：

```text
organization_id
subject_type
subject_id
conversation_id
task_id
asset_query canonical hash
requested_actions hash
policy_snapshot hash
```

规则：

```text
同 key 且 active handle 未过期时可复用
策略变化后 policy_snapshot hash 变化，必须重新决策
资产禁用后即使 key 相同也不得复用
高风险句柄默认不跨 turn 复用
```

### 句柄 TTL

默认：

| 场景 | TTL |
|---|---:|
| 聊天资源摘要 | 30 分钟 |
| 单个 turn 查询 | 10 分钟 |
| 任务执行预留 | 2 小时 |
| 高风险账号动作 | 10 分钟 |
| 钱包相关 | 5 分钟 |

过期后：

```text
Tool / Task / Skill 必须重新申请
Context Gateway 不得复用过期句柄
asset_handles.status 更新为 expired
```

### 撤销传播

以下事件必须撤销相关 active handles：

```text
资产 disabled / archived / deleted
secret deleted
授权边删除或改成 deny
资产 policy 改成 deny
用户显式撤销某成员授权
知识库来源被禁用
```

撤销流程：

```text
查询匹配 active handles
更新 asset_handles.status = revoked
写 asset_handle_events.revoked
写 audit: asset.handle.revoked
后续 validate_handle 返回 ASSET_HANDLE_INVALID
```

### AssetHandle 使用限制

```text
句柄不能兑换 secret
句柄只能用于 allowed_actions
blocked_actions 必须显式保留
approval_required_actions 未来必须进入 Approval Service
句柄绑定 subject 和上下文
句柄跨 member 不可复用
句柄跨 organization 不可复用
句柄跨高风险 action 不可扩权
句柄不能新增 allowed_actions
```

### 验收

```text
Asset Broker 查询账号只返回脱敏句柄
没有授权时返回 ASSET_ACCESS_DENIED

```

## 小阶段 4.8：知识库资产索引与检索

### 目标

把知识库做成授权资源，而不是把文件夹直接塞进模型。

知识库检索必须经过：

```text
Knowledge Asset
  -> Knowledge Source
  -> Index Job
  -> Chunk
  -> FTS / Vector
  -> Capability Graph
  -> Asset Broker
  -> Context Gateway
```

### Indexer 目录

```text
services/knowledge/
  indexer.py
  source_reader.py
  chunker.py
  summarizer.py
  retriever.py
  vector_index.py
  fts.py
  injection_guard.py
```

### 支持来源

本阶段支持：

```text
markdown
txt
pdf 文本层
本地文件夹
单个网页 URL
```

暂不支持：

```text

邮箱
网盘
数据库
私有 SaaS
```

这些来源留给后续插件、MCP、任务执行接入。

### 索引规则

```text
只索引用户显式添加的路径或 URL
本地路径必须在授权 root_uri 内
单个文件过大要分块
每个 chunk 记录 source_id、chunk_index、content_hash
外部网页内容标记 untrusted_external_content
高敏来源默认不向量化或需要用户确认
索引失败不影响聊天主链路
```

### 索引生命周期

知识库来源和索引任务必须有可观察状态：

```text
source_added
  -> scanning
  -> extracting
  -> chunking
  -> summarizing
  -> indexing_fts
  -> indexing_vector
  -> indexed
  -> degraded
  -> disabled
```

状态说明：

| 状态 | 含义 |
|---|---|
| scanning | 扫描来源和元数据 |
| extracting | 提取文本 |
| chunking | 切片并生成 hash |
| summarizing | 生成摘要 |
| indexing_fts | 同步 FTS |
| indexing_vector | 同步向量 |
| indexed | 可检索 |
| degraded | 部分片段或向量失败，FTS 可用 |
| disabled | 来源禁用，不可检索 |

### 路径与来源安全

本地来源必须做路径约束：

```text
解析真实路径
禁止 .. 逃逸 root_uri
禁止跟随未授权符号链接
禁止索引系统敏感目录
禁止索引 secret store 文件
禁止索引隐藏凭证文件
```

网页来源必须记录：

```text
url
fetch_time
content_hash
content_type
trusted = false
```

### Chunk 规则

默认：

```text
chunk token 目标：500-900
chunk token 上限：1200
chunk overlap：80-120
保留标题路径
保留 source_id 和 chunk_index
大文件按 hash 做增量更新
```

### 验收

```text

```

### 资源上下文预算

资源摘要也要受 token budget 控制，不能因为用户有很多资产就污染上下文。

默认预算：

| 内容 | 上限 |
|---|---:|
| resource_handles | 5 个 |
| 每个资产摘要 | 120 tokens |
| knowledge chunks | 6 个 |
| 单个知识片段摘要 | 180 tokens |
| capabilities 摘要 | 400 tokens |

裁剪顺序：

1. 当前用户明确点名的资产。
2. 与 intent 直接相关的知识库。
3. allowed 且低风险资源。
4. approval_required 资源摘要。
5. 低相关资产摘要。

默认不注入：

```text
blocked only 的资产
disabled / archived 资产
高敏且未被明确请求的资产
没有 source 的知识片段
过期句柄
```

### 用户可见表达

Response Composer 可用自然语言说明资源状态，但不能暴露内部句柄。

允许：

```text
我可以使用“项目设计文档知识库”来参考已有设计。
“小红书主账号”可以用于生成草稿，但发布需要你确认。
当前成员没有访问这个知识库的授权。
```

禁止：

```text

policy_sources 包括 edge_001
这个账号的 secret_ref 是 secret://...
```

### 验收

```text
asset_query 可以返回授权资产摘要
未授权资产不会进入 ContextPacket
ContextPacket.resource_handles 只包含句柄和摘要
知识库检索结果可进入上下文
账号、钱包、硬件只进入摘要，不执行动作
聊天页不展示资产管理面板
资源摘要受 token budget 控制
聊天回复不暴露 handle_id 和 secret_ref
```

## 小阶段 4.14：资产权限评测与质量门禁

### 目标

### 指标

|---|---|---:|

| CGE | Capability Graph Explainability，决策可解释率 | 0.95 |
| KRS | Knowledge Retrieval Safety，知识库授权过滤 | 0.95 |
| PIS | Prompt Injection Safety，知识库注入防护 | 1.00 |

### 评测用例

```text

EVAL-ASSET-007：disabled 资产不返回句柄
EVAL-ASSET-008：过期句柄不可使用
EVAL-ASSET-009：未授权知识库不能检索

EVAL-ASSET-014：知识库路径逃逸不会被索引
EVAL-ASSET-015：Response Composer 不暴露 handle_id 和 secret_ref
EVAL-ASSET-016：approval_required 不会被当作 allow 执行动作
```

### 单元测试

| 模块 | 测试重点 |

| Asset validators | 五类资产字段、状态机、风险策略 |

## 第四阶段总体验收流程

阶段完成后，必须能完整跑通：

```text
1. 启动应用
2. 进入资产中心
3. 创建账号资产，填写 secret
4. 确认资产详情不回显 secret
5. 给某成员授权 read_profile / draft_post
6. 使用决策模拟器检查 publish_post，结果为 approval_required
7. Asset Broker 查询账号，返回脱敏 handle
8. 禁用账号资产
9. 再次查询，返回 ASSET_DISABLED 或 ASSET_ACCESS_DENIED
10. 创建知识库资产并添加来源
11. 触发索引
12. 使用授权成员搜索知识库
13. 使用未授权成员搜索知识库，返回拒绝
14. 检查 ContextPacket.resource_handles 只含摘要和句柄
15. 检查 trace 和 audit
16. 检查聊天页没有资产管理面板
```

成功标准：

```text
五类资产可管理
secret 不泄露
Capability Graph 可解释
Asset Broker 只发短期句柄
知识库可授权检索
Context Gateway 能注入资源摘要
高风险动作只产生确认策略，不执行
聊天页保持极简
```

## 第四阶段禁止通过验收的情况

出现以下任一情况，本阶段不能通过：

```text
资产二级分类随壳改名
聊天页显示资产列表、权限图或资产数量
模型直接查询 assets 表
模型看到 secret 明文
前端 API 回显 secret
Asset Broker 绕过 Capability Graph
Capability Graph 决策没有 reason
deny 没有优先于 allow
disabled 资产仍可发句柄
过期句柄仍可使用
钱包签名被执行
账号发布被执行
硬件控制被执行
知识库未授权也能检索
知识库注入内容变成系统指令
资产记忆绕过 Asset Broker 进入 ContextPacket
```

## 第四阶段风险与缓解

| 风险 | 后果 | 缓解 |
|---|---|---|
| 资产和工具混在一起 | 权限边界不清，后续执行危险 | 本阶段只做资源层和句柄，不执行动作 |
| secret 存错地方 | 泄露到 prompt、trace、前端 | Secret Store、masked response、redaction 测试 |
| 授权图过度复杂 | 用户看不懂为什么允许或拒绝 | 决策模拟器、reason、policy_sources |
| deny 不生效 | 越权访问资产 | deny 优先、评测门槛 ADS = 1.00 |
| 知识库全量注入 | prompt 污染和泄露 | chunk、检索、摘要、token budget |
| 知识库注入攻击 | 模型被资料指令劫持 | untrusted 标记、Context Gateway 分区 |
| 句柄过期不严格 | 后续工具复用旧权限 | TTL、validate_handle、expired 状态 |
| UI 过重 | 聊天页变后台 | 资产管理只在管理页 |

## 与第五阶段的衔接

第五阶段将实现：

```text
Task Engine
Tool Runtime
文件工具
浏览器工具
终端工具
账号草稿动作
硬件查询或低风险控制
Approval Flow
任务回放
```

第四阶段必须保证第五阶段接入时：

```text
工具执行只需要申请和校验 AssetHandle
任务计划只需要调用 CapabilityGraph.decide
高风险动作能根据 approval_policy 生成确认
知识库检索可作为 read_knowledge 工具能力使用
Trace 已能串起 asset.handle.issue 和未来 tool.call
```

第五阶段只应该把“动作执行”接进已经存在的资源和权限底座，而不是重新设计资产中心。
