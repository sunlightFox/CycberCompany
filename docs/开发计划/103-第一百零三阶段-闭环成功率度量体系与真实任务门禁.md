# 第一百零三阶段 - 闭环成功率度量体系与真实任务门禁

## 阶段定位

前面阶段更多回答的是：

```text
有没有这个模块
有没有这条链路
有没有这个测试
```

第一百零三阶段要把判断标准进一步升级成：

**它在真实任务里，闭环成功率到底怎样？**

也就是说，项目要从“模块存在性思维”切换到“任务交付率思维”。

## 阶段目标

```text
建立闭环成功率口径，而不是只统计单项 case pass
把真实任务集引入 release gate
固定追踪一次成功率、人工接管率、审批打断率、错误恢复率、最终可交付率
让不同做事域都能复用统一任务级评测框架
把“做了很多”与“真正交付”区分开
```

## 核心问题

现在项目虽然有很多 phase 测试，但还缺一类更接近真实代理成熟度的指标：

```text
完成一个真实任务要经过多少轮
有多少任务中途需要人工接管
有多少任务被审批正确打断
有多少失败能被系统自己修回来
最终真正可交付的比例是多少
```

如果没有这套指标，就容易出现：

```text
模块很多
测试不少
但真实做事体验仍然不稳
```

## 本阶段范围

### 必须完成

```text
定义统一 task closure metrics contract
建立真实任务集和任务分层
给多执行域统一接入闭环成功率门禁
让 release / readiness / diagnostics 能输出任务级成熟度
形成跨执行域 scorecard，而不是只看单域 aggregate
形成可持续跟踪的周报或批次报告格式
```

### 明确不做

```text
不把纯单元测试通过率伪装成闭环成功率
不把审批阻断误算为系统失败
不为追求指标好看而删掉诚实中断、审批等待、人工接管记录
```

## 主要指标

### 一级指标

```text
一次成功率
人工接管率
审批打断率
错误恢复率
最终可交付率
```

### 二级补充指标

```text
平均完成轮数
平均工具调用数
replan 发生率
stop_reason 分布
untrusted observation 触发率
residual risk 留存率
```

## 指标口径约束

### 一次成功率

定义：

```text
任务在不需要人工接管、不过度重试、无未解决失败的情况下直接完成
```

### 人工接管率

定义：

```text
任务执行过程中需要用户或人工明确介入才能继续推进的比例
```

### 审批打断率

定义：

```text
因 Safety / Approval 正常进入等待确认的比例
```

说明：

```text
这不是坏指标
它是做事域风险结构的一部分
```

### 错误恢复率

定义：

```text
任务首次失败后，系统通过局部重试、重规划或替代路径恢复并最终完成的比例
```

### 最终可交付率

定义：

```text
任务结束时存在可交付结果，且结果没有被未解释的失败或未验证状态污染的比例
```

## 实施拆解

### 103.1 任务级闭环指标契约

目标：

```text
为每个任务定义统一 closure record
```

交付：

```text
task_closure_record
delivery_status
delivery_blockers
handoff_reason
approval_interruption
recovery_summary
verification_status
```

### 103.2 多执行域真实任务集与任务分层

目标：

```text
把闭环评估建立在真实任务集上，而不是抽象 smoke case 上
```

交付：

```text
L1 轻任务
L2 中等闭环任务
L3 多轮恢复任务
L4 高风险审批任务
```

优先落地：

```text
代码仓执行域 20 个真实任务
Git托管协作域 12 个真实任务
内容平台 Skill 域 12 个真实任务
办公生产力 Skill 域 12 个真实任务
音频工作流域 10 个真实任务
视频工作流域 10 个真实任务
```

要求：

```text
任务集必须按执行域分层，不混成一个总分
每个执行域都要有轻任务、恢复任务、审批任务样本
```

### 103.3 release gate 接入

目标：

```text
让闭环成功率成为真正门禁，而不只是报告附件
```

交付：

```text
release closure report
closure threshold config
per-domain scorecard
blocking reasons
```

要求：

```text
低于门槛时给出具体掉分原因
不要只报一个 aggregate score
scorecard 至少覆盖 repo_local / code_hosting / content_platform / office_productivity / audio_workflow / video_workflow
```

### 103.4 长期趋势与回归告警

目标：

```text
让闭环能力退化能被持续观察
```

交付：

```text
weekly closure trend
domain-by-domain regression delta
approval / handoff / recovery drift report
```

### 103.5 真实任务人工复核样本

目标：

```text
防止系统用“形式上完成”骗过指标
```

交付：

```text
人工 spot check 样本池
可交付性人工复核模板
closure false-positive 追踪
```

### 本阶段新增测试重点

```text
任务 closure record 字段完整
审批等待不会被误算成 completed
恢复成功与恢复失败能被区分统计
“改了但没验证”不会被算作最终可交付
真实任务集结果能稳定输出 per-domain scorecard
不同执行域可以横向比较，但不会把不同风险结构粗暴混成一个结论
```

## 完成定义

```text
项目建立了统一的闭环成功率度量体系
真实任务而不是模块清单成为做事能力主门禁
repo_local、code_hosting、content_platform、office_productivity、audio_workflow、video_workflow 都拥有统一 closure gate 口径
后续任何新做事域都必须先回答自己的闭环成功率，而不是只回答有没有模块
```
