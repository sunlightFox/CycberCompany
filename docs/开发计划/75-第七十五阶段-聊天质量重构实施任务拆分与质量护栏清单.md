# 第七十五阶段 - 聊天质量重构实施任务拆分与质量护栏清单

## 阶段定位

本阶段不是新增一个聊天能力点，而是把第六十九到第七十四阶段的聊天质量重构真正转成可执行实施清单，并且明确：

```text
每一批先动哪些文件
每一批只允许接管什么
每一批绝对不能动什么
如何验证没有把聊天质量越改越差
什么时候才允许删旧逻辑
```

本阶段是整个聊天质量重构的“施工总控文档”，用于约束后续实现顺序、回归护栏和删除时机。

## 一、总目标

这一阶段完成后，不要求聊天质量重构已经全部完成，但必须做到：

```text
聊天质量重构有明确的最小落地顺序
每一步都有“旁路观测 -> 建议模式 -> 小范围接管 -> 主路径接管”的策略
不会因为文件拆分而先把闲聊、深聊、多轮、记忆、工具回复改差
不会因为新中层引入而先破坏当前执行诚实性
不会因为追求架构整洁，把真实回复改成更像系统回执
```

## 二、必须守住的聊天质量底线

后续任何代码提交，只要触发以下任一回退，就视为本阶段实施失败：

### 1. 当前消息优先回退

表现为：

```text
用户改口后仍沿用旧目标
第二句明明在追问当前问题，却被旧任务或旧记忆带偏
聊天回复开始像在响应系统状态，而不是当前用户输入
```

### 2. 执行诚实性回退

表现为：

```text
未执行说成已完成
等待确认说成已继续
工具失败说成处理好了
浏览器卡在登录或验证码时仍假装拿到结果
```

### 3. 自然度回退

表现为：

```text
更频繁出现模板开头
更频繁出现“系统状态报告味”
更像操作系统提示，而不是在聊天
长回复更工整但更空
```

### 4. 多轮承接回退

表现为：

```text
第二轮像重新开始
记不住刚刚的焦点
只记住事实，不记得用户此刻在问什么
```

### 5. 边界回复回退

表现为：

```text
隐私/审批/能力边界回复更冷、更硬、更像报错
只说不能做，不给替代路径
```

## 三、实施总策略

本次重构只允许采用如下四段式推进：

### 第 1 段：Shadow 旁路观测

只新增新模块，不接管最终结果。

允许：

```text
读取当前 turn
产出理解结果
产出状态结果
写 trace metadata
写调试 payload
```

禁止：

```text
改变最终回复文本
改变审批行为
改变 task / tool / browser 实际执行顺序
```

### 第 2 段：Advisory 建议模式

允许新模块给旧链路提供建议，但旧链路仍拍板。

允许：

```text
建议 route
建议回复风格
建议 continuation 是否启用
建议是否先澄清
```

禁止：

```text
直接替换旧 route
直接改写用户可见长文案
```

### 第 3 段：Soft Control 小范围接管

只接管低风险聊天场景。

建议先接管：

```text
闲聊
深聊
非工具问答
记忆更新/纠错回复
```

暂不接管：

```text
审批
高风险工具
浏览器执行
系统命令
多渠道复杂交互
```

### 第 4 段：Primary 主路径接管

在已有质量对照证据后，再把新中层正式接入主路径。

## 四、实施批次划分

### 批次 A：只建立类型与骨架

目标：

```text
建立核心中层文件
建立 schema / dataclass / service 接口
不改变主链路回复
```

#### 新增文件

```text
apps/local-api/app/services/conversation_understanding.py
apps/local-api/app/services/dialogue_state.py
apps/local-api/app/services/response_policy.py
apps/local-api/app/services/context_session.py
apps/local-api/app/services/context_visibility.py
apps/local-api/app/services/context_budget.py
apps/local-api/app/services/action_dialogue_mapper.py
apps/local-api/app/schemas/chat_quality.py
apps/local-api/app/schemas/context_runtime.py
```

#### 本批次只允许实现

```text
基础类型
空实现或最小实现
trace-only 输出
最小单测
```

#### 本批次绝对不允许

```text
接管 chat.py 最终回复生成
替换 chat_continuation.py 逻辑

```

#### 出口验收

```text
新文件存在
类型可导入
无主链路行为变化
trace 中能看到 shadow 结果
```

### 批次 B：Conversation Understanding 旁路接入

目标：

```text
让 chat.py 能调用 ConversationUnderstanding
但只写 metadata，不拍板
```

#### 影响文件

```text
apps/local-api/app/services/chat.py
apps/local-api/app/services/conversation_understanding.py
apps/local-api/app/services/chat_intent_router.py
```

#### 本批次只允许接入的位置

```text
turn 上下文准备后
真正 route 分支前
continuation 决策前
```

#### 本批次输出内容

建议写入：

```text
conversation_mode
user_goal
emotional_state
relationship_expectation
current_turn_priority
must_not_do
confidence
reason_codes
```

#### 本批次绝对不允许

```text
直接替换 ChatIntentRouter 的最终结果
直接改 reply 文本
直接切 task / tool 路径
```

#### 出口验收

```text
shadow 结果可被 trace / payload 观察
现有回复无明显变化
对 5 类典型输入的理解结果基本合理
```

### 批次 C：Dialogue State 旁路接入

目标：

```text
把“当前聊天状态”从散落 working state 提炼成统一结构
仍然不改变主路径输出
```

#### 影响文件

```text
apps/local-api/app/services/chat.py
apps/local-api/app/services/dialogue_state.py
apps/local-api/app/services/natural_chat.py
apps/local-api/app/services/context_gateway.py
```

#### 重点字段

```text
identity_state
relationship_state
conversation_state
action_state
memory_state
session_state
```

#### 本批次重点观测问题

必须确认：

```text
pending action 是否识别正确
当前话题是否识别正确
用户是否改口是否识别正确
是否错误把执行状态当成对话主状态
```

#### 本批次绝对不允许

```text
直接让 DialogueState 决定最终 route
直接让 DialogueState 覆盖记忆检索结果
```

#### 出口验收

```text
多轮对话 shadow state 合理
pending action 场景 shadow state 合理
无真实回复回退
```

### 批次 D：Response Policy 建议模式接入

目标：

```text
让新中层开始建议“怎么说”
但仍不完全接管 reply
```

#### 影响文件

```text
apps/local-api/app/services/response_policy.py
apps/local-api/app/services/chat.py
apps/local-api/app/services/chat_quality.py
apps/local-api/app/services/chat_continuation.py
apps/local-api/app/services/chat_response.py
```

#### 可建议的内容

```text
回复长度
结构程度
是否先回应情绪
是否先澄清
是否更适合短答
是否该启用 continuation
```

#### 暂不接管的内容

```text
审批类最终文本
任务状态类最终文本
工具结果类最终文本
严格格式的最终保护
```

#### 本批次最大风险

风险不是“报错”，而是：

```text
更会写模板句
更喜欢先结构化，导致聊天感下降
```

所以必须加额外断言：

```text
短闲聊不要被结构化
情绪类输入不要被变成分析报告
严格格式不要被“自然化”
```

#### 出口验收

```text
ResponsePolicy 可稳定输出 advisory 结果
闲聊/深聊质量不退化
严格格式场景不退化
```

### 批次 E：Action Dialogue Mapper 旁路接入

目标：

```text
先不替换工具结果表达
只把 machine state -> dialogue facts 输出到 trace
```

#### 影响文件

```text
apps/local-api/app/services/action_dialogue_mapper.py
apps/local-api/app/services/tools.py
apps/local-api/app/services/browser_workflows.py
apps/local-api/app/services/natural_chat.py
apps/local-api/app/services/chat.py
```

#### 先接的场景

优先顺序：

```text
approval pending
terminal readonly success
tool blocked
browser login required
task failed
```

#### 本批次绝对不允许

```text
直接替换所有 action reply
在审批场景改掉当前 approve / deny 真实逻辑
```

#### 必查风险

如果出现以下任一现象，应立即停止接管：

```text
blocked 被表达成 completed
pending 被表达成 in progress 但像已开始执行
browser login required 被表达成已经拿到页面内容
tool failed 被写成“我已经处理好了，只是...”这类假完成口吻
```

#### 出口验收

```text
ActionDialogueFacts 在 trace 中稳定可见
能清晰区分 pending/running/blocked/completed/failed
未改坏真实对外回复
```

### 批次 F：SessionContext / Visibility / Budget 接入

目标：

```text
提升多轮、多渠道、引用/转发、workbench/multimodal 场景的上下文质量
但先不大拆 context_gateway.py
```

#### 影响文件

```text
apps/local-api/app/services/context_gateway.py
apps/local-api/app/services/context_session.py
apps/local-api/app/services/context_visibility.py
apps/local-api/app/services/context_budget.py
apps/local-api/app/services/chat_ingress.py
apps/local-api/app/services/channel_connectors.py
```

#### 接入顺序

```text
先 Budget
再 Visibility
最后 SessionContext
```

#### 为什么这样排

因为最容易把聊天质量搞差的，不是 session 字段少，而是：

```text
上下文选错
预算裁错
当前消息被旧内容挤压
不可信内容混进主上下文
```

#### 本批次绝对不允许

```text
先把 context_gateway.py 打散再慢慢补预算/可见性
```

那样最容易直接把多轮质量打坏。

#### 出口验收

```text
当前消息仍稳居最高优先级
引用/转发/工具结果默认不可信
同 session / 同 thread 过滤更稳定
多轮质量不退化
```

### 批次 G：低风险主路径接管

目标：

```text
只接管闲聊、深聊、普通问答、记忆更新/纠错
不接高风险执行场景
```

#### 本批次允许正式接管

```text
ConversationUnderstanding
DialogueState
ResponsePolicy
```

#### 本批次不允许正式接管

```text
审批类
工具执行类
浏览器执行类
系统命令类
复杂渠道交互类
```

#### 出口验收

```text
闲聊更自然
深聊不更模板
多轮承接不回退
当前消息优先不回退
```

### 批次 H：执行语义主路径接管

目标：

```text
让 Action Dialogue Mapper 正式接管审批/工具/浏览器/系统命令的人话表达
```

#### 接管顺序

建议：

```text
approval pending
tool blocked
terminal readonly
task failed/completed
browser login required / challenge_detected
高风险动作等待确认
```

不要上来就全接。

#### 出口验收

```text
执行诚实性无回退
边界自然度不回退
动作状态表达更像人话而不是回执
```

### 批次 I：主文件拆分与旧逻辑缩减

前面都稳定后，才进入：

```text
chat.py 拆分
tools.py 拆分
browser_workflows.py 拆分
channel_connectors.py 拆分
```

这一批的本质不是改行为，而是：

```text
把已经稳定的新逻辑搬到正确文件边界
```

## 五、每个批次的质量检查清单

### 1. 必跑场景

每个批次至少要跑以下 10 个核心场景：

| 场景 | 目的 |
|---|---|
| 闲聊 | 看是否更系统化 |
| 深聊 | 看是否更模板化 |
| 多轮承接 | 看是否忘上下文 |
| 用户改口 | 看当前消息优先是否回退 |
| 记忆更新 | 看是否更机械 |
| 记忆纠错 | 看是否嘴硬或沿用旧记忆 |
| 工具只读 | 看是否假装已执行 |
| 审批等待 | 看是否错误放行 |
| 浏览器需登录 | 看是否假完成 |
| 系统命令失败 | 看是否假成功 |

### 2. 必查文案坏味道

重构期间只要这些显著增加，就说明方向错了：

```text
“当前状态如下”
“处理结果如下”
“我先帮你分析一下”
“下面我从三个方面说”
“作为 AI”
“根据系统策略”
```

注意：不是要把这些词简单 rg 掉，而是要看**是否更频繁、更默认地出现**。

### 3. 必查执行坏味道

```text
“已经” 用得过早
“处理好了” 用得过早
“我看过了” 但实际上没访问成功
“我已经继续了” 但实际上还在等确认
```

## 六、建议补充的 feature flag

建议新增配置项：

```text
chat_quality.enable_conversation_understanding_shadow
chat_quality.enable_dialogue_state_shadow
chat_quality.enable_response_policy_advisory
chat_quality.enable_response_policy_primary_low_risk
```

## 八、删除旧逻辑的条件

旧逻辑不允许因为“看起来重复”就先删，必须同时满足：

### 1. 语义等价

```text
新路径已覆盖旧路径场景
structured payload 没丢
trace 没丢
approval / safety / memory / task 语义没丢
```

### 2. 质量不退化

以下全守住：

```text
闲聊自然度
深聊质量
多轮承接
执行诚实性
边界自然度
```

### 3. 已经过至少两轮质量回归

```text
单元回归一轮
主链路体验回归一轮
```

