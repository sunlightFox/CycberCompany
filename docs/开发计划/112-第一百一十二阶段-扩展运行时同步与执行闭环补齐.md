# 第一百一十二阶段 - 扩展运行时同步与执行闭环补齐

## 阶段定位

phase101 已经把 OpenClaw 风格 skill / plugin / MCP 兼容层搭了起来，也有 `preview-import`、`install`、`bind`、`diagnostics` 这些后端接口。

但从 phase103 的执行域 scorecard 看，扩展生态目前仍存在：

```text
extension_runtime_sync_missing
runtime_contribution_count 偏低
diagnostics_ready_count 偏低
final_deliverable_rate 为 0
```

也就是说，扩展已经“能导入”，但还没有稳定形成“装进去就能真实参与运行时”的闭环。

## 目标

```text
补齐扩展安装、启用、绑定、运行时贡献、诊断之间的同步闭环
让 OpenClaw 兼容扩展真正进入 skill / mcp / external platform 主链路
让 extension_ecosystem 域从存在性通过，转成可执行闭环通过
```

## 重点

### 112.1 扩展状态机统一

```text
统一 imported / installed / enabled / bound / ready / degraded 的语义
避免 bundle、skill、mcp server、extension runtime 各自维护不一致状态
```

### 112.2 运行时贡献同步

```text
扩展启用后，skill / mcp / external platform / channel contribution 要能稳定同步到运行时注册表
扩展停用、降级、回滚后，运行时贡献也要同步撤出
防止数据库状态已更新，但运行时仍是旧快照
```

### 112.3 诊断与执行闭环

```text
diagnostics 要能解释为什么扩展 ready / degraded / unrunnable
plan-run、bind、compatibility、runtime diagnostics 使用同一份贡献快照
让扩展域真实任务能形成 deliverable proof，而不是停在 install/bind 成功
```
