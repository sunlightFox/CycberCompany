# 第一阶段：最终态工程底座与产品骨架

## 阶段定位

第一阶段不是原型阶段，也不是临时 demo 阶段。它是最终产品的工程地基阶段。

这一阶段要解决的问题不是“先随便跑起来”，而是：

```text
最终产品的目录结构怎么定
最终产品的核心对象怎么命名
最终产品的本地运行方式怎么启动
最终产品的壳系统怎么加载
最终产品的聊天页边界怎么立住
最终产品的主链路契约怎么预留
最终产品的 trace / config / migration / tests 怎么从第一天存在
```

总设计和聊天主链路设计反复强调：这个产品真正的质量不取决于单次 prompt，而取决于 Context Gateway、Memory、Capability Graph、Task Engine、Safety、Response Composer、Trace 这些中间层是否从一开始就有正确位置。

所以第一阶段的核心不是实现完整智能体能力，而是把后续完整聊天主链路所需的**接口、目录、表、页面入口、事件入口、边界约束**全部立住。

## 第一阶段优化后的核心结论

相比原计划，第一阶段必须加强四件事：

| 加强点 | 原因 |
|---|---|
| 链路契约前置 | 第二阶段接入真实聊天时，不能再临时发明 `ChatTurnRequest`、`ChatEvent`、`ContextPacket` |
| 模块目录前置 | Brain、Heart、Persona、Memory、Tools、Safety 等目录必须从第一天存在，避免后续横向搬家 |
| 模型/大脑基础表前置 | 创建默认成员时已经有“大脑”概念，不能等模型阶段才补最基础表 |
| trace 语义前置 | 后续模型调用、工具调用、审批、记忆写入都要接同一套 span，不允许变成多套日志 |

第一阶段仍然不实现完整模型聊天、不实现完整记忆、不实现任务执行，但要把它们的**最终态接入点**留下。

## 必须遵守的最终态规则

本阶段从第一行代码开始就必须遵守：

1. 核心对象使用 `Organization / Member / Department / Role / Shell / Asset / Skill / Task`，不在核心层写死公司壳概念。
2. 首发加载公司壳，但壳只影响菜单、标签、模板、文案和主题，不自动修改底层数据值。
3. 聊天页顶部只显示当前聊天对象的人名、头像和状态，不显示组织、壳、部门、岗位、资产归属。
4. 系统管理一级菜单和二级菜单固定，不随壳变化。
5. 资产管理一级菜单允许后续随壳映射，资产二级分类固定为大脑、账号、钱包、硬件、知识库。
6. 智能体默认独立，不天然感知组织、其他成员、所有资产和系统秘密。
7. 所有 API schema、数据库表、前端类型都按最终态命名，不使用临时 mock 字段污染契约。
8. 所有可执行链路从第一阶段就预留 trace，不允许后续再补一个旁路日志系统。
9. 本地单机优先，任何云模型、MCP、插件都只能作为可选配置接入。
10. 阶段只是交付顺序，不代表产品有 v1/v2/临时版。
