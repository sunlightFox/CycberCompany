# 第四十四阶段 - 阶段回归门禁与迁移哨兵修复

## 阶段背景

第三十五到第四十二阶段已经形成后端契约、service、API、migration、tests 和 release summary。第四十三阶段正在开发媒体能力，因此仓库最新 migration、测试矩阵和阶段报告会继续前进。当前暴露出的一个典型问题是：旧阶段测试仍断言“最新 migration 必须是 030”，当第四十三阶段加入 `031_media_runtime.sql` 后，35-42 的阶段回归会被误判为失败。

第四十四阶段专门处理这种“阶段间进度差异”导致的验收噪声，把测试从“固定最新文件名”升级为“阶段契约语义检查”。本阶段不新增业务能力，只修正回归门禁、release 证据、diagnostic 和阶段验收语义。

## 核心目标

本阶段完成后，应满足：

```text
35-43 阶段测试允许后续 migration 存在
每个阶段只检查自己需要的表、schema、runtime contract、eval suite 和 release evidence
release report 能准确表达阶段完成、开发中、accepted risk 和后续阶段覆盖关系
diagnostic bundle 不因为新阶段加入而破坏旧阶段判断
测试失败能指向真实能力缺口，而不是进度哨兵过期
```

## 阶段原则

1. 回归门禁检查语义契约，不检查“当前目录最后一个文件名”。
2. 阶段测试可以知道后续阶段存在，但不能要求后续阶段完成。
3. migration 验收以“所需 migration 已存在且表结构正确”为准。
4. release summary 必须区分 `implemented`、`in_progress`、`accepted_risk`、`blocked`。
5. 不为了让测试通过隐藏真实缺口。
6. 不删除第四十三阶段已有迁移、文件或测试。
7. 不新增 UI。

## 本阶段必须完成

```text
替换 phase35-42 测试里的 latest_migration 固定断言
新增 migration contract helper：assert_required_migration_at_least / assert_tables_exist
更新 test_db_migrations 覆盖 031 及后续迁移的兼容关系
release report 增加 per_phase migration_contract 字段
diagnostic bundle 增加 current_latest_migration 与 phase_required_migrations
eval suite 注册检查改为按 suite_id 和 case_key 前缀
阶段测试统一使用 helper，避免每个文件复制 _latest_migration
```

## 本阶段不做

```text
不改 35-43 的业务逻辑
不新增新的后端能力表
不把失败用例标成通过
不降低 secret、DLP、approval、trace 检查
不清空 data 或重置测试库
```

## 小阶段总览

| 小阶段 | 名称 | 核心交付 |
|---:|---|---|
| 44.1 | 阶段契约梳理 | 列出 35-43 各自 required migrations、tables、contracts |
| 44.2 | 测试 helper | 统一阶段 migration 和表结构断言 |
| 44.3 | phase35-42 测试改造 | 移除固定 latest migration 断言 |
| 44.4 | release summary 改造 | 增加阶段契约状态与后续阶段兼容说明 |
| 44.5 | diagnostic 改造 | 输出 current_latest_migration 和 required_migrations |
| 44.6 | 全量阶段回归 | 重跑 35-43 相关 pytest 和 release gate |

## 验收标准

```text
phase35-42 测试不再因 031 或更高 migration 存在而失败
如果某阶段必需表缺失，测试仍然失败并给出明确阶段名和表名
release report 中 35-42 仍保持 ready/implemented 状态
43 可处于 in_progress，不影响 35-42 的已完成判定
test_db_migrations 从空库可初始化全部当前 migration
阶段报告和 diagnostic 不泄漏 secret/token/cookie/private key/local path
不新增任何前端 UI 或桌面交互代码
```

## 与后续阶段关系

第四十四阶段是后续优化阶段的门槛修复。没有它，第四十五阶段以后每新增 migration 都可能让旧阶段回归误红，导致真实缺口和测试噪声混在一起。
