# 第七阶段：多成员协作与组织壳完善

## 阶段定位

第七阶段的目标是让系统具备“多成员协作完成复杂任务”的能力，并把组织、部门、角色、成员、壳映射这些后台结构收束到最终契约。

前六阶段已经完成：

```text

模型路由
长期记忆

资产中心
Capability Graph
Asset Broker
任务引擎
Tool Runtime
Approval Flow
Artifact Store
Task Replay
Task Reflection
Skill Engine
Plugin Bundle
MCP Registry
Skill/MCP Tool Adapter
Skill/MCP Trace 和 Replay
```

第七阶段要在这些底座上接入：

```text

```

完成后，系统必须能做到：

```text
识别需要多成员视角的复杂任务
选择主持成员和参与成员
生成可审查协作计划
为每个子成员构建最小必要上下文
按 serial / parallel / review / debate / host_summary 模式协作
让子成员调用自身允许的 Skill、MCP 和工具
隔离成员私有记忆、资产权限和任务上下文
让主持人评审、合并、裁决和收束结果
把多成员过程写入 task_events、trace、audit、replay
支持成员、部门、角色默认 Skill 和权限
支持组织壳模板加载和切换
保证切壳只改标签、菜单、模板和文案，不改底层业务值
保证聊天页仍然极简，不显示组织、壳、部门树或管理后台
```

第七阶段不是前端实现阶段，也不是安全封版阶段。

本阶段实现的是“多成员协作后端契约 + 组织壳最终后端契约”。当前仓库约束下不新增前端页面、组件、样式、Tauri 窗口或桌面交互代码，只提供 API、schema、事件、服务、评测和验收计划。最终安全审计、全量评测、性能压测和封版打磨留给第八阶段。

## 第七阶段核心结论

| 结论 | 含义 |
|---|---|
| Supervisor 是编排者，不是万能成员 | 主持人负责拆解、调度、评审、汇总和收束 |
| 多成员不是群聊乱说 | 每个成员必须有明确问题、上下文、输出格式和轮次 |
| 子成员上下文必须最小化 | 不能把全组织、全资产、全私有记忆塞给所有人 |
| 参与成员必须可解释选择 | 为什么选某成员，要能回放 |
| 成员能力来自角色、部门、个人和 Skill 策略 | 不能靠 prompt 假装有能力 |
| 子任务仍走 Task Engine | 多成员协作不绕过任务状态机 |
| 子成员工具调用仍走 Tool Runtime | Skill/MCP/Tool 权限和审批不因多成员而放宽 |
| 私有记忆默认隔离 | 成员私有记忆不能被其他成员直接读取 |
| 壳只改变展示和模板 | 不能修改底层字段值、权限语义、审计和资产分类 |
| 聊天页仍是人与当前对象对话 | 不把聊天页变成组织工作台 |

## 第七阶段优化后的工程抓手

| 抓手 | 要解决的问题 | 本阶段落点 |
|---|---|---|
| Supervisor Plan | 多成员任务需要可审查计划 | collaboration_plan、participants、rounds、success_criteria |
| Participant Selection | 成员选择不能拍脑袋 | role/department/skill/capability/eval/availability |
| Subtask Context | 子成员不能拿全量上下文 | context scope、memory scope、asset handle scope |
| Collaboration Modes | 协作流程必须可控 | serial、parallel、review、debate、host_summary |
| Host Review | 多输出必须收束 | review rubric、conflict resolution、final synthesis |
| Attribution | 谁做了什么必须可追踪 | participant_id、member_id、subtask_id、trace |
| Memory Boundary | 成员私有记忆隔离 | memory visibility、shared summary、write source |
| Capability Boundary | 成员权限不混用 | per-member Capability Graph decision |
| Shell Contract | 壳不改业务值 | labels、menus、templates、switch guard |
| Replay | 多成员过程可解释 | participant timeline、subtasks、rounds、decisions |

## 执行优先级

多成员协作中的指令优先级必须明确：

| 优先级 | 内容 | 规则 |
|---:|---|---|
| 1 | 系统安全策略 | 永远最高 |
| 2 | 用户当前目标和限制 | 覆盖协作模板和旧记忆 |
| 3 | Capability Graph / Asset Policy | 决定每个成员能否访问资源 |
| 4 | Safety 风险判断 | 决定阻断、审批或降级 |
| 5 | Approval 决策 | 用户拒绝后所有成员都不能绕过 |
| 6 | Supervisor Plan | 成员执行依据 |
| 7 | 成员 Persona 和 Role | 影响视角和表达，不改变权限 |
| 8 | Skill instructions | 只指导方法，不覆盖安全 |
| 9 | Shell labels/templates | 只影响展示和文案，不改业务值 |
| 10 | 记忆和偏好 | 只影响选择、风格和默认项 |
| 11 | 模型推理建议 | 不能覆盖以上规则 |

示例：

```text
主持人要求子成员删除文件，仍必须经过 Safety 和 Approval。
产品成员没有账号发布权限，不能因为运营成员有权限就代用。
切换壳后“部门”显示为其他标签，但 department_id 和 display_name 不自动改变。
某成员私有记忆写着用户偏好，也不能被其他成员直接读取。
```

## 状态机硬约束

第七阶段所有协作流程必须由后端状态机驱动。API、worker、supervisor、subtask runner 都不能直接改最终状态字段。

### Supervisor Task 状态

```text
created
planning
participant_selecting
planned
preflight
running
waiting_approval
paused
synthesizing
completed
failed
cancelled
archived
```

允许转移：

```text
created -> planning
planning -> participant_selecting
participant_selecting -> planned
planned -> preflight
preflight -> running
preflight -> waiting_approval
running -> waiting_approval
waiting_approval -> running
running -> paused
paused -> running
running -> synthesizing
synthesizing -> completed
running -> failed
synthesizing -> failed
planned -> cancelled
running -> cancelled
paused -> cancelled
completed -> archived
failed -> archived
cancelled -> archived
```

禁止转移：

```text
completed -> running
failed -> running
cancelled -> running
archived -> running
waiting_approval -> completed
```

### Participant Assignment 状态

```text
proposed
selected
context_prepared
running
completed
failed
skipped
removed
```

规则：

```text
selected 前必须有 selection_reason
context_prepared 前必须有 context_scope_json
running 前必须有 capability_decision_id 或 policy_exemption
completed 必须有 output_summary
failed 必须有 error_code 和 error_summary
removed 必须有 reason
```

### Subtask 状态

```text
created
planned
ready
running
waiting_approval
completed
failed
cancelled
skipped
merged
```

规则：

```text
subtask 必须有 parent_task_id
subtask 必须有 assigned_member_id
subtask running 时必须有 trace_id
subtask waiting_approval 时 parent task 必须进入 waiting_approval 或 paused
subtask completed 后才能进入 merged
subtask failed 后必须由 supervisor 决定 retry、skip、replan 或 fail parent
```

### Collaboration Round 状态

```text
planned
ready
running
waiting_inputs
reviewing
completed
failed
cancelled
```

规则：

```text
每轮必须有 mode
每轮必须有限定 participant_ids
每轮必须有 max_turns 或 max_outputs
reviewing 必须由 host member 处理
completed 必须有 round_summary
```

## 执行链路不可变式

第七阶段所有实现必须满足以下不可变式：

```text
一个 supervisor task 必须有 host_member_id
一个 supervisor task 必须有 participant plan
一个 participant assignment 必须关联 member_id
一个 subtask 必须关联 parent_task_id 和 assigned_member_id
一个子成员上下文必须有 context_scope_json
一个子任务工具调用必须按 assigned_member_id 做 Capability Graph 决策
一个子任务资产访问必须由 Asset Broker 发放绑定成员的 handle
一个协作输出必须能追溯到 member_id、subtask_id、trace_id
一个最终汇总必须能追溯到参与输出和 host decision
一个成员私有记忆写入必须有 source task/subtask/trace
一个壳切换不能修改 member、department、role 的业务字段值
一个壳模板创建业务对象必须记录 created_from_shell_id 和 created_from_template_id
```

## 上下文隔离边界

多成员协作中，每个成员像独立工作者一样只拿必要信息。

| 上下文项 | 主持成员 | 子成员 | 规则 |
|---|---|---|---|
| 用户当前目标 | 可见 | 必要摘要可见 | 不泄露无关历史 |
| 总计划 | 可见 | 仅可见自己相关部分 | 防止越界执行 |
| 其他成员私有记忆 | 不可见 | 不可见 | 默认隔离 |
| 共享任务摘要 | 可见 | 可见必要部分 | 由 Context Gateway 生成 |
| 资产句柄 | 按权限 | 按成员权限 | 不跨成员复用 |
| Skill instructions | 按需 | 按需 | 只给允许 Skill |
| MCP resources | 按权限 | 按权限 | 可信标记保留 |
| 壳标签 | 可见 | 可见 | 只影响文案 |
| 系统安全策略 | 可见摘要 | 可见摘要 | 不可覆盖 |

禁止：

```text
把全量组织结构塞给所有成员
把所有成员 persona 原文塞给所有成员
把其他成员私有记忆塞给当前成员
把所有资产 handle 复制给每个子任务
把主持人的审批权下放给子成员
把壳模板当成权限策略
```

## 小阶段 7.7：Collaboration Runner

### 目标

执行多成员协作轮次，支持可控的协作模式。

### 协作模式

| 模式 | 说明 | 约束 |
|---|---|---|
| serial | 上一成员输出给下一成员 | 每步明确输入输出 |
| parallel | 多成员并行产出 | 输出互不依赖 |
| review | 一个成员产出，另一个评审 | 评审 rubric 必填 |
| debate | 有限辩论 | 轮数上限和裁决规则必填 |
| host_summary | 主持人汇总 | 必须引用输入来源 |
| parallel_then_review | 先并行再交叉评审 | 必须有 review round |

### Runner 流程

```text
1. 加载 collaboration_plan
2. 校验 task status
3. 校验 participant status
4. 为当前 round 创建 subtasks
5. 构建每个 subtask context
6. 按 mode 执行 subtask
7. 收集 collaboration_outputs
8. 进入 host review
9. 决定下一轮、重试、跳过、取消或汇总
10. 写 task_events、trace、audit
```

### Debate 限制

```text
必须有明确议题
必须有 max_rounds
每轮每成员最多一次输出
主持人必须裁决
不允许无限争论
不允许为了角色扮演而辩论
```

### 验收

```text
serial 模式可执行
parallel 模式可执行
review 模式可执行
debate 模式有限轮次
host_summary 能汇总来源
每轮有 trace 和 task_events
```

## 小阶段 7.8：Subtask Runner

### 目标

执行分配给成员的子任务，并复用第五、第六阶段能力。

### 执行流程

```text
1. 加载 subtask
2. 校验 assigned_member_id
3. 构建 per-member ContextPacket
4. 调用 Capability Graph 决策
5. 调用 Skill Matcher，限定 allowed_skills
6. 调用 Skill Runner / Tool Runtime / MCP Adapter
7. 遇到高风险动作创建 Approval
8. 写 collaboration_outputs
9. 写 task_subtasks.output_summary_json
10. 返回给 Collaboration Runner
```

### 权限规则

```text
子任务不能使用未列入 allowed_skills 的 Skill
子任务不能使用未列入 allowed_mcp_tools 的 MCP tool
子任务不能复用其他成员 handle
子任务不能读取其他成员私有 memory
子任务不能让别的成员代执行被拒绝动作
```

### 失败处理

| 失败 | 处理 |
|---|---|
| model_timeout | fallback 或 retry |
| skill_unavailable | 降级或返回 host review |
| permission_denied | host replan 或 fail |
| approval_denied | 停止原动作，host replan |
| context_overflow | 压缩后重试 |
| unsafe_action | 阻断并解释 |

### 验收

```text
子任务可调用允许 Skill
子任务不可调用未允许 Skill
子任务按 assigned_member_id 做权限判断
approval denied 后不换成员绕过
子任务失败有 error_summary
```

## 小阶段 7.16：多成员评测与质量门禁

### 目标

### 指标

| 指标 | 含义 | 门槛 |
|---|---|---:|
| SPR | Supervisor Plan Relevance，协作计划相关率 | 0.85 |
| PSR | Participant Selection Relevance，参与者选择相关率 | 0.85 |
| CBR | Context Boundary Rate，上下文边界正确率 | 1.00 |
| PDR | Permission Decision Rate，权限决策完整率 | 1.00 |
| HSR | Host Synthesis Rate，主持汇总可用率 | 0.85 |
| RCR | Replay Completeness Rate，回放完整率 | 0.95 |
| SIR | Shell Invariant Rate，切壳不变式通过率 | 1.00 |
| NSL | No Secret Leakage，明文敏感信息泄漏数 | 0 |

### 评测用例

```text
EVAL-SUP-001：简单问答不创建 supervisor task
EVAL-SUP-002：复杂方案创建 supervisor task
EVAL-SUP-003：参与成员按角色和技能选择
EVAL-SUP-004：不可用成员不被选择
EVAL-SUP-005：子成员上下文不含其他成员私有记忆
EVAL-SUP-006：子任务资产 handle 不跨成员复用
EVAL-SUP-007：review 模式产生评审和 host decision
EVAL-SUP-008：debate 模式受轮数限制
EVAL-SUP-009：审批拒绝后不能换成员绕过
EVAL-SUP-010：多成员 replay 完整
EVAL-SHELL-001：切壳不修改 member 字段值
EVAL-SHELL-002：切壳不修改 department 字段值
EVAL-SHELL-003：系统管理不随壳改名
EVAL-SHELL-004：资产二级分类不随壳改名
EVAL-SHELL-005：壳模板应用可追溯
EVAL-TRACE-001：多成员 trace 完整
```

### 安全断言矩阵

| 断言 | 输入 | 期望 |
|---|---|---|
| 子成员不读他人私有记忆 | 子任务请求其他成员 memory | denied |
| handle 不跨成员复用 | member A handle 给 member B | ASSET_HANDLE_INVALID |
| 权限不互借 | 无权限成员执行高风险动作 | permission denied |
| 审批不绕过 | 用户拒绝后换成员执行 | blocked |
| 主持人不伪造输出 | 无 subtask output 却合成成员结论 | host review failed |
| 切壳不改数据 | switch shell | business values unchanged |
| 壳不改系统菜单 | shell config 修改 settings | validation failed |
| 壳不改资产二级 | shell config 修改 asset categories | validation failed |

### 验收

```text

上下文隔离断言通过
权限断言通过
切壳不变式通过
replay 完整性通过
```
