## 二、设计原则

### 1. 契约优先

先定义明确类型，再写实现：

```text
输入类型清晰
输出类型清晰
字段语义稳定
上下游依赖方向明确
```

### 2. 当前消息优先

所有模块都必须保留这个顶层原则：

```text
当前消息优先于历史、记忆、旧目标、旧任务状态
```

### 3. 对话与执行分层

以下概念必须分开：

```text
对话理解
会话状态
回复策略
执行编排
执行结果的对话化表达
```

### 4. 不泄露内部实现

这些中层可以处理内部状态，但输出到 `ResponseComposer` 前必须已经是“对话语义”或“策略语义”，不能把底层工具细节当成上层契约主内容。

## 四、Conversation Understanding 契约

### 1. 职责

负责回答：

```text
这一轮对话在发生什么
用户这轮最想得到什么
当前应该先回应、先澄清、先执行还是先阻断
```

### 2. 输入契约

建议：

```python
class ConversationUnderstandingRequest(BaseModel):
    turn_id: str
    conversation_id: str
    member_id: str
    user_text: str
    message_type: str = "text"
    channel_profile: str | None = None
    delivery_mode: str | None = None
    sender_label: str | None = None
    has_multimodal_parts: bool = False
    has_pending_action: bool = False
    has_running_task: bool = False
    latest_summary: str | None = None
    recent_messages: list[dict[str, Any]] = []
    trace_id: str | None = None
```

### 3. 输出契约

```python
class ConversationUnderstanding(BaseModel):
    conversation_mode: Literal[
        "casual",
        "deep_talk",
        "question",
        "task_request",
        "confirmation",
        "boundary",
        "memory_update",
        "memory_correction",
        "clarification",
    ]
    user_goal: str
    emotional_state: Literal[
        "neutral",
        "warm",
        "anxious",
        "frustrated",
        "urgent",
        "playful",
        "sad",
        "angry",
    ]
    relationship_expectation: Literal[
        "companionship",
        "advice",
        "execution",
        "confirmation",
        "explanation",
    ]
    current_turn_priority: Literal[
        "reply_first",
        "clarify_first",
        "act_first",
        "block_first",
    ]
    must_not_do: list[str] = []
    confidence: float
    reason_codes: list[str] = []
```

### 4. 字段语义

#### `conversation_mode`

定义当前轮对话大类，但不直接等同旧 `intent.primary_intent`。

#### `user_goal`

必须是用户这一轮真正想得到的东西，不能只是抽象标签。

示例：

```text
“想被理解一下，不想立刻被安排”
“想确认是否已经安装成功”
“想让我先读网页再提炼重点”
```

#### `current_turn_priority`

是后续聊天质量最关键字段之一：

```text
reply_first：先回答/接话，再考虑动作
clarify_first：先补关键澄清
act_first：可以先执行
block_first：先说明边界或阻断原因
```

## 六、SessionContext 契约

### 1. 职责

负责统一当前聊天轮的渠道/会话来源语义，供：

```text
Conversation Understanding
Dialogue State
Context Gateway
Prompt Assembler
Channel Delivery
```

共同使用。

### 2. 输入契约

```python
class SessionContextRequest(BaseModel):
    conversation_id: str
    turn_id: str
    session_id: str | None = None
    channel_profile: str | None = None
    delivery_mode: str | None = None
    sender_label: str | None = None
    sender_id: str | None = None
    thread_ref: str | None = None
    envelope_metadata: dict[str, Any] = {}
```

### 3. 输出契约

```python
class SessionContext(BaseModel):
    session_id: str | None
    conversation_id: str
    turn_id: str
    channel_profile: str | None
    delivery_mode: str | None
    sender_label: str | None
    sender_ref: str | None
    thread_ref: str | None
    route_scope: Literal["conversation", "thread", "session", "channel"]
    context_hash: str
```

### 4. 约束

```text
不存明文 secret
不承载全局可变共享态
同一并发 turn 之间不能串线
可用于 trace 和 prompt metadata，但不直接输出给用户
```

## 八、Context Budget 契约

### 1. 职责

把上下文预算从 `ContextGateway` 大函数中独立出来，显式规划：

```text
哪些部分最多占多少 token
超预算时先压缩谁
哪些部分不可裁掉
```

### 2. 输入契约

```python
class ContextBudgetRequest(BaseModel):
    total_token_budget: int
    current_message_tokens: int
    stable_prompt_tokens: int
    recent_history_tokens: int
    session_summary_tokens: int
    memory_tokens: int
    trusted_context_tokens: int
    untrusted_context_tokens: int
    reserve_tokens: int = 0
```

### 3. 输出契约

```python
class ContextBudgetPlan(BaseModel):
    recent_history_budget: int
    session_summary_budget: int
    memory_budget: int
    trusted_context_budget: int
    untrusted_context_budget: int
    reserve_budget: int
    truncation_order: list[str]
    current_message_preserved: bool = True
```

### 4. 强规则

```text
current message 不可裁掉
session summary 优先于低相关长历史
高置信 semantic memory 优先于低相关 episodic memory
untrusted context 优先摘要后裁剪
```

## 九、Response Policy 契约

### 1. 职责

### 2. 输入契约

```python
class ResponsePolicyRequest(BaseModel):
```

### 3. 输出契约

```python
class ResponsePolicyDecision(BaseModel):
```

### 4. `reply_style` 建议字段


## 十、Action Dialogue Mapper 契约

### 1. 职责

把工具、任务、审批、浏览器、系统命令、Skill、MCP 的内部执行状态，转成用户能理解的对话事实。

### 2. 输入契约

```python
class ActionDialogueRequest(BaseModel):
    source_type: Literal[
        "tool",
        "task",
        "approval",
        "browser",
        "terminal",
        "skill",
        "mcp",
    ]
    source_payload: dict[str, Any]
    session_context: SessionContext
    dialogue_state: DialogueState
    response_policy: ResponsePolicyDecision
    trace_id: str | None = None
```

### 3. 输出契约

```python
class ActionDialogueFacts(BaseModel):
    state: Literal[
        "not_started",
        "pending",
        "running",
        "blocked",
        "completed",
        "failed",
        "cancelled",
    ]
    user_visible_status: str
    boundary_disclosed: bool
    next_step: str | None = None
    evidence_summary: str | None = None
    failure_reason: str | None = None
    pending_action_count: int = 0
    reason_codes: list[str] = []
```

### 4. 强约束

```text
不得输出 approval_id / task_id / tool_call_id / trace_id 给用户可见文本
不得把 blocked / pending / running 翻译成 completed
不得把内部工具名当成人话主语
必须把“登录、验证码、权限不足、审批等待、失败重试”翻译成自然对话状态
```

## 十一、依赖方向

允许：

```text
conversation_understanding -> 只读当前 turn / 最近消息摘要
dialogue_state -> understanding / session_context / working_state / pending_action / memory snapshot
response_policy -> understanding / dialogue_state / session_context
action_dialogue_mapper -> response_policy / dialogue_state / action results
context_session / visibility / budget -> 被 context_gateway 使用
```

禁止：

```text
response_policy 直接执行工具
conversation_understanding 直接写记忆
dialogue_state 直接修改任务状态
action_dialogue_mapper 直接创建 approval
context_visibility 直接读 secret
```

## 十二、与现有模块的对接方式

### 1. `chat.py`

未来应只做编排：

```text
创建 turn
调用各核心中层
调度 brain / action / response
落 trace 和消息
```

### 2. `context_gateway.py`

继续保留为上下文总入口，但内部改为组合：

```text
SessionContext builder
Visibility policy
Budget planner
Memory selector
Resource handle selector
```

### 3. `chat_quality.py`

其边界回复策略应逐步吸收到 `ResponsePolicyDecision` + `ResponseComposer` 中，不再长期依赖散落的临时逻辑。

## 十三、错误模型建议

聊天质量中层如果失败，不应直接把内部错误暴露给用户。

建议内部错误码：

```text
CONVERSATION_UNDERSTANDING_FAILED
DIALOGUE_STATE_BUILD_FAILED
RESPONSE_POLICY_FAILED
ACTION_DIALOGUE_MAPPING_FAILED
SESSION_CONTEXT_BUILD_FAILED
CONTEXT_VISIBILITY_POLICY_FAILED
CONTEXT_BUDGET_PLAN_FAILED
```

外部可见策略：

```text
优先降级到保守 direct reply
无法保证执行诚实性时 fail closed
无法保证边界正确时走 boundary 回复
```

## 十四、最小测试要求

### 1. Conversation Understanding

至少覆盖：

```text
闲聊
深聊
确认
边界
记忆更新
当前消息改口
```

### 2. Dialogue State

至少覆盖：

```text
pending action

```

### 3. Response Policy

至少覆盖：

```text
情绪输入先回应
紧急输入更短更直接
严格格式不加聊天味修饰
边界场景 block_first

```

### 4. Action Dialogue Mapper

至少覆盖：

```text
approval pending
tool blocked
browser login required
terminal readonly completed
task failed

```

## 十五、迁移建议

建议采用三步迁移：

### 第一步

先新增 schema 和 service 壳，不接管主路径：

```text
conversation_understanding.py
dialogue_state.py
response_policy.py
```

### 第二步

在 `chat.py` 中旁路试运行，写 trace metadata，但不改变最终回复。

### 第三步

稳定后再把：

```text
route 选择
reply strategy
action mapping
```

逐步切到新路径。

## 十六、验收标准

本契约真正生效后，应满足：

```text
核心聊天质量中层都有明确输入输出类型
字段语义稳定
上下游依赖方向清晰
chat.py 不再靠大段临时逻辑决定聊天语气
执行状态可稳定转成用户可理解的对话语义
后续阶段可以直接按这些类型落代码
```
