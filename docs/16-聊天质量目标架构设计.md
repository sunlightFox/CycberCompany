# 聊天质量目标架构设计

## 文档目的

本文件用于定义“聊天质量”在本项目中的目标架构。它不讨论 UI，不讨论壳展示，不讨论营销式体验，而只讨论后端聊天主链路如何设计，才能稳定产出高质量回复。

这里的“聊天质量”包括：

```text
闲聊自然度
深度聊天质量
多轮承接
记忆使用与纠错
工具调用后的回复自然度
Skill / MCP / 浏览器 / 系统命令接入时的人话表达
安全边界与审批边界的可见回复质量
渠道聊天的即时对话感
```

本设计默认遵守仓库根 `AGENTS.md`、`docs/03-聊天主链路最终设计.md`、`docs/11-后端服务与模块详细设计.md` 的约束，不改变以下底层原则：

```text
Asset 访问必须经过 Asset Broker
权限判断必须经过 Capability Graph
高风险动作必须经过 Safety 和 Approval
每次模型调用、工具调用、审批和记忆写入都要有 trace
记忆写入必须包含 source
壳不改底层业务值
当前阶段只开发后端
```

## 一、问题定义

### 1. 当前主要问题

当前项目已经具备聊天、记忆、任务、工具、Skill、MCP、浏览器、系统命令、渠道接入等大量能力，但聊天质量仍受以下结构性问题影响：

```text
聊天主链路过于集中，`chat.py` 承担过多运行时职责
route / intent / tool shortcut / natural action / recovery 混在同一大文件
聊天回复策略没有独立中层，过度依赖 route 与补丁式文案
当前消息、历史、记忆、工具结果、渠道原文之间的优先级边界不够清晰
工具状态和审批状态没有稳定转译为“对话语义”
continuation 和 visible filter 在补后果，但没有从架构层减少系统味
```

### 2. 结果上的典型缺陷

如果不重构，常见退化会是：

```text
闲聊像模板
深聊像问答系统
多轮像“忘了上一句”
工具场景像状态回执
记忆场景像机械提醒
审批场景像后台系统提示
边界场景像安全产品报错
```

## 二、目标原则

目标不是“让系统看起来会聊天”，而是让系统在工程上具备稳定的聊天生成控制能力。

### 1. 目标原则

```text
先判断这轮对话在发生什么，再决定要不要执行能力
先决定怎么说，再决定调不调工具
当前消息永远高于旧目标、旧记忆和旧执行状态
执行状态不能替代关系状态
模型输出的是内容，系统输出的是最终对话行为
未执行的动作绝不能说成已完成
边界回复也必须像聊天，而不是像系统回执
```

### 2. 禁止性原则

```text
不允许依靠“我先接住你”“我来帮你拆一下”这类模板口头禅充当自然度
不允许在 system prompt 中直接教模型说模板句
不允许让记忆接管当前回复主导权
不允许让工具结果原样污染用户可见文本
不允许让审批状态直接暴露技术 ID
不允许让聊天质量依赖单个超大编排文件
```

## 三、目标架构总览

聊天质量目标架构采用“对话控制优先、执行能力挂接”的分层方式：

```text
Ingress
-> Conversation Understanding
-> Dialogue State Resolver
-> Context Gateway
-> Response Policy
-> Brain Decision
-> Action Orchestrator
-> Model Generation
-> Action Dialogue Mapper
-> Response Composer
-> Continuation Quality Gate
-> Visible Filter
-> Channel Delivery
-> Memory Writeback
-> Trace
```

其中真正决定聊天质量的核心中层是：

```text
Conversation Understanding
Dialogue State Resolver
Response Policy
Action Dialogue Mapper
```

## 四、核心分层设计

### 1. Ingress Layer

职责：

```text
接收用户输入
规范化文本、多模态、附件、渠道元数据
生成 message envelope
去重与 debounce
建立 turn 基础记录
```

输入：

```text
ChatTurnRequest
渠道入站 payload
附件与 context refs
```

输出：

```text
NormalizedTurnInput
MessageEnvelope
基础 trace
```

关键要求：

```text
只做输入归一化，不做复杂回复决策
保留 session_id / sender / thread / channel 元数据
外部原文不默认视为 trusted instruction
```

### 2. Conversation Understanding Layer

职责：

```text
判断这一轮对话在发生什么
拆解社交含义、任务含义、风险含义
产出当前轮对话语义，而不是只产出传统 intent 标签
```

建议输出结构：

```json
{
  "conversation_mode": "casual|deep_talk|question|task_request|confirmation|boundary|memory_update",
  "user_goal": "用户这轮最想得到什么",
  "emotional_state": "calm|anxious|frustrated|urgent|playful",
  "relationship_expectation": "陪伴|建议|执行|确认|解释",
  "current_turn_priority": "reply_first|clarify_first|act_first|block_first",
  "must_not_do": [
    "不要直接执行",
    "不要假装完成"
  ]
}
```

与旧 intent 的关系：

```text
旧 intent 仍可作为下游兼容输入
但聊天质量主判断不再只依赖 primary_intent
```

### 3. Dialogue State Resolver

职责：

```text
维护当前会话的“关系状态”和“动作状态”
从历史、working state、pending action、最近 turn 中提炼当前对话态
把“正在聊什么”和“正在执行什么”分开
```

状态建议拆为：

```text
identity_state
relationship_state
conversation_state
action_state
memory_state
session_state
```

示例：

#### identity_state

```text
当前聊天对象稳定身份
不是现实真人
有哪些稳定风格约束
```

#### relationship_state

```text
熟悉度
松紧度
允许的幽默程度
用户是否在赶时间
```

#### conversation_state

```text
当前主话题
当前问题焦点
用户是否改口
当前是否偏离主题
```

#### action_state

```text
是否有 pending confirmation
是否存在执行中任务
是否刚刚完成某个动作
是否处于失败恢复态
```

#### memory_state

```text
本轮应参考的记忆快照集合
记忆可信度
记忆是否与当前消息冲突
```

### 4. Context Gateway

职责：

```text
按预算组装当前轮真正需要的上下文
控制 trusted / untrusted 边界
控制 session / quote / forwarded / workbench / multimodal 可见性
```

上下文输入不再是“能拿到什么都尽量塞进去”，而是围绕当前对话态挑选：

```text
当前消息
最近相关消息
会话摘要
高相关长期记忆
必要的 persona / heart / skill index
必要的工具/浏览器结果摘要
必要的渠道来源上下文
```

上下文目标不是“信息尽可能全”，而是：

```text
让模型准确回答当前这句话
```

### 5. Response Policy Layer

职责：

```text
决定这一轮“怎么说”
决定是否该先回应情绪、先澄清、先确认、先执行
决定回复长度、结构、语气密度
给模型和 composer 提供明确回复契约
```

建议输出：

```json
{
  "reply_style": {
    "warmth": "low|mid|high",
    "directness": "low|mid|high",
    "humor": "none|light",
    "structure": "free|light_structured|structured",
    "length": "short|medium|long"
  },
  "reply_contract": {
    "answer_current_message_first": true,
    "acknowledge_emotion_if_present": true,
    "do_not_use_systemic_phrasing": true,
    "do_not_claim_done_without_evidence": true,
    "do_not_explain_internal_rules_unless_needed": true
  },
  "action_contract": {
    "can_execute_now": false,
    "needs_confirmation": true,
    "needs_clarification": false
  }
}
```

这是聊天质量的关键中层。  
没有这层，系统就会继续依赖 route 和模板话术输出“像助手的话”。

### 6. Brain Decision Layer

职责：

```text
在 Response Policy 已经确定回复策略后，再决定 direct / workflow / agent / supervisor
选择模型和推理强度
决定是否需要 task、tool、skill、mcp、browser、terminal
```

顺序要求：

```text
先有对话策略，再做执行路由
```

而不是：

```text
先走到 task/tool 分支，再补对话
```

### 7. Action Orchestrator

职责：

```text
统一调度 task、tool、skill、mcp、browser、terminal
管理执行前确认、执行中状态、执行后结果
把“执行系统”从“聊天系统”中隔离出来
```

建议统一动作模式：

```text
reply_only
reply_plus_prepare
reply_plus_execute
```

含义：

#### reply_only

```text
纯聊天
纯边界解释
纯记忆修正
纯概念回答
```

#### reply_plus_prepare

```text
先回复
再创建任务、计划、待确认动作
但不直接执行高风险动作
```

#### reply_plus_execute

```text
回复与执行同时发生
仅在满足安全边界、能力边界、上下文充分时允许
```

### 8. Model Generation

职责：

```text
依据 prompt 与回复契约生成内容
不直接负责最终用户可见文案的全部真实性兜底
```

要求：

```text
当前消息优先
不能被旧目标绑架
工具与任务内容只作辅助
不可信上下文隔离
```

### 9. Action Dialogue Mapper

职责：

```text
把 machine state 转成 dialogue state
把工具状态、审批状态、浏览器状态、系统命令状态翻译成自然会话语义
```

这是当前架构非常缺的一层。

例如内部状态：

```json
{
  "tool": "browser.snapshot",
  "status": "blocked",
  "reason": "login_required"
}
```

不能直接给用户：

```text
当前浏览器工具被阻断，需要登录。
```

应该先映射成：

```text
我已经看到入口了，但这一步卡在登录上。你如果要我继续，我得在登录完成后再往下走。
```

所以必须有一层专门负责：

```text
execution state -> conversation-ready facts
```

### 10. Response Composer

职责：

```text
整合模型内容、动作语义、审批语义、记忆语义
生成 summary / plain_text / sections / action buttons / structured payload
控制结构、可见边界和话术一致性
```

它不是 prompt builder，也不应该变成大型模板拼句器。

### 11. Continuation Quality Gate

职责：

```text
识别系统味、模板味、假完成、多模态空泛回复、严格格式污染
必要时触发一次修订
```

它应该是“质量闸门”，不是“聊天质量主逻辑”。

### 12. Visible Filter

职责：

```text
用户可见脱敏
内部字段剔除
技术 ID 隐藏
严格格式保护
```

### 13. Channel Delivery

职责：

```text
把最终可见回复投递到微信、飞书、CLI 等渠道
处理流式消息、tool progress、approval UI、最终消息
```

要求：

```text
渠道差异只体现在投递层
不改变聊天核心决策
```

### 14. Memory Writeback

职责：

```text
从当前 turn 提取可写入记忆
与 source、trace、纠错关系绑定
但不反向篡改当前回复
```

## 五、数据结构建议

### 1. ConversationUnderstanding

建议 schema：

```python
class ConversationUnderstanding(BaseModel):
    conversation_mode: str
    user_goal: str
    emotional_state: str
    relationship_expectation: str
    current_turn_priority: str
    must_not_do: list[str] = []
    confidence: float = 0.0
    reason_codes: list[str] = []
```

### 2. DialogueState

建议 schema：

```python
class DialogueState(BaseModel):
    identity_state: dict[str, Any]
    relationship_state: dict[str, Any]
    conversation_state: dict[str, Any]
    action_state: dict[str, Any]
    memory_state: dict[str, Any]
    session_state: dict[str, Any]
```

### 3. ResponsePolicyDecision

建议 schema：

```python
class ResponsePolicyDecision(BaseModel):
    reply_style: dict[str, Any]
    reply_contract: dict[str, Any]
    action_contract: dict[str, Any]
    scenario: str
    tone_mode: str
    quality_contract: dict[str, Any]
```

### 4. ActionDialogueFacts

建议 schema：

```python
class ActionDialogueFacts(BaseModel):
    state: str
    user_visible_status: str
    boundary_disclosed: bool
    next_step: str | None = None
    evidence_summary: str | None = None
    failure_reason: str | None = None
    pending_action_count: int = 0
```

## 六、Prompt 设计原则

聊天质量提升不应通过 system prompt 直接教授模板句式完成。

### 1. 正确方式

system prompt 只约束行为：

```text
优先回应当前消息
情绪输入先不要急着给方案
未执行的动作不能说成已完成
边界回复要给替代帮助
多模态必须回应具体识别内容
```

### 2. 错误方式

不要写成：

```text
先接住用户情绪
先说“我懂你”
先说“别急”
先给三步法
```

因为模型会把这些句式本身当成输出模板。

### 3. Prompt 中不应出现的隐性坏味道

```text
“我来帮你拆一下”
“先接住你这句话”
“让我先共情一下”
“下面给你一个框架”
“我先从三个层面分析”
```

这些都不一定是错话，但不能成为系统级默认语气。

## 七、与工具、Skill、MCP、浏览器、终端的关系

### 1. 工具系统是聊天的挂接能力，不是聊天本体

聊天回复生成顺序必须是：

```text
先决定怎么回应这个人
再决定要不要调用能力
```

### 2. Skill

Skill 进入上下文时只作为：

```text
方法线索
可复用流程提示
```

不能直接变成：

```text
已执行结果
资源权限
secret 来源
```

### 3. MCP

MCP 不只是工具注册表，也应该是聊天态的桥接能力层，但 MCP 输出仍必须经过：

```text
Action Dialogue Mapper
Response Composer
Visible Filter
```

### 4. 浏览器

浏览器状态必须先转成页面语义：

```text
已观察
需登录
需确认
遇到挑战
已获取候选结果
```

不能把内部执行器细节直接给用户。

### 5. 系统命令

系统命令回复必须严格区分：

```text
未执行
已执行
执行中
等待确认
失败
```

这是执行诚实性的底线。

### 1. 纯闲聊

```text
-> Context Gateway
-> Response Policy
-> Brain direct
-> Model Generation
-> Response Composer
-> Continuation
-> Visible Filter
-> Delivery
-> Memory Writeback
```

### 2. 带确认的执行请求

## 十、阶段落地顺序

建议实现顺序：

### 第一阶段

```text
conversation_understanding
dialogue_state
response_policy
```

先把聊天控制层立起来。

### 第二阶段

```text
context_session
context_visibility
context_budget
```

把上下文治理立起来。

### 第三阶段

```text
action_dialogue_mapper
tool/browser/terminal runtime 拆分
```

把执行系统的人话转译立起来。

### 第四阶段

```text
channel session context
channel stream / approval bridge
```

把渠道聊天体验收口。

## 十一、验收标准

达到目标架构后，聊天质量应至少满足：

```text
闲聊不像系统提示
深聊不是纯问答模板
多轮能承接当前话题
用户改口后立即以当前消息为准
工具和浏览器结果能翻译成人话
审批、失败、等待确认、已完成四类状态明确区分
边界回复自然且有替代路径
多模态回复必须回应具体识别内容
用户可见文本中无内部字段和技术 ID
```

## 十二、与开发计划的关系

本设计文档对应以下后续开发计划：

```text
第六十九阶段 - ChatRuntime 主链路拆分与路由收敛
第七十阶段 - SessionContext 与上下文可见性治理重构
第七十一阶段 - ToolRuntime、终端队列与执行语义拆分重构
第七十二阶段 - 浏览器任务会话连续性与页面状态重构
第七十三阶段 - Skill、MCP 与渠道桥接运行时重构
第七十四阶段 - 聊天质量运行时总收口与文件清理
```

它是这些阶段的上位目标设计，不替代具体开发计划。
