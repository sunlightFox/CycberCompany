# 组织壳与公司壳设计

## 设计目标

组织壳系统要解决两件事：

1. 首发用公司壳让用户容易理解。
2. 底层不写死公司，未来可以扩展到宗门、朝廷、学院、舰队等壳。

核心原则：

> 壳是展示层、模板层和引导层，不是底层数据模型。

## 底层通用对象

底层对象使用通用命名：

```text
Organization
Member
Department
Role
Shell
Asset
Skill
Task
Memory
Capability
Policy
Trace
```

代码里避免写死：

```text
Company
Employee
Boss
ImperialCourt
Disciple
Emperor
```

公司壳、朝廷壳、宗门壳都通过 Shell 配置映射到底层对象。

## 壳系统职责

| 职责 | 说明 |
|---|---|
| 术语映射 | organization 显示为公司、宗门、朝廷 |
| 菜单映射 | chat 显示为企业聊天、朝议、宗门议事 |
| 字段标签 | role 显示为岗位、身份、官职 |
| 创建模板 | 公司壳给员工模板，宗门壳给弟子模板 |
| 默认结构 | 公司壳默认部门，朝廷壳默认六部 |
| 世界观文案 | 欢迎语、空状态、提示文案 |
| 主题 | 颜色、背景、图标风格 |
| Prompt 修饰 | 轻量语气，不改变安全和真实数据 |

壳不负责：

```text
改变成员真实字段值
改变权限语义
改变资产分类
改变系统管理结构
改变审计日志
绕过 Safety
替代 Capability Graph
```

## 壳切换规则

### 正确行为

```text
字段标签变
菜单名变
模板变
空状态文案变
主题变
已存在数据值不变
```

示例：

| 公司壳 | 宗门壳 | 底层字段 |
|---|---|---|
| 员工 | 弟子 | Member |
| 部门 | 堂口 | Department |
| 岗位 | 身份 | Role |
| 工作任务 | 宗门事务 | Task |
| 资产管理 | 宗门资源 | Asset |

如果公司壳里成员数据为：

```text
name = 阿珩
department = 研发部
role = 技术经理
```

切到宗门壳后显示为：

```text
姓名：阿珩
堂口：研发部
身份：技术经理
```

只有用户主动编辑为：

```text
身份：炼器长老
```

底层 `role` 值才改变。

### 禁止行为

```text
切壳后自动把技术经理改成炼器长老
切壳后自动把研发部改成炼器堂
切壳后重写任务历史
切壳后修改记忆内容
切壳后让系统管理改名
切壳后让资产二级分类改名
```

## 菜单规则

底层菜单 key 固定：

```yaml
menus:
  chat: /chat
  members: /members
  organization: /organization
  assets: /assets
  tasks: /tasks
  settings: /settings
```

公司壳映射：

```yaml
menus:
  chat: 企业聊天
  members: 员工管理
  organization: 部门管理
  assets: 资产管理
  tasks: 工作任务
  settings: 系统管理
```

朝廷壳可映射：

```yaml
menus:
  chat: 朝议
  members: 官员管理
  organization: 衙署管理
  assets: 国库资源
  tasks: 政务中心
  settings: 系统管理
```

系统管理固定：

```text
系统管理名称固定
系统管理路由固定
系统管理二级菜单固定
系统管理页面排版固定
```

资产管理一级可变，二级固定：

```text
大脑
账号
钱包
硬件
知识库
```

## 公司壳 MVP

### 公司壳定位

公司壳把产品包装成“一人公司”：

```text
用户 = 老板/CEO/创始人
AI 生命体 = 员工
部门 = 部门
角色 = 岗位
任务 = 工作任务
资产 = 公司资产
Skill = 技能
```

### 默认部门

MVP 只内置五个部门：

| key | 名称 | 职责 |
|---|---|---|
| ceo_office | 总裁办 | 统筹、拆解、复盘、协调 |
| product | 产品部 | 需求、产品、体验、路线图 |
| engineering | 研发部 | 架构、代码、部署、排错 |
| operations | 运营部 | 内容、账号、增长、数据 |
| life_service | 生活管家部 | 生活、提醒、家居、陪伴 |

### 默认角色

| key | 名称 | 默认部门 | 职责 |
|---|---|---|---|
| chief_of_staff | 首席助理 | 总裁办 | 任务统筹、计划、复盘 |
| product_manager | 产品经理 | 产品部 | 产品设计、需求拆解 |
| architect | 技术架构师 | 研发部 | 架构设计、代码、部署 |
| content_operator | 内容运营 | 运营部 | 文案、标题、账号内容 |
| personal_butler | 私人管家 | 生活管家部 | 日程、陪伴、家居 |

## 组织协作设计

### 单聊

```text
老板 ↔ 小曜
老板 ↔ 阿珩
老板 ↔ 墨白
```

适合明确任务或连续关系。

### 部门群聊

```text
老板 ↔ 产品部
老板 ↔ 研发部
老板 ↔ 运营部
```

群聊必须有主持人，不能让所有成员乱说。

流程：

```text
用户提出问题
主持人拆解问题
相关成员分别产出
主持人汇总
用户确认下一步
```

### 全公司会议

适合复杂规划。必须限制发言轮次。

示例：

```text
主持：小曜
产品：宁宁
技术：阿珩
运营：墨白
体验/生活：小栖
```

输出要服务结果，不做戏剧化角色扮演。

## 权限主体

底层权限主体：

```text
User
Organization
Department
Member
Role
Skill
Asset
Tool
MCPServer
```

权限动作：

```text
read
write
execute
approve
delegate
manage
admin
```

权限展示由壳决定，但底层动作不变。

## 组织和壳的验收标准

| 验收项 | 标准 |
|---|---|
| 公司壳 | 默认菜单、部门、角色、成员模板可加载 |
| 底层抽象 | 代码核心表和 schema 不写死 Employee/Company |
| 切壳 | 只改变标签和菜单，不改变数据值 |
| 聊天页 | 不展示组织名、壳名、部门名 |
| 系统管理 | 不随壳改名或改结构 |
| 资产二级分类 | 永远固定为大脑、账号、钱包、硬件、知识库 |
| 旧数据兼容 | 换壳后旧成员、旧任务、旧记忆保持可读 |
