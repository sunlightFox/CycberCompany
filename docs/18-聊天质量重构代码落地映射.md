# 聊天质量重构代码落地映射

## 文档目的

本文件回答一个非常具体的问题：

> 在当前项目里，聊天质量架构要怎么落代码，才能严格提升质量，而不是“越拆越乱、越改越差”？

这里不讨论抽象目标，而是把：

```text
现有文件
新文件
迁移内容
不能先动的内容
回归护栏
灰度切换方式
删除时机
```

全部映射清楚。

本文件是以下文档的实施层补充：

```text
16-聊天质量目标架构设计.md
17-聊天质量核心接口契约设计.md
69-74 阶段开发计划
```

## 一、总原则：聊天质量重构必须“保守上线”

聊天质量和普通重构不一样。普通重构只要测试过了就行，聊天质量重构如果方法不对，会出现一种很危险的情况：

```text
架构更“优雅”
代码更“干净”
但真实回复更机械、更慢、更空、更像系统
```

所以这次必须遵守五条硬原则：

### 1. 不直接全量替换主链路

必须采用：

```text
新增旁路
写 trace
做对照
局部接管
回归通过后再扩大
```

禁止：

```text
一口气重写 chat.py 再一起接管全部场景
```

### 2. 先保回复自然度，再保结构优雅

任何拆分如果导致这些退化，都算失败：

```text
更像模板
更像系统回执
更不承接上一轮
更容易误判当前消息
更容易把未执行说成已完成
```

### 3. 先保当前消息优先，再谈更复杂记忆

重构期间，宁可少用一点记忆，也不能让旧记忆、旧任务、旧审批状态压过当前消息。

### 4. 先保执行诚实性，再保“聪明感”

以下质量绝对不能回退：

```text
未执行不说完成
失败不说成功
等待确认不说继续执行中
登录/验证码/权限卡住时不假装完成
```

### 5. 所有新中层先“只读观测”，再“参与决策”

这点最重要。

新模块第一阶段只允许：

```text
读取输入
产出理解结果
写 trace metadata
做对照评估
```

第一阶段不直接决定最终用户回复。

## 二、必须先保护的聊天质量指标

在任何代码迁移开始前，先把聊天质量最核心的保护目标固定下来。

### 1. 第一优先级指标

这些一旦退化，就说明重构方向错了：

| 指标 | 说明 |
|---|---|
| 当前消息响应准确性 | 不被旧上下文带偏，能回答当前这句话 |
| 执行诚实性 | 未执行不说完成，失败不说成功 |
| 对话自然度 | 不出现明显系统话、模板话、回执话 |
| 多轮承接 | 第二轮能承接第一轮，不像重新开对话 |
| 边界自然度 | 拒绝/确认/风险提示像聊天，不像报错 |

### 2. 第二优先级指标

这些也重要，但可以在第一优先级守住后逐步拉升：

| 指标 | 说明 |
|---|---|
| 记忆引用质量 | 记忆有用但不抢主导权 |
| 长回复结构质量 | 长回复更清晰，但不更模板 |
| 多模态具体性 | 图片/语音/文件不是“收到”，而是具体回应内容 |
| 渠道即时感 | 微信/飞书回复更像即时对话 |

## 三、现有文件风险分级

不是所有文件都该同时动。先按“聊天质量回退风险”分级。

### A 级：主链路高风险文件

这些文件一动就可能直接影响真实回复质量：

```text
apps/local-api/app/services/chat.py
apps/local-api/app/services/context_gateway.py
apps/local-api/app/services/natural_chat.py
apps/local-api/app/services/chat_continuation.py
apps/local-api/app/services/chat_quality.py
apps/local-api/app/services/chat_response.py
```

策略：

```text
只做旁路接入
不一次性重写
每次只迁一个职责
每迁一处就跑质量回归
```

### B 级：执行语义高风险文件

这些文件会影响“说没说真话”：

```text
apps/local-api/app/services/tools.py
apps/local-api/app/services/browser_workflows.py
apps/local-api/app/services/terminal_sandbox.py
apps/local-api/app/services/mcp_runtime.py
apps/local-api/app/services/skill_plugin.py
```

策略：

```text
先抽 domain runtime，不动用户可见话术
先标准化结果结构，再接入 action dialogue mapper
```

### C 级：渠道语义高风险文件

这些文件影响微信、飞书等渠道的即时聊天感：

```text
apps/local-api/app/services/channel_connectors.py
apps/local-api/app/services/chat_ingress.py
apps/local-api/app/services/turn_events.py
```

策略：

```text
先补 session context 和 stream bridge
后拆 provider 文件
先不改变真实渠道文案风格
```

## 四、代码落地总顺序

严格建议按下面顺序来，不要跳。

### 第 0 步：先建观测，不接管回复

新增：

```text
conversation_understanding.py
dialogue_state.py
response_policy.py
```

但这一阶段只允许：

```text
在 chat.py 中调用
把结果写入 trace metadata / structured payload
与旧主链路并行存在
不改变最终输出文本
```

为什么必须这样做：

```text
先看这些中层“理解得对不对”
如果一上来接管回复，错了很难定位到底是理解错、策略错、还是 composer 错
```

### 第 1 步：先接管“理解”，不接管“说法”

这一阶段让新中层开始参与：

```text
route 辅助判断
continuation 触发条件判断
当前消息优先级判断
pending action 模糊确认拦截
```

但仍然不直接接管：

```text
最终文本生成
长文案结构
边界话术
```

### 第 2 步：接管“回复策略”，但不接管“执行结果表达”

让 `ResponsePolicyDecision` 开始决定：

```text
长短
结构程度
是否先回应情绪
是否先澄清
是否应该进入 act_first
```

但工具、浏览器、审批、终端等执行结果仍旧按老路径表达。

为什么：

```text
先证明“怎么说”这层不会把闲聊和深聊搞坏
不要同时掺入工具结果语义重构
```

### 第 3 步：接管执行状态的人话映射

新增：

```text
action_dialogue_mapper.py
```

这一步开始把：

```text
approval
browser login required
terminal readonly done
tool blocked
task running / failed / completed
```

统一翻译成对话语义。

这是第二个高风险切换点。  
如果这里做不好，就会立刻出现：

```text
更像系统回执
更容易假完成
更容易说得绕
```

### 第 4 步：最后再拆主大文件

等理解层、策略层、动作对话层都稳定后，再拆：

```text
chat.py
tools.py
browser_workflows.py
channel_connectors.py
```

因为这时已经不是“边拆边猜行为”，而是“按既定契约搬家”。

## 五、现有文件到新文件的精确映射

### 1. `chat.py`

现状问题：

```text
主链路总控过重
直接持有 route-specific 逻辑
直接持有工具快捷回复逻辑
直接持有不少用户可见补丁
```

建议拆分为：

```text
chat.py                            兼容入口层
chat_runtime_orchestrator.py       turn 编排
chat_turn_lifecycle.py             turn 持久化与结束
chat_direct_routes.py              direct route 处理
chat_tool_shortcuts.py             browser/filesystem/readonly terminal shortcut
chat_stream_runtime.py             事件流与流式投递
chat_recovery_runtime.py           turn 恢复与 retry
```

迁移顺序：

```text
先迁 _handle_browser_read_page / _handle_terminal_readonly_command / _handle_host_filesystem_list
再迁 _execute_turn 主流程
最后缩薄 ChatService
```

严禁先动：

```text
create_turn
stream_turn_events
消息落库核心逻辑
trace 根 span 生命周期
```

这些必须最后再整理。

### 2. `context_gateway.py`

现状问题：

```text
summary / recent / memory / handles / persona / heart 都集中在 build()
可见性和 token budget 没有独立策略层
```

建议拆分为：

```text
context_gateway.py
context_session.py
context_visibility.py
context_budget.py
context_untrusted.py
context_message_selection.py
```

迁移顺序：

```text
先抽 token budget
再抽 visibility
再抽 session context
最后缩 build()
```

为什么不能反过来：

```text
先拆 build() 再补预算和可见性，最容易造成上下文质量回退
```

### 3. `natural_chat.py`

现状问题：

```text
自然语言确认做得多，但越来越像规则入口
确认语义、审批语义、可见文本语义仍绑得太紧
```

建议保留并逐步拆出：

```text
approval_dialogue.py
pending_action_binding.py
action_dialogue_mapper.py
```

迁移顺序：

```text
先抽 pending action 绑定结构
再抽 resolution 结果结构
最后把用户可见文本交给 ResponseComposer
```

严禁先动：

```text
当前能正常工作的 approve / deny / edit 真实业务流程
```

先保语义正确，再保结构优雅。

### 4. `chat_continuation.py`

现状问题：

```text
承担续跑质量修订，但大量 cue-based heuristic
```

迁移策略：

```text
先不拆文件
先让 ConversationUnderstanding / ResponsePolicy 接管“是否值得启用 continuation”
再逐步收缩 heuristic 体量
```

为什么不先拆：

```text
continuation 本来就是质量兜底层
先拆它最容易让输出一下子变差
```

### 5. `tools.py`

现状问题：

```text
工具域太多
结果字段不够统一
聊天层容易直接理解某一类工具的私有结果结构
```

建议拆分为：

```text
tool_runtime.py
tool_terminal_runtime.py
tool_browser_runtime.py
tool_skill_runtime.py
tool_mcp_runtime.py
tool_memory_runtime.py
tool_knowledge_runtime.py
tool_artifact_runtime.py
```

迁移顺序：

```text
先统一返回结构
再抽 terminal runtime
再抽 browser runtime
最后抽 skill/mcp/knowledge 等域
```

### 6. `browser_workflows.py`

现状问题：

```text
意图、计划、执行、恢复、页面语义揉在一起
```

建议拆分为：

```text
browser_workflow_runtime.py
browser_intent_resolver.py
browser_plan_runtime.py
browser_session_runtime.py
browser_page_state.py
browser_replay_store.py
```

迁移顺序：

```text
先抽 page state
再抽 session runtime
再抽 plan / execute
```

为什么：

```text
页面状态语义是浏览器聊天质量的核心
不先抽这个，后面所有“人话表达”都不稳
```

### 7. `channel_connectors.py`

现状问题：

```text
provider IO、绑定、发消息、轮询、音频、事件、状态都堆一起
还承担了一部分会话级语义
```

建议拆分为：

```text
channel_session_context.py
channel_stream_bridge.py
channel_approval_bridge.py
channels/providers/wechat.py
channels/providers/feishu.py
```

迁移顺序：

```text
先抽 session context
再抽 stream bridge
最后再拆 provider 文件
```

为什么不能先拆 provider：

```text
先拆 provider 容易只得到更碎的 IO 代码
但聊天质量问题仍然不解决
```

## 六、每一步必须看的质量回退信号

### 1. 一旦出现这些，说明改坏了

| 回退信号 | 说明 |
|---|---|
| 回复更短但更空 | 策略层把信息压没了 |
| 回复更结构化但更像模板 | ResponsePolicy 或 Composer 过强 |
| 更容易答非所问 | ConversationUnderstanding 或 Context 选取变差 |
| 更像“当前状态如下” | ActionDialogueMapper 没翻译成人话 |
| 更常说“我可以/我不能”但不继续聊 | 边界语义太系统化 |
| 更常忘记上一轮重点 | DialogueState 没真正接住会话 |
| 多模态更常只说收到 | 新策略没把具体识别内容拉进主回复 |

### 2. 一旦出现这些，必须立即回滚该阶段接管

```text
未执行说成已完成
审批前声称已继续执行
浏览器卡在登录却说拿到了结果
系统命令失败却用成功口吻总结
当前消息改口后仍沿用旧目标
```

## 七、灰度接管建议

### 第 1 级：trace-only

```text
新模块只输出 metadata
不改变 route
不改变 reply
```

### 第 2 级：advisory-only

```text
新模块给建议
旧模块仍做最终决策
记录新旧分歧
```

### 第 3 级：soft-control

```text
新模块接管非高风险场景
高风险动作仍保留旧逻辑兜底
```

### 第 4 级：full-control

```text
新模块成为主路径
旧路径只作回退兼容
```

严格建议：

```text
闲聊 -> 深聊 -> 多轮 -> 记忆 -> 工具只读 -> 审批 -> 浏览器 -> 系统命令
按这个风险顺序逐步接管
```

## 八、具体回归矩阵

每接管一层，至少回归以下场景：

### 1. ConversationUnderstanding 接管前后

必须测：

```text
闲聊
深聊
当前消息改口
pending action 模糊确认
```

### 2. ResponsePolicy 接管前后

必须测：

```text
情绪输入
赶时间短答
长结构化问题
严格 JSON 请求
```

### 3. ActionDialogueMapper 接管前后

必须测：

```text
approval pending
tool blocked
browser login required
terminal readonly success
task failed
```

### 4. Channel Session / Stream Bridge 接管前后

必须测：

```text
微信短消息即时回复
多轮聊天承接
工具执行中间态
审批等待态
多模态消息
```

## 九、删除时机

只有满足以下条件，旧逻辑才允许删除：

### 1. 功能对齐

```text
新路径语义覆盖旧路径
structured payload 不丢关键信息
trace 不丢
approval / safety / memory / task 边界不丢
```

### 2. 质量不退化

以下全部不回退：

```text
闲聊自然度
深聊质量
多轮承接
执行诚实性
边界自然度
```

### 3. 已经过两轮回归

至少：

```text
单元回归一轮
主链路质量回归一轮
```

才能删旧逻辑。

## 十、建议新增的护栏开关

为了防止越改越差，建议新增几个配置开关：

```text
chat_quality.enable_conversation_understanding_shadow
chat_quality.enable_dialogue_state_shadow
chat_quality.enable_response_policy_advisory
chat_quality.enable_action_dialogue_mapper_shadow
chat_quality.enable_action_dialogue_mapper_primary
chat_quality.enable_session_context_primary
```

语义：

```text
shadow：只观测，不接管
advisory：给建议，不拍板
primary：正式接管
```

这样能保证：

```text
不是靠改代码分支硬切换
而是能在真实链路里做受控灰度
```

## 十二、验收标准

这份映射真正执行成功，必须满足：

```text
每一步迁移都有明确“先观测后接管”的策略
聊天质量第一优先级指标不回退
主链路不出现“越拆越像系统”的现象
执行诚实性全程守住
最终形成稳定文件边界，而不是把大文件拆成更多碎补丁
```

## 十三、与下一步实施的关系

本文件之后，真正进入代码实现时，建议顺序是：

```text
19：聊天质量重构实施任务拆分清单
20：ConversationUnderstanding / DialogueState / ResponsePolicy 最小代码骨架
21：trace-only 旁路接入
22：第一轮质量回归与对照报告
```

也就是说，下一步仍然不应该直接大改主链路，而应该先进入：

```text
最小骨架 + 旁路观测 + 质量对照
```

这是避免“越改越差”的唯一稳妥路径。
