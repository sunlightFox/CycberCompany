# 任务引擎与多成员协作详细设计

## 任务引擎目标

任务引擎负责把聊天里的自然语言目标变成可执行、可暂停、可恢复、可回放的任务。

目标：

```text
用户一句话创建任务
任务有明确成功标准
固定任务走 workflow
探索任务走 agent
复杂任务走 supervisor
高风险动作暂停确认
每一步可回放
失败可重试
结果可沉淀为记忆和 Skill
```

## 任务模式

| 模式 | 使用条件 | 示例 |
|---|---|---|
| direct | 不需要任务持久化 | 普通问答 |
| workflow | 步骤明确、风险可控、结果可验证 | 整理文件、生成日报 |
| agent | 需要搜索、探索、动态判断 | 调研竞品、查资料 |
| supervisor | 需要多个成员视角 | 产品方案、技术方案、运营方案 |

默认规则：

```text
能 workflow 不 agent
能单成员不 supervisor
高风险任务先计划后执行
长任务必须可暂停恢复
```

## 任务状态机

```text
created
planning
planned
waiting_approval
running
paused
completed
failed
cancelled
archived
```

状态转移：

```text
created -> planning
planning -> planned
planned -> running
running -> waiting_approval
waiting_approval -> running
waiting_approval -> paused
running -> completed
running -> failed
running -> paused
paused -> running
any -> cancelled
completed -> archived
failed -> archived
```

禁止：

```text
completed -> running
cancelled -> running
archived -> running
```

## Step 类型

| 类型 | 说明 |
|---|---|
| memory_recall | 检索记忆 |
| asset_query | 查询资源 |
| skill_match | 匹配 Skill |
| tool_call | 调用工具 |
| mcp_call | 调用 MCP 工具 |
| subtask | 分配给子成员 |
| approval | 等待用户确认 |
| compose | 汇总输出 |
| eval | 结果校验 |
| reflection | 任务复盘 |

## Workflow Runner

适合固定流程。

示例：整理文件夹

```text
扫描目录
分类文件
识别重复
生成建议
等待用户确认移动/删除
执行确认后的操作
生成报告
写入任务经历
```

Workflow 特点：

```text
步骤提前定义
每一步输入输出明确
失败点清楚
可重复执行
适合 Skill 固化
```

## Agent Runner

适合探索型任务。

循环：

```text
observe
think
act
evaluate
continue_or_stop
```

停止条件：

```text
达到成功标准
达到最大步数
遇到高风险确认
用户取消
成本超限
连续失败
```

Agent 必须受限：

```text
最大工具调用次数
最大模型调用次数
最大运行时间
最大成本
允许工具列表
风险策略
```

## Supervisor Runner

用于多成员协作。

### 核心原则

```text
主持人负责拆解和汇总
子成员只拿必要上下文
每个子成员输出有格式
限制发言轮数
不做无意义角色扮演
最终结果必须收束
```

### 典型流程

```text
用户提出目标
主持人判断是否需要多成员
主持人生成协作计划
Context Gateway 为每个成员构建子上下文
子成员并行或串行执行
主持人评审和汇总
Response Composer 输出给用户
Memory 写入经历和偏好
```

### 协作模式

| 模式 | 说明 | 示例 |
|---|---|---|
| serial | 上一步输入给下一步 | 产品 -> 技术 -> 运营 |
| parallel | 多个方向同时产出 | 产品方案、技术方案同时写 |
| review | 一个产出，一个评审 | 阿珩写架构，宁宁评审需求匹配 |
| debate | 有限辩论 | 每人一轮，主持人裁决 |
| host_summary | 主持汇总 | 小曜统一整理 |

## 子任务上下文

子成员不能拿全量上下文。

子任务上下文包含：

```text
任务目标
自己的角色
需要回答的问题
相关记忆摘要
允许工具
允许资产句柄
输出格式
安全规则
```

不包含：

```text
其他成员私有记忆
无关资产
用户敏感信息
完整系统配置
明文 secret
```

## 任务审批

当 step 需要确认：

1. Task Engine 创建 approval。
2. 任务状态变为 `waiting_approval`。
3. Chat UI 收到 `approval.required`。
4. 用户确认、拒绝或修改。
5. 任务恢复或暂停/取消。

审批必须绑定：

```text
task_id
step_id
tool_call_id
risk_level
action
payload_redacted
```

## 任务工件

任务输出不只是一段回复，还可能有工件：

```text
Markdown 文档
JSON 数据
截图
浏览器 trace
终端日志
生成文件
报告
```

工件路径：

```text
data/artifacts/{task_id}/
  outputs/
  screenshots/
  traces/
  logs/
```

## 失败和重试

失败分类：

| 类型 | 处理 |
|---|---|
| model_timeout | fallback 模型 |
| tool_error | 重试或降级 |
| permission_denied | 提示授权 |
| approval_denied | 任务暂停或改计划 |
| mcp_unavailable | 停用相关能力 |
| context_overflow | 压缩上下文重试 |
| unsafe_action | 阻断并解释 |

重试规则：

```text
同一步最多重试 2 次
高风险动作不自动重试
用户拒绝后不自动换方式绕过
连续失败进入 failed
```

## 任务复盘

任务结束后执行 Reflection：

```text
是否成功
为什么成功或失败
哪些步骤可复用
是否产生记忆
是否产生 Skill 候选
是否需要更新资产使用偏好
```

复盘输出：

```json
{
  "task_success": true,
  "memory_candidates": [],
  "skill_candidates": [],
  "lessons": []
}
```

